# ox-mf2 Phase 2 Language Bindings Design

## Purpose

This document defines the Phase 2 language binding boundary for ox-mf2.

Binary AST snapshot wire format, snapshot-producing APIs, decoder/accessor APIs, and snapshot ownership are defined in [003-ox-mf2-phase-2-binary-ast-snapshot-design.md](./003-ox-mf2-phase-2-binary-ast-snapshot-design.md). This document focuses on how N-API and WASM expose that snapshot-backed model without reimplementing MF2 semantics outside Rust.

API error code ranges are defined in [appendix-ox-mf2-error-code.md](./appendix-ox-mf2-error-code.md). Bindings expose those codes through the unified `OxMf2ErrorCode` namespace while keeping parser diagnostics in the separate `DiagnosticCode` namespace.

## Basic Policy

ox-mf2 uses the Rust core as the single semantic implementation. Bindings are ergonomic wrappers around the Rust parser, snapshot writer, decoder/accessor view, diagnostics, and later formatter/linter APIs.

Bindings must not expose a nested JSON AST as the standard hot-path output. The standard public CST/AST view is a lazy accessor over the versioned Binary AST snapshot. Debug JSON may exist, but it is not the compatibility boundary.

![ox-mf2 Binary AST and binding architecture](./assets/003-ox-mf2-binary-ast-binding-architecture.svg)

## Package And Runtime Targets

Phase 2 provides two public runtime npm packages under the `@intlify` scope:

- `@intlify/ox-mf2-napi`
- `@intlify/ox-mf2-wasm`

There is no `@intlify/ox-mf2` wrapper package in Phase 2. The N-API and WASM packages expose the same public TypeScript API as much as possible, but consumers choose the runtime-specific package explicitly.

N-API native binary distribution uses platform optional packages generated by release CI. These are implementation packages for native binary delivery, not checked-in workspace packages and not additional public API packages. Consumers import `@intlify/ox-mf2-napi`, not the platform packages directly.

Generated platform package names use the host package name plus platform / architecture / ABI suffixes:

- `@intlify/ox-mf2-napi-darwin-arm64`
- `@intlify/ox-mf2-napi-darwin-x64`
- `@intlify/ox-mf2-napi-linux-x64-gnu`
- `@intlify/ox-mf2-napi-linux-x64-musl`
- `@intlify/ox-mf2-napi-linux-arm64-gnu`
- `@intlify/ox-mf2-napi-linux-arm64-musl`
- `@intlify/ox-mf2-napi-win32-x64-msvc`

Both packages are ESM-only packages. Their package `exports` map exposes only `.` as the public entry point in Phase 2. Additional subpath exports such as `./debug`, `./types`, or `./package.json` are not part of the v0.1 public compatibility surface.

Runtime support:

- N-API package: Node.js 22 or later
- WASM package: browser, bundler, and edge runtime

Node.js should use the N-API package as the primary path. WASM is the portable path for browser, playground, editor extension, and edge runtime environments where native addons are unavailable.

## Binding Scope

This document specifies only the Phase 2 N-API and WASM bindings. Other language bindings are out of scope for this document and for Phase 2 implementation.

Out-of-scope bindings include, but are not limited to:

- stable C ABI
- Go
- Swift
- C#
- Zig
- Python
- Ruby
- Java / Kotlin

Future non-JavaScript bindings should be designed in separate documents. They should reuse the same Rust core and versioned Binary AST snapshot compatibility boundary, but their ABI shape, ownership model, error mapping, package format, threading model, and host-language accessor API are not defined here.

## Crate Layout

The Rust implementation is split by binding target:

```text
crates/
  ox_mf2_parser/    # Rust parser, CST table, semantic model, snapshot writer/decoder
  ox_mf2_napi/      # N-API binding crate
  ox_mf2_wasm/      # wasm-bindgen binding crate
```

Both binding crates depend on `ox_mf2_parser`. They must not duplicate parser grammar, snapshot encoding rules, semantic lowering, diagnostic code definitions, or snapshot validation logic.

N-API uses `napi-rs`. WASM uses `wasm-bindgen`.

Phase 2 does not require a stable C ABI implementation. C ABI remains design preparation. However, snapshot record layout, numeric error codes, handle ids, buffer ownership, `toBytes()` copy semantics, and decode error boundaries stay C-ABI-friendly.

## Package Layout

The workspace package layout is:

```text
packages/
  ox-mf2-napi/              # public N-API runtime package
  ox-mf2-wasm/              # public WASM runtime package
  ox-mf2-shared/            # private internal shared types/helpers
```

Platform-specific N-API packages are not checked into `packages/`. Release CI builds native `.node` artifacts for the target matrix, generates platform optional npm package directories in a temporary release workspace, and publishes those generated packages together with the host `@intlify/ox-mf2-napi` package.

The checked-in `packages/ox-mf2-napi` package is the host package. It owns the public TypeScript API, package exports, loader, native load error messages, and dependency metadata that points to generated platform packages at publish time.

`@intlify/ox-mf2-shared` is an internal private workspace package. It contains the hand-written public TypeScript declarations, shared JS runtime helpers, option validation, error class shape, and test normalization helpers.

Published `@intlify/ox-mf2-napi` and `@intlify/ox-mf2-wasm` packages do not have a runtime dependency on `@intlify/ox-mf2-shared`. Build scripts bundle or copy the shared helper output into each public package before publish.

Each public package and relevant internal package exposes package-local scripts:

- `build`
- `typecheck`
- `test`
- `bench`

## Initialization

`@intlify/ox-mf2-napi` has no explicit initialization API. Importing the package attempts to resolve the native addon through the host loader, but native load failure does not throw during import. If the native binding is unavailable, the first API call throws `OxMf2InitializationError`. This lets consumers import the package for capability checks or explicit runtime selection without crashing module evaluation.

`@intlify/ox-mf2-wasm` requires explicit async initialization:

```ts
import { init, parseMessage } from '@intlify/ox-mf2-wasm'

await init()
const result = parseMessage('Hello {$name}')
```

The WASM initialization signature is:

```ts
type WasmInitInput = URL | RequestInfo | WebAssembly.Module | ArrayBuffer | Uint8Array

init(input?: WasmInitInput): Promise<void>
```

When `input` is omitted, the package uses the bundler/default wasm-bindgen loading path. Explicit input supports browser, bundler, and edge runtime environments that need to supply a URL, request, compiled module, ArrayBuffer, or Uint8Array manually.

Calling WASM APIs before `init()` throws `OxMf2InitializationError`. WASM does not switch to lazy initialization on first API call.

Only the first WASM `init()` call may provide explicit input. Calling `init()` after successful initialization with no arguments is a no-op. Calling `init(input)` with any explicit input after successful initialization throws `OxMf2InitializationError`, even if the caller believes the input is equivalent. This avoids defining equality for `URL`, `RequestInfo`, `WebAssembly.Module`, `ArrayBuffer`, and `Uint8Array`.

Concurrent `init()` calls share the same in-flight initialization promise when they are compatible with the first call. A no-argument `init()` call is compatible with any in-flight initialization. If initialization is in flight and another call provides a different explicit input, the later call throws `OxMf2InitializationError`; the runtime does not replace an already-started or already-initialized WASM module.

After initialization, the public parser and decoder APIs are synchronous in both packages:

- `parseMessage`
- `parseBatch`
- `decodeSnapshot`

Async parsing can be added later as a separate worker/thread-pool API. It is not part of the Phase 2 standard binding surface.

## Parser And Decoder API

The standard parse API returns snapshot-backed result objects. It does not return raw snapshot bytes and does not materialize a nested JS AST by default.

Single-message input:

```ts
type ParseInputObject = {
  source: string
  path?: string
  locale?: string
  messageId?: string
  baseOffset?: number
}

const result = parseMessage(source)
const resultWithMetadata = parseMessage({
  source,
  locale: 'en',
  messageId: 'hello'
})
```

`parseMessage(source)` is a convenience overload. `parseMessage(input)` is the standard metadata-aware form. `path`, `locale`, `messageId`, and `baseOffset` are metadata for SourceRecord mapping, diagnostics mapping, LSP/editor mapping, benchmarking/reporting, and root mapping. Metadata does not affect parser semantics.

```ts
parseMessage(input: string | ParseInputObject, options?: ParseMessageOptions): ParseMessageResult
```

Batch input:

```ts
const result = parseBatch([
  { source: 'Hello {$name}', locale: 'en', messageId: 'hello' },
  { source: 'こんにちは {$name}', locale: 'ja', messageId: 'hello' }
])
```

`parseBatch` accepts `ParseInputObject[]` only. It does not accept `string[]` and does not allow mixed `string | ParseInputObject` arrays.

```ts
parseBatch(items: ParseInputObject[], options?: ParseBatchOptions): ParseBatchResult
```

Snapshot decode:

```ts
const decoded = decodeSnapshot(bytes)
```

`decodeSnapshot(bytes)` accepts only `Uint8Array` and has no options. It copies the input bytes into a binding-owned immutable buffer before creating accessors.

## Options

`parseMessage` and `parseBatch` use one flat options object. Options are not nested under `parse` or `snapshot`.

Default option values match the Rust/snapshot defaults exposed by the binding API:

```ts
type ParseMessageOptions = {
  collectTrivia?: boolean // default: true
  includeTrivia?: boolean // default: true
  includeDiagnostics?: boolean // default: true
  includeSourceText?: boolean // default: false
}

type ParseBatchOptions = ParseMessageOptions & {
  batchExecution?: 'sequential' | 'parallel'
}
```

`batchExecution` applies only to `parseBatch`. If `batchExecution: 'parallel'` is requested before parallel execution is implemented, the binding degrades to sequential execution and reports that in the batch result. It does not add a parser diagnostic and does not throw.

`parseSemantic` is not part of the Phase 2 binding API. SemanticView / semantic snapshot exposure is a future design topic. Phase 2 bindings expose syntax snapshot accessors and diagnostics only.

When `includeDiagnostics = false`, diagnostics are not encoded into the snapshot and are not exposed through binding result accessors. `result.diagnostics` and `root.diagnostics()` return empty arrays. This keeps binding diagnostics snapshot-backed and avoids a second non-snapshot diagnostic channel.

`collectTrivia = false` and `includeTrivia = true` is invalid and throws `TypeError`. The parser cannot encode trivia that it did not collect, and silently downgrading `includeTrivia` would make benchmark and formatter behavior harder to reason about.

When `includeTrivia = false`, trivia is not encoded into the snapshot. `token.leadingTrivia()` and `token.trailingTrivia()` return empty arrays. Formatter or linter entry points that require trivia should validate options before calling the parser.

Unknown options throw `TypeError`. This catches option typos early and keeps the public compatibility surface explicit. Since `batchExecution` is accepted only by `parseBatch`, passing it to `parseMessage` is also a `TypeError`.

## Input Validation

Input type and range errors use built-in JS errors:

- wrong input type: `TypeError`
- invalid integer/range value: `RangeError`
- indexed accessor out of range: `RangeError`
- `parseBatch([])`: `RangeError`

`baseOffset` must be an integer in `0 <= baseOffset <= u32::MAX`. It defaults to `0`.

`path`, `locale`, and `messageId` are optional strings. Empty strings are allowed. The binding layer does not validate BCP 47, filesystem paths, or message id semantics.

Bindings map `messageId` to Rust/snapshot `message_id`, and `baseOffset` to `base_offset`. `baseOffset` is a UTF-8 byte offset. JavaScript string UTF-16 position conversion is a binding/editor adapter responsibility and is not stored in snapshot node fields.

Phase 2 bindings reject JavaScript strings that contain unpaired surrogates with `TypeError`. The Rust core path remains UTF-8-only in Phase 2. WTF-8 or UTF-16 ingestion for full ECMAScript String compatibility is deferred to a separate measured design.

## Result Object Boundary

Single-message result shape:

```ts
type ParseMessageResult = {
  snapshot: SnapshotAccessor
  roots: RootHandle[]
  root: RootHandle
  sources: SourceView[]
  source: SourceView
  diagnostics: DiagnosticView[]
}
```

Batch result shape:

```ts
type ParseBatchResult = {
  snapshot: SnapshotAccessor
  roots: RootHandle[]
  sources: SourceView[]
  diagnostics: DiagnosticView[]
  execution: 'sequential' | 'parallel'
  degraded: boolean
}
```

Decoded result shape:

```ts
type DecodedSnapshotResult = {
  snapshot: SnapshotAccessor
  roots: RootHandle[]
  sources: SourceView[]
  diagnostics: DiagnosticView[]
  withSources(sources: string[]): DecodedSnapshotResult
}
```

`parseMessage` result does not include `execution` or `degraded` because single-message parsing has no batch execution strategy.

`result.roots[i]` always corresponds to the batch input item at `items[i]`. The snapshot format allows multiple roots to point to the same SourceRecord, but the v0.1 default writer does not deduplicate SourceRecord entries. Batch result root order does not change from input order.

`result.source` and `result.sources` are SourceRecord-backed SourceViews, not raw source strings. Use `source.sourceSlice(span)` or SourceView accessors to read source text.

## Snapshot Accessor Boundary

`result.snapshot` is an accessor object, not raw bytes. Root, node, token, trivia, diagnostic, and source metadata are read lazily through accessors. Raw snapshot bytes are not included in the default result shape.

Public Root / Node / Token / Trivia handles are pairs of snapshot owner and section-local id, not object pointers.

```text
RootHandle   { snapshot: SnapshotRef, id: RootId }
NodeHandle   { snapshot: SnapshotRef, id: NodeId }
TokenHandle  { snapshot: SnapshotRef, id: TokenId }
TriviaHandle { snapshot: SnapshotRef, id: TriviaId }
```

N-API and WASM handles keep references to the accessor object that owns the snapshot buffer. Handles do not copy snapshot bytes and do not point directly into Rust memory in a way that can dangle after JS GC, WASM lifetime changes, or worker transfer.

Handles are JS class instances, not plain objects. Immutable scalar `id` is a property. Other data is exposed through methods so callers can distinguish stable identity from snapshot-backed reads.

```ts
type ChildHandle = NodeHandle | TokenHandle

root.id
root.node(): NodeHandle
root.diagnostics(): DiagnosticView[]

node.id
node.kind(): SyntaxKind
node.span(): Span
node.childCount(): number
node.childAt(index: number): ChildHandle
node.children(): ChildHandle[]

token.id
token.kind(): SyntaxKind
token.span(): Span
token.leadingTrivia(): TriviaHandle[]
token.trailingTrivia(): TriviaHandle[]

trivia.id
trivia.kind(): SyntaxKind
trivia.span(): Span
```

Array-returning accessors return a new array on every call. They do not return cached mutable arrays. Allocation-sensitive consumers use count/indexed APIs such as `node.childCount()` and `node.childAt(index)`.

JS iterators may be added as convenience APIs, but they are not the primary compatibility surface.

## Snapshot Accessor API

`SnapshotAccessor` exposes direct indexed accessors:

```ts
snapshot.root(id: number): RootHandle
snapshot.node(id: number): NodeHandle
snapshot.token(id: number): TokenHandle
snapshot.trivia(id: number): TriviaHandle
snapshot.source(id: number): SourceView
snapshot.diagnostic(index: number): DiagnosticView
```

It also exposes count methods:

```ts
snapshot.rootCount(): number
snapshot.nodeCount(): number
snapshot.tokenCount(): number
snapshot.triviaCount(): number
snapshot.sourceCount(): number
snapshot.diagnosticCount(): number
```

`snapshot.section(kind)` is included as an advanced/debug API for fixture investigation, snapshot validation, and benchmark analysis. It takes numeric enum-like `SectionKind` values, not strings, and returns read-only section metadata.

```ts
type SectionMetadata = {
  kind: SectionKind
  offset: number
  byteLength: number
  recordSize: number
  count: number
  required: boolean
}

snapshot.section(kind: SectionKind): SectionMetadata | null
```

If an optional section is absent, `snapshot.section(kind)` returns `null`. If the section exists but is empty, it returns `SectionMetadata` with `count = 0` and `byteLength = 0`. This lets debug tools distinguish absent optional sections from present empty sections.

Missing optional sections behave as empty sections for count accessors. For example, if the trivia section is absent, `snapshot.triviaCount()` returns `0`. Direct indexed accessors still validate indexes against the count, so `snapshot.trivia(0)` or `snapshot.diagnostic(0)` throws `RangeError` when the corresponding count is `0`.

## Source Text Retention

Bindings may retain original source text on the caller side or in the binding result. Since the default snapshot uses `includeSourceText = false`, context-bound `source.sourceSlice(span)` reads external source text, not snapshot bytes, when provided by the decoder/accessor.

If the binding result keeps external source text, context-bound `source.sourceSlice(span)` may succeed even with `includeSourceText = false`. If the binding result does not keep source text and the snapshot has no source text data, `source.sourceSlice(span)` throws `OxMf2SourceTextError`.

For `parseMessage` and `parseBatch`, bindings retain source text as binding-owned UTF-8 bytes. They do not keep only the original JS string and do not convert UTF-8 byte offsets to JS string indexes on every slice. This keeps `Span` slicing and byte-oriented location computation aligned with the Rust core and snapshot.

Line-start tables are built lazily per source and cached on the binding-owned source text object. Parsing and snapshot creation do not eagerly build line indexes for every source. Diagnostics or future location helpers pay the line-index cost only when locations are read.

In batch results, `result.sources` order is SourceId order, not input order. In v0.1 default writer output this usually matches input order, but consumers must not rely on that because the snapshot format permits shared SourceRecord entries.

With `includeSourceText = false` in a batch result, each root has a `source_id`, source metadata is in SourceRecord, and input source text references are retained by the binding result. With `includeSourceText = true`, source text data is included in the snapshot so the snapshot alone can resolve source slices after worker transfer or persistence.

## Source Reattachment

`decodeSnapshot(bytes)` reads only snapshot bytes. If a snapshot does not include source text, source text can be reattached with `decoded.withSources(sources)`.

```ts
const decoded = decodeSnapshot(bytes)
const withText = decoded.withSources(['Hello {$name}'])
```

`withSources(sources)` maps strings by snapshot SourceId order. It takes source text only and does not accept metadata. Metadata already lives in SourceRecord.

Each source string passed to `withSources` follows the same Phase 2 UTF-8 source policy as parser input. Strings containing unpaired surrogates are rejected with `TypeError`, and accepted strings are stored as binding-owned UTF-8 bytes.

`withSources(sources)` returns a new `DecodedSnapshotResult`. It does not mutate the original decoded result. Handles created from the original decoded result do not gain source text; only handles obtained from the returned decoded result can read the attached source text.

If `sources.length !== snapshot.sourceCount()`, `withSources` throws `OxMf2SourceTextError`.

`withSources()` is only for decoded snapshots that do not already carry source text. If the decoded snapshot already has source text data, `withSources()` throws `OxMf2SourceTextError` instead of replacing the embedded source text. This keeps `source.sourceSlice(span)` and diagnostic locations unambiguous.

## Source Location Boundary

Binding `Span` and `SourceLocation` values use the same coordinate system as the Rust core and Binary AST snapshot. `Span.start`, `Span.end`, and `SourceLocation.column` are UTF-8 byte offsets. `SourceLocation.line` is one-based and `SourceLocation.column` is zero-based within the line.

Phase 2 bindings do not expose UTF-16 location helpers such as `source.locationUtf16(offset)` or `source.spanLocationUtf16(span)`. UTF-16 code unit columns are an LSP/editor/source-map adapter concern. Those adapters may add explicit UTF-16 helper APIs later, but the standard binding accessor surface remains byte-oriented.

## Source View

Source metadata accessors use method style:

```ts
source.id
source.path(): string | null
source.locale(): string | null
source.messageId(): string | null
source.baseOffset(): number
source.sourceSlice(span: Span): string
```

Optional string metadata returns `string | null`, not `undefined`.

`source.sourceSlice(span)` validates that source text is available and that `span.start <= span.end <= sourceTextByteLength`. It also validates that both byte offsets are UTF-8 code point boundaries before returning a JavaScript string. If either offset is not a UTF-8 boundary, it throws `OxMf2SourceTextError` instead of replacing invalid bytes.

Because `Span` does not carry `source_id`, the binding does not try to prove that a span came from a particular SourceView. It validates only the UTF-8 byte range against the selected source text. If validation fails, it throws `OxMf2SourceTextError`.

## Raw Bytes Export

Advanced use cases such as debug, fixtures, persistence, worker transfer, and external process transport can explicitly extract snapshot bytes.

```ts
const bytes = result.snapshot.toBytes()
```

`toBytes()` returns a `Uint8Array` copy of the internal snapshot buffer in both N-API and WASM. The binding does not expose the internal buffer directly. This prevents consumers from retaining, mutating, or transferring returned bytes in a way that breaks the lifetime or validation invariants of existing accessor objects.

There is no result-level `toBytes()` shortcut. Raw bytes are exported through `result.snapshot.toBytes()` only.

## Diagnostics

`result.diagnostics` is a flat array in root order. In a single-message result, it contains diagnostics for one root. In a batch result, it contains all diagnostics grouped by `result.roots` order. Each root handle can also expose its own diagnostics range through `root.diagnostics()`.

The flat diagnostics array and root-local diagnostics accessor read the same snapshot diagnostics section. Bindings do not duplicate diagnostics into another table. For JS/WASM ergonomics, diagnostic view objects may be materialized when consumers read `result.diagnostics`.

Diagnostic objects include labels and an optional source location:

```ts
type DiagnosticView = {
  rootId: number
  sourceId: number
  severity: DiagnosticSeverity
  code: DiagnosticCode
  message: string | null
  span: Span
  location: SourceLocation | null
  labels: DiagnosticLabelView[]
}

type DiagnosticLabelView = {
  span: Span
  message: string | null
}
```

`DiagnosticView.rootId` and `DiagnosticView.sourceId` are snapshot-local numeric ids. They let consumers handle `parseBatch()` flat diagnostics without first grouping by `root.diagnostics()`. `DiagnosticLabelView` does not repeat root/source identity because labels belong to their parent diagnostic.

`DiagnosticView.sourceId` is read directly from `DiagnosticRecord.source_id`. `DiagnosticView.rootId` is derived from `RootRecord.diagnostic_start` / `diagnostic_count` ranges because `DiagnosticRecord` does not store `root_id`. Bindings build the diagnostic-index-to-root-id lookup lazily when flat diagnostics are materialized or when `snapshot.diagnostic(index)` needs root identity.

`DiagnosticView.location` is computed when the diagnostic object is materialized. It is a property, not a lazy `diagnostic.location()` method.

If source text is unavailable, `location` is `null`. This can happen for `decodeSnapshot(bytes)` when the snapshot does not include source text and the caller has not used `withSources`.

When `includeDiagnostics = false`, `result.diagnostics` and `root.diagnostics()` return empty arrays because the diagnostics section is absent from the snapshot.

Recoverable parser diagnostics do not throw. When diagnostics are included, they are returned through `result.diagnostics` and `root.diagnostics()`.

## Error Mapping

Bindings throw exceptions for API errors. They do not expose `Result<T, Error>` as the standard TypeScript API.

Dedicated error classes:

- `OxMf2ParseError`
- `OxMf2SnapshotError`
- `OxMf2SourceTextError`
- `OxMf2InitializationError`

Input type/range errors and indexed accessor misuse use built-in `TypeError` and `RangeError`.

Dedicated error classes expose:

```ts
type OxMf2ErrorShape = {
  code: OxMf2ErrorCode
  message: string
  cause?: unknown
  sectionKind?: SectionKind
  offset?: number
  recordIndex?: number
}
```

Each dedicated error class sets `error.name` to the class name, such as `OxMf2SnapshotError`.

The binding boundary preserves error code, message, optional section kind, optional offset, and optional record index. Human-readable messages may be returned for developer ergonomics, but compact error codes are the base for programmatic handling.

`OxMf2ErrorCode` follows the range policy in [appendix-ox-mf2-error-code.md](./appendix-ox-mf2-error-code.md). Binding-owned errors start at `10000` so future Rust core error domains can be added under `10000` without moving binding codes.

Binding error class mapping:

| Error domain                 | Binding error class        |
| ---------------------------- | -------------------------- |
| `DecodeErrorCode`            | `OxMf2SnapshotError`       |
| `SnapshotWriteErrorCode`     | `OxMf2SnapshotError`       |
| `SourceTextErrorCode`        | `OxMf2SourceTextError`     |
| `InitializationErrorCode`    | `OxMf2InitializationError` |
| `BindingValidationErrorCode` | `OxMf2ParseError`          |

Wrong input types, invalid numeric ranges, and indexed accessor misuse still use built-in `TypeError` or `RangeError` when those built-ins express the failure clearly. `BindingValidationErrorCode` is reserved for ox-mf2-specific validation failures that need a stable numeric code.

`OxMf2ErrorCode` exposes detailed numeric constants that are close to the Rust error domains, while error classes provide coarse JS exception grouping. Example constant names:

```ts
OxMf2ErrorCode.DecodeInvalidMagic
OxMf2ErrorCode.DecodeUnsupportedMajorVersion
OxMf2ErrorCode.DecodeInvalidSectionBounds
OxMf2ErrorCode.SnapshotWriteTooManyNodes
OxMf2ErrorCode.SnapshotWriteInvalidSourceId
OxMf2ErrorCode.SourceTextNotIncluded
OxMf2ErrorCode.SourceTextSpanOutOfBounds
OxMf2ErrorCode.InitializationWasmNotInitialized
OxMf2ErrorCode.InitializationNativeBindingUnavailable
OxMf2ErrorCode.BindingValidationInvalidOptions
```

The exact constant set is generated or hand-maintained from the Rust error code domains, but the public shape stays as a numeric const object plus `oxMf2ErrorCodeName(code)`.

## Numeric Constants

Snapshot and diagnostic kinds are exposed as numeric const objects with string-name helper functions:

```ts
SyntaxKind.Message
DiagnosticCode.UnclosedPlaceholder
DiagnosticSeverity.Error
SectionKind.Nodes
OxMf2ErrorCode.DecodeInvalidMagic

syntaxKindName(kind)
diagnosticCodeName(code)
diagnosticSeverityName(severity)
sectionKindName(kind)
oxMf2ErrorCodeName(code)
```

The public TypeScript type may be modeled as enum-like numeric unions, but the runtime value is a const object. String literal unions are not the primary representation because the snapshot format is numeric.

## TypeScript Declarations

TypeScript declarations are hand-written common public `.d.ts` files shared by the N-API and WASM packages.

Generated `napi-rs` or `wasm-bindgen` types must not leak directly as the public compatibility surface. Generated types may be used internally during build validation, but the published API shape is the shared hand-written declaration.

`packages/ox-mf2-shared` is the source of truth for shared public TypeScript types and shared JS helper behavior. Public package build steps copy or bundle the relevant generated output from `ox-mf2-shared` into `ox-mf2-napi` and `ox-mf2-wasm`.

## Tests And Fixtures

N-API and WASM API parity is mandatory. The same TypeScript fixture suite runs against both packages and compares normalized structured output.

Parity tests cover:

- function names and accepted options
- error classes and error codes
- root/node/token/trivia/source/diagnostic accessor behavior
- `Span` and `SourceLocation` values
- `withSources()` behavior
- `toBytes()` / `decodeSnapshot()` roundtrip behavior

Binding fixtures reuse the Rust snapshot `.mf2` corpus. Binary golden bytes remain the Rust snapshot layer's responsibility. Binding tests compare decoded structural/accessor output and language-boundary behavior.

## Benchmarks

Binding work must not be hidden inside parser or snapshot numbers.

Relevant binding benchmark phases:

- `parse_message_binding`
- `parse_batch_binding`
- `decode_snapshot_binding`
- `snapshot_to_bytes_copy_binding`

The runtime benchmark matrix includes:

- Node.js N-API
- Node.js WASM
- browser WASM

Browser WASM benchmarks run through Playwright with a headless browser so browser runtime cost is measured automatically and reproducibly.

`snapshot_to_bytes_copy_binding` is reported both from the snapshot perspective and binding perspective. The binding report includes language boundary allocation and copy cost.

## Build And Distribution

The WASM package publishes JS glue plus a separate `.wasm` file. The `.wasm` is not base64-inlined into JS by default.

The N-API package uses platform optional packages for native prebuilds, but those platform packages are generated release artifacts. They are not checked into the source workspace.

The host package is `@intlify/ox-mf2-napi`. At release time, CI builds native artifacts for each target, creates temporary platform package directories, wires them into the host package's optional dependency metadata, and publishes the host package plus generated platform packages.

Consumers install and import only `@intlify/ox-mf2-napi`. The loader inside the host package resolves the current platform artifact from the generated optional package set.

Generated package names:

If no platform optional package exists for the current environment, the N-API package reports a native binding load failure through the host package loader. Phase 2 does not require a source build fallback and does not automatically fall back to the WASM package. Consumers that need a portable runtime should import `@intlify/ox-mf2-wasm` explicitly.

## CI And Release Policy

Binding CI must run:

- build
- typecheck
- API parity tests
- smoke import tests
- benchmark smoke tests

`@intlify/ox-mf2-napi` and `@intlify/ox-mf2-wasm` are published with the same version. Independent package versions are not used while public API parity is a design requirement.

## Implementation Milestones

Phase 2 language binding implementation is staged as:

1. Shared types/helpers: build `packages/ox-mf2-shared`, hand-written declarations, option validation, error classes, normalization helpers, and shared tests.
2. N-API binding: implement `crates/ox_mf2_napi`, `packages/ox-mf2-napi`, generated platform optional package loading, and native binding load errors.
3. WASM binding: implement `crates/ox_mf2_wasm`, `packages/ox-mf2-wasm`, explicit `init(input?)`, and wasm asset packaging.
4. API parity tests: run the same fixture suite against N-API and WASM and compare normalized structured output.
5. Benchmarks and release validation: add binding benchmarks, benchmark smoke tests, package smoke imports, and publish validation.
