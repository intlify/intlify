# ox-mf2 Phase 3D LSP and Editor Design

This document tracks the detailed LSP and editor integration design for ox-mf2.

The Phase 3 tooling boundary is defined in [005-ox-mf2-phase-3-tooling-transport-design.md](./005-ox-mf2-phase-3-tooling-transport-design.md). That document fixes the high-level tooling workflow. This document is the implementation-facing place to refine host document mapping, diagnostic publication, formatting edits, configuration reload behavior, and future editor features.

Where the Phase 3 boundary document describes resource and catalog files, the editor workflow is specified against the host-format-agnostic message entry model. Localizable MF2 messages are managed in message catalogs with many different host formats — JSON, YAML, JSON5, XLIFF, framework-specific single-file-component blocks, and other localization interchange formats — so this document specifies editor behavior against extracted message entries rather than against any concrete host format. The message entry model, the host format adapter contract, the host format registry, catalog configuration, and the host format tier roadmap are owned by [013-ox-mf2-resource-catalog-adapter-design.md](./013-ox-mf2-resource-catalog-adapter-design.md). Application-level reference resolution, locale/fallback resolution, reachability, completeness, and linker findings are owned by [014-ox-mf2-message-linker-design.md](./014-ox-mf2-message-linker-design.md). This document owns the editor-facing behavior built on those shared contracts.

## Goals

- Reuse the parser, formatter, linter, `SnapshotView`, and future `SemanticView` without making the core crates depend on LSP protocol types.
- Support diagnostics and formatting for standalone `.mf2` files.
- Support diagnostics and formatting for MF2 messages embedded in message catalog host files through resource-owned extraction and mapping, starting with JSON catalogs; later host formats follow the tier roadmap owned by [013-ox-mf2-resource-catalog-adapter-design.md](./013-ox-mf2-resource-catalog-adapter-design.md).
- Build the editor workflow on the consumer-neutral message entry contract owned by [013-ox-mf2-resource-catalog-adapter-design.md](./013-ox-mf2-resource-catalog-adapter-design.md), so additional catalog formats extend the shared registry without changing diagnostics publication, formatting edits, caching, or position conversion behavior.
- Convert core UTF-8 byte spans into editor-facing positions, UTF-16 by default, at the editor integration boundary.
- Keep LSP/editor artifact caches, document version checks, and host-file edit ownership outside of the parser, formatter, and linter cores.
- Define the additive project-backed editor session that combines configured disk inventory with current open-buffer contents and exposes capability-specific read-only 014 outputs—findings for L0/L1 diagnostics and the M1 typed-key model for E0 completion—without creating editor-specific link semantics.

## Non-Goals

- Making an LSP server or editor extension a direct Phase 3 product.
- Adding LSP protocol types to parser, formatter, or linter result objects.
- Implementing code actions, quick fixes, hover, completion, go-to-definition, or rename in the initial workflow.
- Implementing true range-only formatting or minimal-diff formatting in the formatter core.
- Owning the message entry model, host format adapter contract, registry, catalog configuration, or host format tier roadmap; those are owned by [013-ox-mf2-resource-catalog-adapter-design.md](./013-ox-mf2-resource-catalog-adapter-design.md).
- Owning editor-facing host-format syntax diagnostics, host-format schema diagnostics, or host-format styling. Resource-owned built-in adapters still perform the complete host parse and validation required for fail-complete extraction and write-back; the editor integration does not reinterpret those checks or present itself as the JSON, YAML, or XML language service. Host syntax diagnostics and document layout remain owned by host-format tooling.
- Owning or reimplementing application-level cross-file and cross-locale analysis. `ambiguous-message-definition`, `unresolved-message`, `missing-translation`, `orphaned-translation`, `unused-message`, `unbounded-dynamic-reference`, and `degraded-analysis` are generated by `intlify_linker` under [014](./014-ox-mf2-message-linker-design.md); a later editor integration only projects its `LinkOutcome`. Single-resource structural checks that need no application reference, fallback, reachability, or completeness model remain future catalog-native lint work under 008 and 013.
- Defining a plugin system for editor integrations or a third-party host format adapter API.

## Initial Workflow

The initial editor workflow focuses on diagnostics and formatting.

For a catalog document, the editor integration asks the resource-owned layered resolver and registry to classify and extract MF2 message entries; the selected built-in host adapter parses the host syntax and populates the resource artifact. For a standalone `.mf2` document, the editor-owned standalone adapter constructs its separate framing-aware artifact. The editor integration then calls core APIs per entry and maps message-local results back to document-level ranges. Core APIs operate on message-local UTF-8 byte spans. The editor integration owns document URI handling, document version checks, editor position conversion, request orchestration, and protocol-specific result shapes.

## Host Document Model

![Phase 3D LSP and editor architecture](./assets/009-ox-mf2-phase-3d-lsp-editor-architecture.svg)

Editor integrations treat every supported document as a host document that yields zero or more message entries over the same message-level core.

- A **host document** is one editor document: a standalone `.mf2` file or an opted-in message catalog file, beginning with JSON catalogs.
- A **message entry** is the behavioral unit that connects one host document region to one MF2 message: a stable concrete entry key, a catalog-key domain and logical catalog key, a raw host value span, MF2 message text, a message-to-raw offset map, and a read-only marker. Only a project-configured linker-participating catalog entry's comparable domain and logical key enter 014 definition projection and linker grouping. A standalone editor artifact retains `CatalogKeyDomain::StandaloneMf2` and its empty key only as editor-local metadata; neither value is projected into 014. The canonical behavioral fields and mapping invariants are owned by [013-ox-mf2-resource-catalog-adapter-design.md](./013-ox-mf2-resource-catalog-adapter-design.md). Resource catalog artifacts and standalone editor artifacts do not share one constructible concrete Rust entry type: the editor presents both through an editor-local read-only entry view.
- An **editor integration** is the LSP server or extension boundary that owns document lifecycle, routing requests into the standalone or resource path, core-call orchestration, editor position conversion, and protocol results. It does not own catalog matching or host syntax.
- A **standalone editor adapter** is the editor-owned `.mf2` entry producer that applies file framing and constructs the standalone document map. It is not a resource host-format adapter.
- A **built-in host format adapter** is the resource-owned, format-specific component that receives an already validated `ResolvedHostFormat`, parses catalog host syntax, extracts and maps entries, and plans and materializes value-identical re-escaping. Catalog assignment, top-level format resolution, and dispatch precede this adapter and belong to the resource resolver and registry, per [013](./013-ox-mf2-resource-catalog-adapter-design.md#host-format-adapter-contract).

Standalone `.mf2` files are the degenerate host format in the editor workflow. Before producing its one editor-owned entry artifact, the standalone editor adapter applies the Phase 3B [File Framing](./007-ox-mf2-phase-3b-formatter-design.md#file-framing) read contract: it removes at most one leading UTF-8 BOM and then one trailing `LF` or `CRLF`, when present. The entry's message text is that unframed text, its raw value span is the whole document, and its editor-owned document map retains the removed framing so message-local positions map back to the original bytes. Standalone and catalog entries take the same downstream per-message parser, linter, formatter, and diagnostic-mapping path through the common view. Final edit construction remains path-specific: the editor integration reapplies standalone file framing, while catalog edits come only from the resource artifact's integrated validated write-back.

All downstream editor behavior in this document — diagnostics publication, formatting edits, artifact caching, and position conversion — is defined against message entries, never against concrete host formats. Supporting a new catalog format means adding its built-in adapter and registry entry to `crates/intlify_resource`; it must not require changes to parser, formatter, or linter core APIs or to the editor workflow contract.

## Entry Production and Ownership Boundaries

The editor integration consumes catalog assignment, format resolution, built-in adapter dispatch, extraction, offset mapping, and validated write-back from the shared resource layer specified by [013-ox-mf2-resource-catalog-adapter-design.md](./013-ox-mf2-resource-catalog-adapter-design.md). The editor-owned standalone adapter produces its separate entry artifact, exposes the same read-only behavioral fields and `map_span` operation through the editor-local common view, and applies the Phase 3B file-framing contract; it does not construct a resource `MessageEntry` or route `.mf2` files through catalog extraction. The editor workflow relies on these guarantees:

- entries are ordered by raw span start and raw value spans do not overlap
- catalog entry text is byte-exact, with no trimming, normalization, or injected framing; standalone entry text differs from the document bytes only by the explicitly removed file framing
- before catalog host parsing, the editor passes the current Unicode document's UTF-8 byte length through the resource-owned `preflight_host_bytes` limit check; `extract` defensively repeats that check, while decoding of editor protocol input remains outside the resource layer
- offset maps are monotonic and mapped ranges never split a host escape sequence
- write-back re-escaping is value-identical, and entries that cannot be safely re-escaped are marked read-only
- project-configured catalog detection resolves identically in editor and CLI resource workflows; the CLI-only config-free direct-file and stdin exception is not editor auto-detection
- the editor validates the normalized ad-hoc value through the resource-owned overlay validator and calls `ResolvedResources::resolve_path_with_overlay`; after its opaque layered result selects one assignment, the editor calls the registry's `resolve_format` boundary to obtain a `ResolvedHostFormat` or the shared typed `resource_format_unsupported`, then moves that resolved context into extraction; it does not compile overlay globs, construct assignments, or reconstruct unsupported-format details or target-extension evidence

Editor routing over the resource registry follows these rules:

- documents with the `.mf2` extension or the exact case-sensitive canonical language id `messageformat2` always select the standalone editor path before any project or ad-hoc catalog match is considered; no `mf2` language-id alias is defined
- among catalog membership sources, the shared layered resolver treats the validated editor overlay as fallback-only and returns project or overlay provenance; editor-specific settings may add editor-only ad-hoc catalog opt-in only when project `resources.catalogs` is absent or a non-empty project policy has no matching include
- an explicitly empty project catalog policy and a path excluded by every including project definition both block overlay opt-in; a project match is authoritative and bypasses path-level overlay resolution, so same-path cross-layer de-duplication and cross-layer format conflicts do not exist
- when overlay fallback is eligible, its own overlapping definitions apply the shared same-format de-duplication and different-format conflict rule
- after standalone precedence, the layered resource resolver returns at most one opaque catalog assignment or one conflict; `HostFormatRegistry::resolve_format` then resolves at most one shipped format, and extraction dispatches exactly that format's one built-in adapter

For ordinary per-document parsing, linting, diagnostics, and formatting, editor document identity is the client-owned document URI together with its protocol version, not filesystem physical identity. That workflow does not stat, canonicalize, or group open documents by device/inode, volume/file id, hard-link target, or symlink target. Two URIs that happen to name the same physical file retain independent text snapshots, versions, diagnostics, caches, and formatting transactions; a change or edit for one URI is never inferred for the other. The CLI's physical alias conflict and post-write fail-stop rules do not apply because the editor returns version-checked protocol edits and does not directly coordinate writes to those aliases.

This URI-local rule does not replace the physical-source grouping required by an active project linker session. That session reuses the shared 005/014 project-inventory boundary and its canonical primary-plus-alias groups, then applies open-buffer substitution to each already formed group under the rules below. The linker grouping is read-only analysis state: it neither merges the URI-local document states nor authorizes an edit for one URI from another URI's version.

## Diagnostics Publication

Editor diagnostics should be produced from the shared diagnostic result contract used by parser, semantic, and linter workflows.

The preferred initial path is to use source-backed `lintMessage` per message entry as the diagnostic source for editor publication because it already includes parser, semantic, and lint diagnostics. A future `lintSnapshot` path is an optimization for parse-artifact reuse after the parser owns a snapshot-to-`SemanticModel` path. If an editor integration composes diagnostics manually from cached parser, semantic, and linter results, that composition is an internal optimization only and must preserve the same ordering, category/code/severity values, and de-duplication behavior as `lintMessage`.

The editor integration consumes the linter result contract from [008-ox-mf2-phase-3c-linter-design.md](./008-ox-mf2-phase-3c-linter-design.md). Parser-owned semantic diagnostic catalog and ordering details remain owned by [012-ox-mf2-parser-semantic-validation-design.md](./012-ox-mf2-parser-semantic-validation-design.md). Rule-facing documentation and rule ids are indexed from [linter-rules/index.md](./linter-rules/index.md). Editor integrations should not redefine diagnostic meaning, rule metadata, or semantic validation behavior.

The initial editor workflow follows the same strict pipeline as CLI and bindings for each message entry:

- parser diagnostics are always reported
- semantic diagnostics are reported only when parsing has no parser diagnostics and semantic validation runs
- lint diagnostics are reported only when parser and semantic diagnostics are clean
- parser diagnostics prevent semantic validation and linter rule execution

Multi-entry host documents add document-level publication rules on top of the per-entry contract:

- Entries are independent. Parser diagnostics in one entry never suppress semantic validation, linting, or publication for other entries; the strict pipeline is per entry, not per document.
- Published document diagnostics are the union of all entries' mapped diagnostics. Entries appear in raw span order; within one entry, the core result order is preserved verbatim. Because entry spans do not overlap, this ordering is deterministic.
- Diagnostic identity across document updates should combine the entry key, the stable diagnostic code, and the message-local span. Entry keys survive unrelated edits elsewhere in the catalog, so this identity is more stable than document-level spans alone.
- Future catalog-native structural findings, such as duplicate concrete host keys, use the dedicated catalog finding contract coordinated by 008 and 013 rather than being attached to an arbitrary message-level diagnostic. They require no application reference, fallback, reachability, or completeness analysis.
- Application-level cross-file and cross-locale findings come only from the 014 `LinkOutcome`. The editor does not infer missing, unused, orphaned, unresolved, ambiguous, dynamic-reference, or degraded-analysis results from its open entries, and it does not rerun linker semantics inside an LSP adapter. Concrete definition evidence maps through the shared entry mapping; reference evidence maps through producer source provenance; a finding with no truthful source range is presented without inventing a synthetic owner entry. One linker finding remains one editor finding regardless of its related-evidence count.

Every diagnostic job captures the exact document version, mapping-generation token, and root config-generation token used for extraction, linting, and span conversion. Immediately before publication, all three must still match the installed state. A stale job is discarded without publishing or clearing diagnostics; the document-change or successful-config-reload event that advanced a token owns scheduling its replacement and has already applied any required clear. Every document text/version change and every membership or mapping change clears the prior mapped publication before rebuilding because its host ranges are no longer trustworthy. A lint-option-only change may retain the last complete publication until the replacement job succeeds, then replaces the complete set atomically; results from the old config generation can never overwrite the new set.

Extraction failure presentation depends on the failure class, and every class returns no formatting edits.

The special projection of selected resource operational codes into editor diagnostics, without admitting them to the parser/semantic/lint diagnostic namespace, is indexed in [appendix-ox-mf2-error-code.md](./appendix-ox-mf2-error-code.md). The behavior below remains authoritative for editor publication and retained state.

- `resource_parse_failed` publishes no MF2 or resource diagnostic and leaves the document with an empty ox-mf2 diagnostic set. The document-change or mapping-change event normally publishes the empty set before scheduling extraction; if no such clear has yet been published for the current generation, the failed job publishes it exactly once. Host-format tooling owns the syntax diagnostic. Old MF2 diagnostics are never retained across this failure because their host ranges were derived from a previous document version or mapping.
- `resource_format_unsupported`, `resource_entry_unsupported`, `resource_document_unsupported`, and `resource_limit_exceeded` with `phase: "extract"` replace the previous ox-mf2 state: stale MF2 diagnostics are cleared and exactly one error-severity editor diagnostic is published with source `ox-mf2-resource` and the corresponding stable resource code. The editor integration uses the precise original-host span carried by the typed resource error. When a document-wide extraction failure has no truthful original-host site, it uses an empty range at document start. These resource diagnostics are not parser, semantic, or lint diagnostics and do not participate in rule configuration or `--max-warnings`.
- `resource_limit_exceeded` with `phase: "validate_write_back"` describes only a rejected formatting candidate, not the installed host document. The formatting request returns this typed request-scoped operational failure with no edits, but publishes no document diagnostic, clears no existing diagnostic, installs no artifact or mapping, advances no mapping or config generation, and schedules no diagnostic refresh merely to clear a candidate-only condition. A later independent request starts from the unchanged installed state. This failure does not activate the root-scoped configuration-failure episode below or disable subsequent formatting requests.
- Configuration and internal failures publish neither MF2 nor resource diagnostics. They use the editor integration's operational error channel and leave the last successfully applied configuration unchanged; when the document version and mapping are also unchanged, they leave its diagnostic publication unchanged. No partial configuration, extraction artifact, or new diagnostic set is installed. An independent document, membership, or mapping change still performs its mandatory diagnostic clear while the failure is active. Formatting requests then return no edits until recovery atomically installs a successful reload and extraction.

The CLI presents resource failures through its target-local operational error contract instead; that surface behavior is owned by [013-ox-mf2-resource-catalog-adapter-design.md](./013-ox-mf2-resource-catalog-adapter-design.md).

Core `"error"` and `"warn"` severities map to editor/LSP diagnostic severity at the editor integration boundary; integrations convert `"warn"` to the editor's warning severity. Editor layers may add protocol-specific advice later, but the core linter does not emit `info` or `hint` diagnostics initially.

## Project-Backed Linker Session

### Activation and ownership

Linker-backed editor diagnostics are an additive 014 integration, not a prerequisite for the initial per-entry parser, semantic, lint, and formatting workflow. M0 defines the five M0 finding kinds and the checked inputs from which `LinkOutcome` can contain them; it does not make configured reference producers a prerequisite for every kind. The effective 008 capability union decides whether one invocation needs application-reference production: definition-only analysis uses the fast path below, while reference-dependent analysis activates the producer inventory. L0 makes the five M0 kinds available as lint/editor diagnostics after the Phase 3C and catalog-level presentation contracts exist. M2 adds `missing-translation` and `orphaned-translation` to linker analysis, and L1 subsequently makes those two locale-aware kinds available as lint/editor diagnostics.

An M0-only implementation therefore starts no editor project-link session and performs no background producer scan merely because its internal orchestration contracts exist. Until L0 or another explicitly shipped editor capability activates the session, only the ordinary per-document workflows run.

The editor activates the project link session for diagnostic presentation only when at least one available L0/L1 linker-backed lint rule is enabled. An L0 rule requests only capabilities available from its M0 finding set; an L1 locale rule is unavailable until both M2 analysis and the L1 adapter exist. Disabled or not-yet-shipped rules do not cause source scanning merely because `resources.catalogs`, `messages`, or a warm cache exists.

E0 is the separately shipped JS/TS catalog-key completion milestone defined below. It may activate the same project link session without an enabled lint rule, but requests only definition, policy, and M1 typed-key-model capabilities. It never activates application-reference inventory or producer execution merely because completion is available. Later navigation features require their own named milestone and capability contract rather than inheriting E0 activation implicitly.

The session exposes only the read-only linker output required by each active editor capability. L0/L1 diagnostic presentation consumes `LinkOutcome` findings; E0 completion consumes the applicable installed M1 typed-key model and its editor-owned freshness token. Neither path consumes bundle plans. The session never executes exporters, registers build output, or applies `messages prune`; deployable generation and catalog mutation remain explicit CLI/build transactions.

### Session identity and source selection

Each resolved project/workspace root owns at most one editor link session. Its immutable input snapshot contains:

- the successfully resolved link-relevant projection of `resources` and `messages` plus a non-reused link-input generation;
- one resolved definition-source inventory from configured `resources.catalogs`;
- the resolved producer participation metadata required by 014's definition-only fast path and, only when the effective capability union requires application-reference input, one resolved reference-source inventory for every configured 014 producer;
- the current delivery graph or the exact 014 milestone-defined single-unit graph used before a bundler supplies a live graph;
- the current open-document override map; and
- the execution-derived `ScopeCompletenessTable`.

Project inventory uses the shared discovery and source-identity rules rather than enumerating files inside `intlify_linker`. Each configured logical definition source, and each configured logical reference source admitted by the effective capability union, participates in exactly one canonical physical-source group. The group identity, primary path, and ordered aliases are fixed before open-buffer substitution.

An unopened group is read once from disk and projected or produced normally. One open document mapped to any participating logical alias replaces the disk bytes for the complete group with its exact current Unicode buffer. Several mapped open documents are also one authoritative group snapshot when all their exact UTF-8 bytes are equal; the session captures every participating URI/version pair and requires all of them to remain current before installation or publication.

If two mapped buffers have different bytes, a URI cannot be mapped unambiguously to the group, or a watcher transition leaves ownership unsettled, the session selects no winner, reads no fallback disk snapshot, emits no definition or reference artifact for that group, and derives `Partial(OpenEditorWorld)` for every affected side and scope. The URI-local documents still retain independent diagnostics, caches, and formatting transactions.

When every enabled linker-backed rule is definition-only, the editor invokes 014's definition-only fast path. It retains the checked producer declarations and scope participation needed to derive the reference side of `ScopeCompletenessTable`, but it does not enumerate producer include matches, read configured external artifact files, read or parse application sources, invoke a producer, or consult the reference-artifact cache. Capability-based omission alone never derives `ProducerOmitted`; that reason remains reserved for a configured participant intentionally excluded from a request that otherwise requires it.

The reference-source selection rules below apply only while the effective capability union requires application-reference input. A later capability change from definition-only to reference-requiring resolves the complete reference inventory and schedules producer work through the ordinary generation and invalidation rules; opening an application source while the definition-only fast path is active does not independently produce reference evidence or widen completeness.

The initial 014 M0 producer admits only its fixed JavaScript and TypeScript source profiles. A selected `.vue` reference source is unsupported and produces the ordinary complete-source `ProducerFailed`; neither an editor language ID nor an open Vue buffer activates an implicit SFC frontend. The complete disk/open-buffer behavior for Vue SFC references is a next-phase producer task in 014's Deferred Follow-Up Notes and becomes editor behavior only when that frontend is promoted.

A file declared by exact path in `messages.producers.artifacts` follows that same override rule. One mapped open document, or several mapped documents with byte-identical current UTF-8 contents, replaces the complete artifact-file bytes. The editor decodes and validates that one selected snapshot through the ordinary bounded 014 artifact contract and captures every contributing URI/version. Successful decode is the authoritative configured snapshot for this editor session and can complete that producer participant. The editor does not regenerate the artifact, inspect its upstream language sources, or demand freshness evidence beyond the same producer semantic claim accepted for the configured disk snapshot.

An invalid selected snapshot is `ProducerFailed` and never falls back to stale disk bytes. Claiming live documents with unequal bytes, an unresolved URI-to-source mapping, or a watcher transition make the reference side `Partial(OpenEditorWorld)` until one authoritative snapshot exists. Closing every override returns selection to the current disk snapshot. A successfully decoded configured open-buffer snapshot is not `ExternalArtifactUnverified`; that reason remains for an ad hoc overlay, cache, package artifact, or integration value outside the exact configured participant contract.

An open document whose path newly matches a linker-participating project `resources.catalogs` definition before inventory refresh may contribute present definitions for advisory analysis because that project definition supplies the required exact scope and locale binding. Its affected definition side remains `Partial(OpenEditorWorld)` until the refreshed inventory admits one unambiguous current project source; merely observing the buffer cannot upgrade that side to closed.

An editor-overlay-only catalog, standalone `.mf2` document, untitled document, or other document outside every linker-participating project catalog definition has no project-authoritative scope and locale binding. It remains available to the ordinary per-document extraction, parsing, linting, and formatting workflows, but it never produces a `MessageDefinitionArtifact` and does not by itself change linker completeness.

An open application source outside the current reference inventory may contribute present reference evidence before inventory refresh only when its project path newly matches an enabled 014 producer include, its suffix selects an admitted source profile, and the configured recognizers supply the exact `kind`, `scope`, `domain`, and `keySyntax` values required by that producer. The built-in M0 editor orchestration separately assigns the exact delivery-unit ID `["main"]`; it is not read from or inferred by a recognizer, source path, or catalog overlay. The editor produces the ordinary checked reference artifact from the current buffer, admits its evidence for advisory analysis, and marks every affected reference scope `Partial(OpenEditorWorld)`. The buffer alone cannot prove that source enumeration, alias grouping, or the rest of the configured producer world is complete.

A newly observed path that matches the enabled producer include but has an unsupported source suffix produces no provisional artifact or producer error before refresh. It still marks every scope bound by that producer `Partial(OpenEditorWorld)` because the inventory has discovered a possible participant whose canonical physical group and profile failure have not yet been established. A path that does not match the include remains outside the linker world and changes no completeness state. Catalog overlay membership never supplies application-reference scope, domain, or delivery-unit values.

After refreshed inventory admits one unambiguous canonical project source, that source becomes an ordinary configured participant and may contribute to `Closed`. An admitted physical source group whose suffix is unsupported fails completely through `message_artifact_failed` with `details.kind: "producer_failed"` and evidence `stage: "profile"` plus `reason: "unsupported_source_suffix"`. It emits no reference artifact, does not fall back to disk, another profile, or a stale artifact, and derives `Partial(ProducerFailed)` for every scope bound by the producer. A selected `.vue` source follows this M0 path. A source that does not match the enabled producer include remains outside the inventory and produces neither an artifact nor a completeness change.

No first-opened or last-opened winner is selected when claiming live documents have unequal bytes, when a logical-to-project-source-group mapping is unresolved, or while a watcher transition leaves inventory membership unsettled. The affected side becomes `Partial(OpenEditorWorld)` until one authoritative current snapshot exists. Byte-identical claims are not a winner selection: they represent the same exact analysis input, and every claiming document version participates in the freshness gate. Existing URI/version isolation for per-document parsing and formatting remains unchanged.

### Completeness derivation

An open buffer does not by itself make a side partial. A scope side is `Closed` when inventory enumeration completed, every configured physical-source group or exact artifact participant has one authoritative current disk-or-buffer snapshot, every required extraction or producer run completed successfully, and no editor-only or unresolved participant affects that scope. One authoritative buffer snapshot may be represented by several byte-identical open documents only when all their versions are captured for freshness.

The session derives partial reasons through the 014 precedence contract. A definition read or extraction failure uses `SourceFailed`; an omitted definition uses `SourceOmitted`; a producer failure or omission, including invalid bytes in the selected configured external-artifact snapshot, uses `ProducerFailed` or `ProducerOmitted`; unverified non-configured external artifacts use `ExternalArtifactUnverified`; and an otherwise complete but intentionally unsettled editor world uses `OpenEditorWorld`. One successfully decoded open buffer, or a byte-identical set whose complete URI/version set is captured, has the same closure authority as its selected disk snapshot. Raw configuration, an artifact outside that inventory proof, unequal claiming buffers, unresolved ownership, or a successful result for only the visible files never asserts closure.

For the definition-only fast path, reference-side completeness follows 014's capability-aware derivation from checked producer participation metadata. It does not require a source inventory or producer execution and does not mark a configured producer omitted merely because the enabled rules requested no reference records.

The editor passes the complete checked table into the ordinary stateless `intlify_linker::link` operation. It relies on 014 to retain present-definition ambiguity and reference evidence, suppress absence-dependent findings for partial sides, emit `degraded-analysis`, and withhold deployable plans. The editor does not reproduce those gates locally and ignores any `bundle_plans` value because generation is outside this workflow.

### Scheduling, caching, and publication

A document change rebuilds only the extraction or producer artifact for sources whose selected bytes changed and then relinks affected project input. The first implementation may call the full stateless `link` operation over reused verified artifacts; an incremental index is an internal optimization and must remain result-equivalent to that full call for the same input snapshot.

Every link job captures the link-input generation, inventory generation, selected-source generation, and exact versions of participating open documents. The link-input generation advances only when the complete checked `LinkRequest` or another linker input in the session snapshot changes. A target name, exporter selection, output root, `eagerLocales`, or another export/registration-only setting does not advance it because the editor neither exports nor registers output and those fields are absent from `LinkRequest`.

Completion first constructs one canonical per-URI publication map. It installs that map into the session only if every captured token still matches, then publishes document updates in canonical URI order. A stale job installs and publishes nothing; the event that changed the generation clears affected prior findings and schedules the replacement. Slow project linking never delays independent per-entry parser/lint diagnostics or formatting responses.

Each resolved scope with an installed M1-or-later typed-key model also owns an opaque, non-reused model-generation token. Installing, replacing, or clearing that checked model advances the token, regardless of whether the replacement exposes equal key bytes. Retaining the already installed model without another installation retains the token. Completion uses this editor-owned token only as a freshness guard; it is not serialized into 014 artifacts or treated as linker semantic input.

The LSP projection uses the one canonical primary evidence location selected by 014 and maps remaining source-backed evidence to `relatedInformation`. Definition evidence uses the 013 entry map; reference evidence uses producer provenance and the current source selection. One `LinkFinding` becomes one diagnostic even when evidence spans several files.

A finding without a truthful source URI and span is retained in the editor session as a workspace finding. An integration with a dedicated workspace-diagnostic or project-problem surface may present it there. Standard per-document `publishDiagnostics` omits it rather than attaching it to the config file, document start, or an arbitrary related entry. It is never converted into a popup or operational error merely because the client lacks a workspace finding surface.

### Operational failures and recovery

Project-link operational failures remain separate from parser, semantic, lint, resource, and `LinkFinding` diagnostics. Each failure retains the stable operational representation defined by the subsystem that emits it: shared input, path, and transport contracts remain owned by 005, 006, and 007; resource extraction and adaptation errors remain owned by 013; and producer, artifact, and linker errors remain owned by 014. The editor consumes those stable codes, kinds, paths, and bounded details without wrapping them in an editor-specific error schema, constructing the CLI JSON envelope, or publishing an operational failure through `textDocument/publishDiagnostics`.

This document owns only the editor operation around those errors: root-scoped episode state, canonical fingerprints, notification and log policy, retry scheduling, and recovery.

Each project/workspace root owns one project-link operational episode independently of the configuration-failure episode. Its active value is the complete canonical ordered set of unresolved operational errors for the installed project-link input. An incremental attempt replaces the state for every participant it re-evaluates and retains errors for unchanged participants; it cannot report recovery merely because the latest job touched only another source. Its notification fingerprint is the canonical sequence of each error's stable `code`, normalized optional `path`, and canonical stable `details`. Human-readable messages, dependency text, source contents, timestamps, backtraces, and worker completion order do not participate.

The first non-empty fingerprint in an episode, and a later transition to a different non-empty fingerprint, sends one `window/showMessage` notification of type `Error` with concise actionable prose. Every such state transition also emits one `window/logMessage` error entry with stable codes and safe path/reason context. Repeated observation of the same fingerprint during requests or an explicit retry sends neither another popup nor another state-transition log. A notification discovered before the protocol lifecycle permits window messages is queued until initialization completes.

A producer failure, including `message_artifact_failed` with `details.kind: "producer_failed"`, may coexist with a sound present-world `LinkOutcome`. The orchestration omits every artifact invalidated by that failure, derives the required partial completeness, and may invoke the linker with every successfully admitted artifact. If linking succeeds, the editor publishes that outcome's sound findings, including its completeness evidence, through the ordinary generation and freshness gates. The operational failure remains active in parallel; it is never converted into a linker diagnostic or hidden merely because advisory analysis succeeded.

A non-invariant `LinkOperationalError` produces `message_link_failed` and no `LinkOutcome`. That attempt installs no linker cache value and publishes no new linker findings. When the triggering source, inventory, configuration, or open-buffer event advanced a captured generation, its ordinary invalidation already cleared the affected prior linker publication, so the failed generation remains empty. An explicit retry with every generation unchanged does not clear or replace an already installed complete outcome for that same generation; failure installs nothing, and the still-current publication remains visible.

Source, inventory, authoritative-buffer, or link-relevant configuration changes schedule a new attempt through their ordinary generation rules. An explicit project reanalysis request may also retry the same generation. Failure itself does not start an unbounded retry loop. A configuration failure continues to use its separate transactional episode and prevents project-link scheduling until a valid configuration is installed.

When a current-generation attempt has replaced its affected participant state and the complete root error set is empty, or when a validated configuration change deactivates the project-link session, the integration clears the active project-link episode. Recovery emits one informational `window/logMessage` entry and no success popup. A later recurrence, including the same fingerprint after recovery, starts a new episode. One root's episode never suppresses publication, notification, retry, or recovery in another root, and ordinary URI-local diagnostics and formatting continue while a project-link episode is active.

## Formatting Edits

Formatter core APIs return formatted message text rather than LSP `TextEdit` objects.

The editor integration finds the containing message entry and calls whole-message formatting. It looks up or creates the entry's paired `SourceStore` / `ParseResult` artifact once, then calls the formatter's workspace-internal `format_parsed` path for both standalone and catalog entries. This path neither reparses a cache hit nor serializes it through a Binary AST snapshot. The initial integration replaces the whole containing message range rather than computing a minimal diff. For standalone `.mf2` files, that range is the whole document and the replacement is the formatted unframed message followed by exactly one `LF`, with no BOM, matching the Phase 3B [File Framing](./007-ox-mf2-phase-3b-formatter-design.md#file-framing) write contract. The standalone path compares those framed replacement bytes with the original document, so a leading BOM, a missing final newline, or a final `CRLF` produces an edit even when the unframed message text is already formatted.

For a catalog host document, the editor observes every artifact entry through one resource-owned `CandidateMessageAdmission` in raw entry order after ordinary formatting completes: a successful selected writable output contributes its complete byte length, while an unselected, diagnostic-bearing, or read-only entry contributes its original message. The editor retains an admitted writable string, including byte-identical output, for candidate construction. A formatter operational error wins over a prospective admission error and returns no edits; otherwise failed admission returns its request-scoped resource error without invoking write-back. Successful admission supplies the retained outputs as `FormattedEntry` input to the shared `build_and_validate_write_back` boundary in [013-ox-mf2-resource-catalog-adapter-design.md](./013-ox-mf2-resource-catalog-adapter-design.md); catalog entries never receive standalone file framing. `WriteBackOutcome::Unchanged` returns no edits. On `Changed`, the editor discards the validated candidate artifact only after success and converts its resource-owned `RawReplacement` values into protocol edits over their original raw value spans. Those spans satisfy the resource contract's stronger edit-disjoint rule, including its zero-length insertion restrictions, so the protocol result does not depend on edit ordering. The deferred bounded formatter path may later avoid materializing an oversized output before this admission check without changing the resource error or write-back contract.

This full-candidate validation remains mandatory for a single-entry format request and for large catalogs. Editor settings and size thresholds cannot disable it. The parsed-artifact reuse path does not weaken candidate validation; a future incremental implementation is valid only when it proves the same complete identity and value invariants and passes the shared equivalence fixtures. Editor latency is tracked by the dedicated single-entry catalog benchmark in the resource design. A candidate `resource_limit_exceeded` follows the request-scoped, no-publication behavior above; every other candidate-validation failure follows its typed operational boundary. No candidate failure installs edits or mutates the retained diagnostic and extraction state.

Document formatting for a catalog host document evaluates entries independently and admits only selected writable outputs:

- Entries whose extracted text has parser diagnostics are skipped as per-entry no-ops, matching the strict invalid-syntax contract in [007-ox-mf2-phase-3b-formatter-design.md](./007-ox-mf2-phase-3b-formatter-design.md). Other entries still format.
- Read-only entries still use ordinary formatter evaluation so parser diagnostics remain available, but a clean formatted output is discarded and the original message is admitted. Its discarded formatted length is not a candidate resource failure because that output cannot participate in write-back.
- The integrated resource method produces one replacement for each byte-different selected writable entry only after the raw-order limit checks and complete candidate validation succeed. Byte-identical entries produce no replacement or edit.
- Formatting edits never touch host syntax outside raw value spans: no key reordering, no indentation or quoting changes outside the message value, and no host layout normalization. Host-format styling belongs to host formatters, so keeping MF2 edits inside value spans lets an MF2 formatter and a host-format formatter coexist on one document without conflicting edits.

A range formatting request first resolves both protocol positions against the current document version using the negotiated position encoding and converts them to one ordered host-document UTF-8 byte range. A position outside the document, a reversed range, or a UTF-8/UTF-16 position that splits a Unicode scalar is rejected as a protocol-level invalid-params failure before entry selection; the editor integration does not clamp, swap, return a successful empty edit merely for malformed input, or call parser, formatter, or resource APIs. A valid request formats selected writable entries through ordinary whole-entry formatting, admits their successful outputs while observing originals for every unselected entry, and passes only admitted outputs to `build_and_validate_write_back`, which validates the resulting complete candidate host document before returning any edit. Unselected entries retain their original messages during candidate total calculation and validation. A non-empty raw value span uses normal half-open intersection with the converted host UTF-8 range. A zero-length span at host position `p`, such as an explicitly paired empty XLIFF element content span, is treated as a point and is selected when `range.start <= p <= range.end`; an empty caret range at exactly `p` therefore selects it. This point rule does not expand the replacement span into surrounding host syntax.

At request start, the editor integration captures one immutable request snapshot containing the document URI, the exact client-supplied protocol version currently installed for that URI, the source text, the active classification/extraction artifact, an opaque mapping-generation token, and the applicable root config-generation token. Both tokens are server-local and never reused for a later installed state during the server process. The mapping token changes whenever the URI is opened or reopened, closed, reclassified, re-extracted, or receives a replacement mapping because of a document or membership change. The config token changes after any semantically different resolved configuration is installed for the root. A source hash or byte comparison is not a substitute for the protocol version or either generation token.

Immediately before returning a successful standard document- or range-formatting response, the editor integration looks up the URI again and requires exact equality of the installed protocol version, mapping-generation token, and root config-generation token. It also verifies that every selected entry handle still belongs to the captured artifact. A closed document, an incremented version, a same-bytes edit that nevertheless changed the version, a classification or mapping replacement, a formatter-option change, or a missing entry makes the request stale. The integration then returns an empty `TextEdit[]`, publishes no operational editor error, and does not retry formatting automatically. Position conversion, entry selection, formatting, and candidate validation always use the captured request snapshot rather than mixing old and current document or configuration state.

The final state comparison and response construction occur under one consistent document-state read boundary. A standard LSP formatting response cannot carry a document version, so this server-side check cannot prevent a client-side edit made after the response leaves the server; client document lifecycle and application ordering remain the protocol integration's responsibility. Future operations that return `WorkspaceEdit` may use versioned `TextDocumentEdit`, but the initial standard formatting handlers do not replace themselves with a custom command solely to obtain that guard.

Editor integration tests must cover unchanged version and generations returning edits; a changed version with identical bytes returning `[]`; close and reopen with a reused client version returning `[]` because the mapping generation differs; configuration-driven reclassification or re-extraction at the same client version returning `[]`; a formatter-only config change returning `[]` because the config generation differs even when the mapping does not; a selected entry disappearing before return; and no automatic core re-invocation after any stale outcome.

True range-only formatting and minimal-diff formatting are deferred. A selection inside an MF2 message should initially format the containing message rather than requiring range-local formatting from the formatter core.

## Span and Position Conversion

Core parser, snapshot, semantic, formatter, and linter APIs use message-local UTF-8 byte spans as their canonical location model. Editor integrations convert spans through a fixed pipeline at the editor integration boundary:

1. Core APIs return message-local UTF-8 byte spans over the entry message text.
2. The entry offset map, per the mapping contract in [013-ox-mf2-resource-catalog-adapter-design.md](./013-ox-mf2-resource-catalog-adapter-design.md), converts message-local spans to host-document UTF-8 byte spans.
3. The editor integration converts host-document UTF-8 spans to the editor-facing position encoding.

For standalone `.mf2` documents, step 2 uses the file-framing-aware document map. A removed leading UTF-8 BOM shifts mapped byte positions by its three-byte length, while a removed trailing newline stays outside the message range; when no framing was removed, the map is the identity over the message body. If framing leaves an empty message, the map retains an independent empty-message anchor at the first message-body byte position: after a removed BOM and before a removed trailing newline. Mapping the only valid message span, `0..0`, returns that exact zero-length host span even when adjacent raw-only framing segments coalesce. The editor-facing default is UTF-16 positions; integrations that negotiate a different LSP `positionEncoding`, UTF-8 or UTF-32 in LSP 3.17 and later, still perform the conversion at the same boundary. The parser, formatter, and linter cores never perform position encoding conversion.

## Configuration Sources

Editor integrations normalize project configuration and editor-specific settings into the same resolved formatter and linter configuration models used by CLI workflows. They also consume the project-authoritative `ResolvedResources` and 014 resolved `messages` configuration produced through the 006 validation order.

The initial cross-editor configuration model accepts exactly four source classes:

1. built-in formatter, linter, and editor integration defaults
2. the unified project configuration discovered and validated through the Phase 3A project-config contract
3. optional editor settings normalized from LSP `initializationOptions` during server initialization
4. optional effective dynamic editor settings delivered through `workspace/configuration` or `workspace/didChangeConfiguration`

An editor client may derive the fourth source from product-specific user, workspace, or workspace-folder scopes, but it resolves those scopes before the settings cross the integration boundary. The server consumes one normalized effective settings object for the applicable project/workspace root and does not model VS Code, another editor's preference hierarchy, or a vendor-specific settings-file format as separate configuration sources. A client without dynamic-configuration support may use initialization options for the lifetime of the server session.

Per-document option overrides, CLI flags, environment variables, editor-specific files read directly by the server, and direct reuse of a client's raw user/workspace settings objects are outside the initial model. `.editorconfig` remains governed by the formatter design and is not an editor-integration configuration source in the initial formatter surface. Editor-only catalog membership uses the normalized ad-hoc overlay below rather than introducing another source class. Source precedence for overlapping non-membership formatter, linter, and editor options is defined separately below; reload application is transactional under the failure-class rules above, so an invalid or internally failed dynamic update does not partially replace the last successful state.

For non-membership formatter, linter, and editor-integration options, resolution is field-by-field in this highest-to-lowest precedence order:

1. effective dynamic editor settings
2. LSP `initializationOptions`
3. unified project configuration
4. built-in defaults

An absent field falls through to the next source. JSON `null` is not an unset or inheritance marker and is rejected unless a future individual option explicitly defines `null` as a value. Each dynamic notification or `workspace/configuration` response replaces the complete previous dynamic layer for its applicable root; it is not a merge patch. An empty normalized object therefore clears all dynamic overrides and reveals initialization, project, or default values. The server resolves a fresh immutable formatter/linter configuration from all four layers into an uninstalled candidate and installs it atomically only after every applicable validation stage below has completed.

Every lower-precedence source is still validated even when a higher-precedence source supplies the same field. In particular, an invalid project configuration cannot be hidden by initialization or dynamic editor settings. The editor integration may retain internal field provenance for logging or debugging, but provenance is not passed into formatter or linter core option types and is not a diagnostic category.

`resources` and `messages` do not participate in the field-by-field editor override order. The unified project configuration is authoritative for project inventory, scope/locale binding, producer recognizers, locale/fallback policy, roots, and delivery targets. Initialization options and dynamic editor settings cannot claim closed completeness, change link semantics, or add a second producer configuration. The resource overlay below contributes only editor-local catalog membership and host-format classification for per-document resource and message processing. It carries no scope or locale binding, never creates a linker definition, and does not change linker completeness.

The editor resolves the effective linter rule set against that authoritative project configuration before installing a new root state. If an initialization or dynamic linter override enables any 008 linker-backed rule while the project `messages` section is absent, the update fails through the root-scoped configuration channel under the same `linker_messages_required` semantic contract. It never starts a definition inventory, source producer, cache lookup, or linker session. A project-config activation retains 008's exact JSON Pointer evidence; a non-project activation reports the owning normalized editor-setting location without fabricating a project-config pointer. In either case, the transactional failure behavior below keeps the last successfully installed configuration and artifacts.

Some 013/014 configuration contradictions require concrete catalog assignments, physical-source groups, or successfully extracted definition entries and therefore cannot be decided by the structural configuration pass alone. When a different candidate configuration activates or changes project-backed linker inputs, the editor constructs the minimum candidate definition inventory and extraction state needed to evaluate `catalog_format_conflict`, `locale_capture_mismatch`, `catalog_locale_not_production`, `catalog_binding_conflict`, `catalog_scope_key_domain_conflict`, and `scope_key_domain_mismatch` in the shared total late-gate order. This work uses the candidate configuration and the same authoritative disk-or-open-buffer snapshot rules as an installed session, but it remains isolated from the installed root state and starts no reference-source scan. It retains successful resource extractions without projecting any `MessageDefinitionArtifact` until the project-global catalog and recognizer/root domain gates have succeeded.

A configuration-derived contradiction discards that candidate state and enters the root-scoped configuration-failure episode without installing any part of it. In contrast, an enumeration, metadata, read, extraction, freshness, or projection failure attributable to a candidate source is not a configuration-validation failure. It is retained as the ordinary execution evidence for the candidate generation, contributes the applicable `Partial(SourceFailed)` state after installation, and cannot manufacture a domain mismatch from unavailable entries. Thus a valid configuration may be installed with partial execution evidence, while a proven configuration contradiction leaves the complete prior configuration and derived state in force.

Catalog membership and host-format classification are the deliberate exception to ordinary field precedence. Project `resources.catalogs` state remains authoritative: absence or a non-empty policy with no matching include permits editor fallback, while an explicit empty policy, an explicit project exclusion, or a project match blocks it. The overlay cannot override, overlap with, or conflict with a project-selected format because it is path-resolved only in the fallback-eligible states. These routing rules apply after both source shapes have been validated.

Every integration maps its product-specific editor setting into this common normalized ad-hoc catalog overlay shape before classification:

```jsonc
{
  "catalogs": [
    {
      "include": ["locales/**/*.json"],
      "exclude": ["locales/generated/**"],
      "format": "json"
    }
  ]
}
```

`catalogs` defaults to an empty array. Overlay field absence and an explicit empty array are equivalent because this layer is additive and has no disable-all meaning of its own. Each entry requires a non-empty string `include` array; `exclude` is optional and defaults to an empty array. Patterns use the same project-root-relative, slash-normalized glob semantics as project `resources.catalogs`. A path without glob metacharacters is an exact document-path match, so the model does not need a second per-document URI variant. `format` is optional and accepts only a canonical id for an adapter shipped in the current release. When omitted, classification uses the matched document's extension; an extension-ambiguous document therefore requires an explicit `format`. `scope`, `locale`, and every other project-only catalog field are unknown overlay fields and are rejected rather than treated as advisory linker configuration. The initial overlay applies only to `file` URI documents whose paths are representable relative to `projectRoot`; untitled and other non-file documents are not catalog targets.

This is a normalized cross-editor contract, not a mandated product-facing setting key. VS Code extensions, LSP initialization options, or other integrations may expose an idiomatic outer key, but they only map that product-specific value into the common object. The resource-owned overlay validator performs the exact object, field, shipped-format, and resource-glob validation and produces an opaque `ResolvedCatalogOverlay`; integrations do not compile or resolve the inner shape themselves. Its normalized RFC 6901 pointers are rooted at `""` and `/catalogs`, independently of the product-facing outer setting name.

Catalog opt-in and extraction scope configuration are owned by [013-ox-mf2-resource-catalog-adapter-design.md](./013-ox-mf2-resource-catalog-adapter-design.md) as the `resources` section of the unified project config. The unified loader validates that section through the resource-owned validator and produces `ResolvedResources`; the resource-owned overlay validator independently produces `ResolvedCatalogOverlay` only after the normalized editor value succeeds completely. For each eligible file URI, the editor calls `ResolvedResources::resolve_path_with_overlay` once rather than resolving and recombining the layers itself. Its `LayeredCatalogMatch` returns project or overlay provenance plus an opaque assignment.

Project precedence is authoritative and stateful. Project `PolicyAbsent` or `Unmatched` permits overlay fallback. Project `PolicyEmpty` or `Excluded` yields no catalog target and blocks fallback. Project `Matched` returns the project assignment without resolving same-path overlay definitions. Only the fallback branch can report an overlay-internal assignment conflict; there is no cross-layer assignment conflict. Invalid project configuration is reported before overlay validation and cannot be bypassed by editor settings, and structural overlay validation still runs even when a particular document will not use overlay fallback.

Overlay structural failures and same-layer assignment conflicts reuse `config_validation_failed` in the root-scoped editor configuration channel; they are never document diagnostics. Overlay structural details use the resource validator's normalized pointer and omit a filesystem config path. A `catalog_format_conflict` fingerprint uses `targetPath` and the resource-provided `assignment` and `conflictingAssignment` objects with exact `source: "project" | "overlay"` and zero-based `definitionIndex` fields; both references always name the same layer. A config path is retained only for a project-layer conflict. Reload remains transactional, so neither an invalid overlay nor a project- or overlay-layer conflict replaces the last successful overlay, classification, extraction artifact, mapping, or diagnostics.

Changing either source re-runs editor document classification and extraction, with the invalidation consequences described below. The integration should identify an ad-hoc-only document as editor-local rather than CI-covered; once an equivalent project match appears, the classification becomes project-owned without running extraction twice. Editor-only membership is never implied to exist in CLI or CI.

When a previously matched catalog document becomes project `PolicyEmpty` or `Excluded`, or otherwise has neither a project match nor an eligible overlay match, it becomes an ordinary non-target document rather than an unsupported-input error. The editor drops its resource extraction artifact, mapped diagnostics, formatting state, and resource-specific cache entries, then publishes an empty current-generation diagnostic set so stale findings disappear. `unsupported_input_file` is a CLI operand error and is never published for an editor document merely because catalog classification was removed.

Configuration loading, validation, and internal reload failures are root-scoped operational editor errors. They are not mixed into parser, semantic, formatter, linter, resource input, catalog, or linker findings and follow the last-successful-state transaction above.

For each project/workspace root, the integration retains at most one active structured configuration error and its notification fingerprint. The fingerprint is the stable operational `code`, normalized optional `path`, and canonical stable `details` value; human-readable `message`, dependency text, timestamps, and backtraces do not participate. The first failure in an episode, or a later failure with a different fingerprint, sends one LSP `window/showMessage` notification of type `Error` with concise actionable prose. Repeated reloads, document requests, and open documents that observe the same active fingerprint do not send another popup.

Every observed configuration failure also emits one `window/logMessage` error entry containing its stable code and safe path/reason context. The log message does not expose source contents, arbitrary dependency debug output, panic payloads, or a backtrace. If the server discovers the failure before the protocol lifecycle permits window notifications, it queues the one popup and log entry until after initialization rather than converting the condition into an initialize-response schema or document diagnostic.

An active failure installs no configuration, advances no config, inventory, source-selection, or mapping generation, invalidates no artifact, and starts no replacement diagnostic or linker job. The last successfully installed configuration and artifacts remain available; its diagnostic publication remains visible only while the document version and mapping stay unchanged. A later document, membership, or mapping change publishes the mandatory empty set without treating the failed configuration as successfully reloaded, and no replacement diagnostics are scheduled until recovery. Formatting requests for an affected root return an empty `TextEdit[]` while the failure is active, even though retained configuration and artifacts remain available for recovery and comparison.

The next successful validated reload atomically installs its resolved state, clears the active error, applies the normal selective invalidation matrix, and records one informational recovery entry through `window/logMessage`. It does not show a success popup. A later recurrence, including the same fingerprint after recovery, begins a new failure episode and may notify once again. Roots maintain independent error episodes, so one broken workspace root does not suppress or duplicate another root's notification.

## Artifact Cache and Invalidation

The per-document workflow may retain three separately owned artifact classes once semantic APIs are exposed:

- extraction artifacts per host document version: classification results, entry lists, and offset maps
- message-level syntax artifacts per entry: the original paired `SourceStore` / syntax-only `ParseResult` and its materialized parser diagnostics
- optional message-level semantic artifacts per diagnostic-free entry: a `SemanticModel` derived by `build_semantic_model` from that exact owner/result pair

The semantic artifact is never inserted into `ParseResult`. A future semantic cache follows the separate compatibility-key, capacity, and failure rules in [ox-mf2-parse-artifact-cache.md](./ox-mf2-parse-artifact-cache.md#semantic-artifact-separation). Decoded snapshots are retained only when a separate serialized-transport consumer actually needs them.

Message-level artifacts should follow the cache-key discipline in [ox-mf2-parse-artifact-cache.md](./ox-mf2-parse-artifact-cache.md), using the entry key as the message id and the document URI as part of the namespace. Because that cache key includes the exact source bytes, entries whose message text is unchanged across host document versions can hit the cache even when unrelated parts of a large catalog changed. Formatter requests pass references to that owner/result pair through `format_parsed`; they do not call `parse_source`, snapshot encode, or snapshot decode on a cache hit. Extraction artifacts remain version-specific.

An active project link session additionally owns four separately invalidated classes:

- one resolved definition/reference inventory snapshot with its non-reused inventory generation;
- definition artifacts retained only while the complete compatibility identity owned by the 013 extraction and 014 definition-artifact contracts still matches;
- reference artifacts retained only while the complete semantic cache identity owned by 014 still matches; and
- a `LinkOutcome` cache keyed by the complete checked `LinkRequest`, including resolved policy, delivery graph, the M0 built-in empty scope-mapping table, and `ScopeCompletenessTable`.

009 defines editor-side invalidation events, not a second spelling of either artifact key. In particular, the exact reference-artifact cache tuple, including its cache-schema revision, artifact and producer revisions, primary and logical-alias identities, grammar profile and source goal, recognizers, source fingerprint, and delivery unit, remains normative only in 014.

An open document override changes selected source bytes and source-selection generation, not the canonical project source identity. Reusing a verified extraction, producer, or linker artifact must follow the owning 013/014 compatibility keys; editor URI, lint severity, reporter choice, and worker completion order do not weaken those keys.

Each project/workspace root owns an immutable resolved configuration plus an opaque, process-local config-generation token. Its active link session separately owns the link-input generation defined above. A reload first resolves all source layers and completes the structural and path-independent cross-section checks in an uninstalled candidate, then compares that resolved candidate semantically with the installed configuration. An equal result performs no candidate inventory or cache work and advances neither token.

A different candidate performs any required isolated late-gate work defined above before installation. The candidate has no installed generation, cannot publish diagnostics or linker results, and cannot invalidate installed artifacts. If every applicable configuration gate succeeds, the editor atomically installs the resolved configuration together with the candidate inventory, execution evidence, and compatible derived artifacts needed by that root state, using one fresh non-reused config-generation token. Candidate source failures remain ordinary installed execution evidence rather than turning the reload into a configuration failure. The link-input generation advances only when the installed change alters the complete checked `LinkRequest` or another linker input in the session snapshot. Affected open documents are then processed according to this dependency matrix:

- A `resources` membership, host-format, scope, locale-binding, or normalized editor catalog-overlay change reclassifies every potentially affected open document in the root. It discards version-specific extraction artifacts, offset maps, mapped diagnostics, and pending formatting candidates and advances the document mapping generation when a classification or installed mapping changes. It also advances project inventory or definition-projection generation, invalidates affected definition artifacts and `LinkOutcome`, and adopts the validated candidate inventory and definition state before scheduling any remaining replacement work. An affected side stays partial when candidate execution evidence or still-pending non-gate work prevents closure. If a document is no longer an MF2 or catalog target, the editor integration publishes an empty diagnostic set. Exact-source message parse or semantic artifacts remain eligible for reuse only through their complete cache keys; no cached host mapping survives.
- A `messages` policy, configured-root, locale, M2-or-later fallback, resolved placement policy, or delivery-graph change advances link-input generation and invalidates `LinkOutcome` while retaining compatible definition and reference artifacts. Raw omission and an explicit `duplicate` placement resolve to the same M0–M3 typed policy and therefore, under the semantic resolved-configuration comparison above, advance neither the root config generation nor link-input generation. A producer include, recognizer, producer identity/revision, or source-to-scope/domain binding change also rebuilds the affected reference inventory and reference artifacts. None of these changes forces resource extraction or source parsing unless its owning artifact compatibility identity changes. The built-in editor has no configurable scope mapping in M0; a future package-composition mapping field adds its own invalidation rule when that configuration contract lands.
- A target name, exporter, output root, `eagerLocales`, or another export/registration-only target option is still validated and installed as part of resolved configuration, so it advances the root config generation. It does not advance link-input generation, invalidate `LinkOutcome`, clear linker findings, or schedule replacement linking. A future target option that changes `LinkRequest` follows the preceding row instead of this one.
- A formatter-option change invalidates cached formatter output or candidate state and makes in-flight formatting requests stale through the root config generation. It does not discard classification, extraction, offset maps, parser artifacts, semantic artifacts, or lint diagnostics.
- A successfully validated linter-option change invalidates resolved lint-result caches and schedules one new diagnostic job for each affected open target. Parser, extraction, offset-map, and parser-owned `SemanticModel` artifacts remain reusable when their own keys still match. A severity-only change for already produced linker kinds reprojects the retained `LinkOutcome`; changing the enabled linker capability union may start, narrow, or stop project inventory and producer work under the 008/014 gating contract. The static capability table is resolved before work. Enabling a linker-backed rule without project `messages` fails before installation and therefore follows none of these invalidations. Rule order, reporter choice, and severity never enter artifact or linker cache identity. The last complete publication may remain until the new generation publishes atomically, as defined above.
- A future parser-result-affecting option must participate in the parse artifact `CacheKey`; changing it causes natural misses or explicit removal for the affected namespace. A semantic-result-affecting option belongs only to a separate semantic artifact key and must not fragment syntax cache entries. Formatter- or linter-only settings must not be added to either artifact key merely to force broad invalidation.
- Editor scheduling or presentation settings that do not affect extraction, core results, mapping, or protocol positions update their owner without evicting core artifacts. The root config generation still prevents an already running request from installing state computed from a superseded resolved configuration.
- A watched project-file create, remove, rename, or classification-relevant metadata change advances inventory generation before asynchronous rediscovery. Affected sides remain partial until the refreshed inventory and all required participants complete; no old closed table survives the generation change.

A document text/version change always publishes an empty ox-mf2 diagnostic set, drops its version-specific extraction and mapping state, advances its mapping generation, and schedules new diagnostics. If it participates in project link input, it also advances source-selection generation, clears affected linker publications, rebuilds only its selected definition or reference artifact, and schedules relinking. Message-level parse artifacts for entries whose exact message bytes and complete cache keys remain equal may survive and be reused across host-document versions. A failed configuration load, validation, or internal reload installs nothing, advances no generation, invalidates no cache, and leaves the last successful configuration in force; it leaves a diagnostic publication unchanged only when no document, membership, or mapping change has independently required the clear above.

## Shared Resource Layer Ownership

Catalog assignment, top-level format resolution, built-in adapter dispatch, host format extraction, mapping, and validated write-back are implemented once in the shared resource layer owned by [013-ox-mf2-resource-catalog-adapter-design.md](./013-ox-mf2-resource-catalog-adapter-design.md). Editor integrations must consume that layer rather than reimplementing catalog membership or host-format knowledge per editor. Message-to-raw mapping and re-escaping correctness is subtle and must not drift between the CLI and editor surfaces.

Reference resolution, locale/fallback resolution, completeness gates, reachability, and linker finding construction are implemented once by [014-ox-mf2-message-linker-design.md](./014-ox-mf2-message-linker-design.md). The editor session owns inventory, buffer substitution, scheduling, cache selection, generation checks, and LSP projection, but it must call the same stateless linker and cannot create an editor-specific semantic shortcut.

Editor-side glue stays thin. Document lifecycle, URI and version tracking, request orchestration, protocol result shapes, and position encoding remain editor integration code; catalog assignment, host-format knowledge, and linker semantics do not.

## Future Editor Features

The initial workflow does not require code actions, quick fixes, hover, completion, go-to-definition, rename, true range-only formatting, or minimal-diff formatting.

Future editor features should build on stable core concepts rather than adding LSP-specific state to parser, formatter, or linter crates. Quick fixes are a future editor-integration-owned feature. They may use stable diagnostic codes, configurable rule metadata, formatter output, and future rule suggestions, but the initial linter core does not expose a fix API. Style fixes should call formatter APIs rather than reimplementing formatting inside editor integrations or linter rules.

Message-local MF2 semantic features should build on future `SemanticView` exposure. Application-source and project-level message features should instead consume the checked 014 `LinkOutcome`, M1 typed-key model, and producer projection contracts. The editor must not create its own project analyzer in either case.

### E0 — JS/TS Catalog-Key Completion

E0 is the first explicitly named editor-feature milestone after the initial diagnostics and formatting workflow. It is owned by this document and depends on 014 M0's checked project configuration and JS/TS recognizer contract plus the M1 typed-key model and reverse projection. It does not depend on L0/L1 lint presentation, M2 locale/fallback analysis, M3 export, or any reference-artifact scan, and it does not make completion part of M1 itself.

An editor product exposes the E0 completion request surface only when that product explicitly includes E0 and its client supports completion requests. Product availability alone starts no project work. For one resolved root, E0 becomes an active project-session capability only while all of the following hold:

- the last successfully installed project configuration has a present valid `resources` and `messages` projection admitted through the coordinated 013/014 contract;
- at least one configured `kind: "lookup"` JS/TS recognizer names a scope with an explicit admitted M1 `coverageBaseline`;
- at least one authoritative open document has a normalized project-relative path matching `producers.js.include` and an exact suffix selecting an admitted M0 JS/TS source profile; and
- the editor product and connected client still expose the E0 request surface.

Activation schedules or retains only the definition inventory, checked policy, definition-side completeness, and M1 model work needed for those statically eligible scopes. It does not enumerate producer include matches beyond the already open document eligibility check, read application sources, load external reference artifacts, invoke `intlify_producer_js` as a reference producer, or consult its reference-artifact cache. The recovery-aware cursor query and reverse projection used by an individual request are not reference production.

Until one matching checked model is installed, a completion request returns `null`; it does not infer keys from open documents or run an unbounded synchronous fallback scan. Closing the last eligible document, removing the last qualifying lookup/baseline pair, losing the client capability, or replacing the configuration deactivates E0 for that root and clears its E0-only model publication under the ordinary generation gates. If L0/L1 or another separately shipped capability still needs the project session, that session remains active with the union of the remaining capabilities rather than being torn down.

Plain application-source key completion builds on the M1-or-later canonical typed-key model and locale-bound catalog configuration. It consumes a checked model for one resolved scope rather than reconstructing a key union from open documents, extraction artifacts, reference artifacts, or the definition union.

An absent `coverageBaseline` means that the resolved scope has no M1 typed-key model and therefore no key completion. This is an unavailable capability for that scope, not a configuration or linker error. The editor never infers a baseline from fallback order, locale order, catalog size, the first open document, or another locally observed key set.

The initial JS/TS completion integration is available only when all of the following conditions hold:

- the current document's normalized project-relative logical path matches the enabled `producers.js.include`, and its exact case-sensitive suffix selects one of the initial M0 JS/TS source profiles defined by 014;
- the cursor is inside the first-argument string container of a call expression whose exact callee matches one configured built-in JS/TS recognizer;
- that recognizer has `kind: "lookup"` and supplies the exact declared scope, domain, and `keySyntax`;
- ordinary scope mapping resolves the declared scope to exactly one scope with an admitted M1 typed-key model;
- the model contains the same canonical `CatalogKeyDomain`; and
- the current argument is an ECMAScript string literal or no-substitution template literal for which the editor can construct a checked value-preserving text edit.

This path/profile check uses the installed producer configuration and the current document identity. It does not enumerate the project-wide reference inventory or make the current document a reference-artifact participant. A path excluded by `producers.js.include`, an unsupported suffix, or a `.vue` source before promotion of the deferred Vue frontend receives no application-source key completion.

The exact cursor callee selects the recognizer binding. Several recognizers may therefore target the same scope while using different source-facing key syntaxes without making completion ambiguous. A cursor outside such a lookup call, inside a dynamic expression or substitution, inside a `kind: "set"` call, or in a source without an exact admitted binding receives no application-source key completion.

Cursor recognition calls the workspace-internal, recovery-aware `intlify_producer_js` [cursor-query boundary](./014-ox-mf2-message-linker-design.md#recovery-aware-cursor-query-boundary). The editor supplies the exact captured source snapshot, normalized project-relative path, UTF-8 cursor offset, and installed resolved JS producer configuration. It does not select a parser dependency, reconstruct a source profile or recognizer grammar, inspect an AST, or match a callee independently.

The producer-owned cursor query may inspect recovery output only when it can prove one unambiguous enclosing call expression, exact callee, first-argument string or no-substitution-template container, present delimiters, and checked content replacement range for the current cursor. A syntax error outside that local structure may be tolerated. If an error intersects that structure, a delimiter or range is missing, a dynamic or substituted expression is involved, or recovered candidates disagree, the API returns `Ok(None)` and the editor returns no completion.

For a bounded-unambiguous source profile, the producer API may inspect module- and script-goal recovery results, but returns a cursor context only when every result that supplies one identifies the same callee, checked recognizer binding, container kind, delimiter, and replacement range. One recovery result cannot win by attempt order. The editor observes only the resulting immutable context and never the recovery results. No recovery result contributes project reference evidence, changes completeness, or enters the producer artifact cache.

`JsCompletionQueryInvariantError` produces no completion, edit, diagnostic, artifact, cache value, or completeness change. The editor reports it through its request-scoped internal operational channel; a concrete LSP product owns the protocol error mapping. Ordinary unsupported, invalid, ambiguous, or incomplete user input remains `Ok(None)` and does not enter that operational channel.

This completion capability requests definition, policy, and M1 model inputs only. The path/profile check and recovery-aware cursor query do not require project-wide reference inventory, reference-artifact production, or reference-side findings. When completion is the only active project-link capability, the session uses the definition-only fast path.

`intlify_producer_js` owns the [checked reverse projection](./014-ox-mf2-message-linker-design.md#jsts-typed-key-completion-projection) from one canonical model key into the selected recognizer's source-facing runtime string. This is the inverse boundary of its forward `keySyntax` conversion; neither `intlify_linker` nor the editor reimplements producer syntax.

The initial reverse projection is:

- `canonical`: return every domain-valid canonical key unchanged;
- `literal` with `json-pointer`: admit only a canonical pointer containing exactly one structural segment, JSON-Pointer-decode that segment, and return it as the complete runtime string; `/` therefore represents one empty segment, while the empty root pointer has no literal representation; and
- `dot-path` with `json-pointer`: admit only a canonical pointer containing one or more non-empty structural segments, JSON-Pointer-decode each segment, escape `\` as `\\` and `.` as `\.`, and join the results with `.`.

An unsupported syntax/domain pair, invalid canonical key, empty segment under `dot-path`, root key without a representation, or any value that does not round-trip through the producer's existing forward conversion is omitted from the candidate set. The integration never inserts the canonical spelling as a fallback under `literal` or `dot-path`.

The producer returns the decoded runtime string, not JavaScript source text. The editor integration encodes that value using the current string or no-substitution-template delimiter and ECMAScript escaping rules, then verifies that decoding the proposed edit yields the exact producer-returned value. It preserves the surrounding expression and does not choose a new quote style, convert a dynamic expression into a literal, or publish an unverified edit.

#### Candidate construction and result bound

The initial operation returns one LSP `CompletionList`. Candidate construction follows this fixed order:

1. Reverse-project every canonical model key through the selected recognizer.
2. Omit every non-representable or non-round-tripping value under the rules above.
3. Decode the exact current string content from its opening delimiter to the cursor as one runtime prefix. An incomplete escape or another value that cannot be decoded exactly returns no completion.
4. Retain only projected runtime strings that begin with that decoded prefix.
5. Group byte-equal projected runtime strings and retain the canonical key that is smallest by unsigned UTF-8 byte order as the deterministic representative.
6. Sort the remaining values first by projected runtime-string unsigned UTF-8 byte order and then by canonical-key unsigned UTF-8 byte order.
7. Return at most the first 512 values.

Filtering, projection, grouping, and ordering occur before truncation. Hash-map iteration, model insertion order, artifact order, cache state, and worker completion cannot affect the result. An implementation may use a checked prefix index or another bounded optimization only when it is result-equivalent to this pipeline.

Each item uses the projected runtime string as its `label` and `filterText`. It uses the three-digit, zero-padded, zero-based decimal ordinal `000` through `511` in canonical result order as `sortText`, so a conforming client does not replace server ordering with locale-sensitive label ordering. `insertTextFormat` is plain text; the checked `textEdit` below carries the escaped source spelling.

The inclusive initial response ceiling is 512 items. If a 513th post-filter candidate exists, the list sets `isIncomplete: true`; otherwise it sets `isIncomplete: false`. The ceiling is an editor transport and responsiveness bound, not a typed-key-model limit or a configuration option. A later request with a longer prefix recomputes the same pipeline rather than paging through an unstable prior enumeration.

#### Text edit and response freshness

Every completion item replaces the complete content range between the current literal or no-substitution-template delimiters. It does not replace either delimiter, preserve an old suffix after the cursor, or edit the surrounding call expression. The replacement text is the complete projected runtime string encoded for the existing delimiter under the checked ECMAScript escaping rules above.

At request start, the editor captures one immutable completion snapshot containing:

- the document URI, exact installed client-supplied version, source text, and a non-reused open-document generation that changes on close or reopen;
- the normalized project-relative logical path, admitted M0 source profile, cursor, selected recognizer binding, and checked content replacement range;
- the applicable root config-generation and link-input-generation tokens; and
- the resolved scope, canonical key domain, installed typed-key model, and that scope's model-generation token.

Immediately before returning the `CompletionList`, the editor performs one consistent state read and requires exact equality of the installed document version and open-document generation, root config generation, link-input generation, resolved logical path and source profile, recognizer binding, resolved scope/domain, and model-generation token. The checked replacement range must still belong to the captured source snapshot. A same-bytes version change is stale; source hashing is not a substitute for any token.

If any value differs, the document closed, or the document was reopened, the request returns `null`, emits no operational editor error, and does not retry automatically. A later client request starts from the new state. Candidate computation and edit construction use only the captured snapshot and never mix a current model or configuration with an older cursor parse.

E0 is the activation milestone; M1 fixes its minimum data dependency and reverse-projection contract but does not itself ship an editor feature. Promotion tests must additionally prove that product/client availability alone performs no project work, the first eligible open document activates only definition/model work, the last eligible document or lookup/baseline pair deactivates E0-only state, and an independently required L0/L1 session survives E0 deactivation.

014-owned `intlify_producer_js` fixtures own the parser-profile, recognizer, recovery-agreement, and checked-context API matrix. E0 integration fixtures consume its `Some`, `None`, and invariant-error outcomes and retain end-to-end coverage for absent baselines, paths excluded by `producers.js.include`, every admitted and unsupported source suffix, deferred `.vue` sources, exact callee selection among several recognizers, scope/domain mismatch, every representable syntax, omitted non-representable keys, escaped dots and backslashes, literal `*` lookup segments, source-literal escaping, exact forward/reverse round trips, unrelated recoverable syntax errors, errors intersecting the cursor structure, missing delimiters, conflicting module/script recovery contexts, zero and exactly 512 post-filter candidates, a 513th candidate setting `isIncomplete`, deterministic projection/filter/deduplication/order, whole-content replacement, and every freshness token independently becoming stale. ### Locale-aware editor features

Missing-translation indicators, locale parity lenses, and completion metadata or filtering that depends on the requested locale or fallback resolution build on M2-or-later 014 linker results, not on an editor-owned locale algorithm. Diagnostic presentation of the two M2 locale findings additionally waits for L1. Catalog-native structural checks remain a separate 008/013 input where such a feature needs them.
