# ox-mf2 リソースカタログアダプター設計

## 目的

本書では、ox-mf2 の共有リソースカタログアダプター層に関する詳細設計を記録する。この層は、JSON カタログなど複数のエントリを含むホストファイルからの MF2 メッセージエントリの抽出、メッセージテキストと生のホスト文書とのマッピング、書き戻し時の再エスケープ、およびフォーマットと lint がこの層を基盤として構築する CLI リソースワークフローを担う。

Phase 3 ツール群の境界は [005-ox-mf2-phase-3-tooling-transport-design.md](./005-ox-mf2-phase-3-tooling-transport-design.md) で定義されている。同文書では、リソースファイル、フレームワーク固有の i18n ファイル、および複数ロケールのカタログを、メッセージレベルのフォーマッターとリンターを利用する上位レイヤーとして扱う。フォーマッター側の要件は [007-ox-mf2-phase-3b-formatter-design.md](./007-ox-mf2-phase-3b-formatter-design.md#resource-and-catalog-formatting)、リンター側の要件は [008-ox-mf2-phase-3c-linter-design.md](./008-ox-mf2-phase-3c-linter-design.md#resource-and-catalog-linting)、エディター側での利用方法は [009-ox-mf2-phase-3d-lsp-editor-design.md](./009-ox-mf2-phase-3d-lsp-editor-design.md) に記録されている。本書は、これらの文書が参照先とするリソースアダプター専用の設計文書である。

メッセージエントリモデルとホスト形式アダプター契約は、当初エディターの観点から草案が作成された。本書では、それらを利用側に依存しない契約として規定する。[009-ox-mf2-phase-3d-lsp-editor-design.md](./009-ox-mf2-phase-3d-lsp-editor-design.md) には、同じ契約に基づくエディター固有の振る舞いだけを残す。リソースカタログ対応は Phase 3 の各プロダクト上に構築される上位レイヤーのマイルストーンであり、それ自体は番号付きの Phase 3 プロダクトフェーズではない。

## 用語

ox-mf2 の文書では、encode と decode という用語を、[003-ox-mf2-phase-2-binary-ast-snapshot-design.md](./003-ox-mf2-phase-2-binary-ast-snapshot-design.md) で定義された Binary AST スナップショットのエンコードとデコードのために予約している。これらの用語を曖昧にしないよう、この層の文字列変換には次の異なる名称を使用する。

- **生テキスト（raw text）**: ホストファイルに現れるとおりのバイト列。ホスト値の生スパンによって位置を表す。
- **メッセージテキスト（message text）**: i18n ランタイムがホスト形式から受け取る MF2 メッセージ文字列そのもの。メッセージレベルのコア API への入力となる。
- **エスケープ解除（unescaping）**: 生テキストからメッセージテキストへの、ホスト形式固有の変換。文字列エスケープ、引用符とスカラー形式、XML エンティティ、および CDATA セクションを扱う。
- **再エスケープ（re-escaping）**: フォーマット済みメッセージテキストから、置換用の生テキストへの書き戻し変換。

[009-ox-mf2-phase-3d-lsp-editor-design.md](./009-ox-mf2-phase-3d-lsp-editor-design.md) の LSP 位置エンコーディングはプロトコル上の用語であり、どちらの変換とも関係しない。

## 目標

- CLI のフォーマット、CLI の lint、およびエディターアダプターで共有する、利用側に依存しない単一のメッセージエントリモデルとホスト形式アダプター契約を定義する。
- ホスト形式の分類、スパンを保持する解析、抽出、オフセットマッピング、および書き戻し時の再エスケープに対して、共有の実装場所を一つに定める。
- CLI とエディターの各サーフェスが、プロジェクト設定で指定された同一のカタログ所属関係を解決できるよう、カタログのオプトイン設定を統合プロジェクト設定の一部として定義する。同時に、エディターが明示的にエディター専用のアドホックなオーバーレイを追加できるようにする。
- CLI リソースワークフローを定義する。対象には、入力選択、エントリ単位のフォーマットと lint、結果報告、書き戻しの合成、および `fmt --check` のセマンティクスを含む。
- メッセージレベルのパーサー、フォーマッター、およびリンターのコアは変更しない。リソース層とその利用側がこれらを組み合わせる。
- 観測可能な CLI 出力を決定的に保ち、Phase 3A で共有される出力規約に従う。

## 対象外

- 候補となるすべてのホスト形式を一度に実装すること。ホスト形式は一つのアダプター契約の背後に、段階的な Tier として追加する。
- ホスト形式の構文検証、ホスト形式のスキーマ検証、またはホスト形式のスタイルを管轄すること。JSON、YAML、XML の構文エラーと文書レイアウトは、ホスト形式のツール群が管轄する。
- 初期マイルストーンで、カタログキーの重複、キーの一致、翻訳の欠落など、カタログレベルおよびロケール横断のルールを提供すること。エントリモデルは将来これらに対応できなければならないが、そのルール設計は将来の作業とする。
- 実行可能な JS/TS カタログモジュールから MF2 メッセージを抽出すること。コードからの抽出には、この層が管轄しないホスト言語の意味解析が必要である。そのサーフェスはバンドラーとビルドツール群が管轄する。
- サードパーティー向けのホスト形式アダプタープラグイン API。ホスト形式レジストリは、リリースごとに組み込まれる固定の集合とする。
- エディター固有の振る舞い。文書のライフサイクル、バージョン、位置エンコーディング、公開タイミング、古い状態の処理、およびエディター編集は、引き続き [009-ox-mf2-phase-3d-lsp-editor-design.md](./009-ox-mf2-phase-3d-lsp-editor-design.md) が管轄する。
- `intlify check` を統合した振る舞いを定義すること。[006-ox-mf2-phase-3a-tooling-foundation-design.md](./006-ox-mf2-phase-3a-tooling-foundation-design.md) の規定どおり、このコマンドは v0.1 の対象外のままとし、抽出の共有、統合結果スキーマ、サマリー集約、および終了コードの合成を含め、v0.1 後の専用追補文書で扱う。
- ネストされた設定の探索、最も近い設定を優先する振る舞い、およびカタログ単位のフォーマッターまたはリンターオプションの上書き。

## アーキテクチャ

![ox-mf2 リソースカタログアダプターのアーキテクチャ](./assets/013-ox-mf2-resource-catalog-adapter-architecture.svg)

リソース層は、メッセージレベルのコアの上ではなく、横に位置する。リソース層はホスト形式の解析だけに依存してメッセージエントリを生成し、パーサー、フォーマッター、リンター自体は呼び出さない。CLI の fmt ワークフロー、CLI の lint ワークフロー、およびエディターアダプターといった利用側が、抽出処理とエントリ単位のメッセージレベルコア呼び出しを組み合わせ、それぞれのサーフェスに固有の表示を管轄する。

この管轄は並行性にも適用する。リソース API は同期的でスレッドセーフとし、CLI が上限付き worker runtime、物理ファイルグループのスケジューリング、および決定的な集約を管轄する。エディターアダプターは、別個の文書ライフサイクル用スケジューリングモデルを引き続き管轄する。

アーキテクチャ概要では、ホストアダプターの処理を意図的に `extract` と再エスケープへ集約して示している。完全なフェイルセーフの処理フローは、[抽出](#抽出)および[書き戻し時の再エスケープ](#書き戻し時の再エスケープ)で詳述する。再エスケープは、利用側が単独で呼び出せる処理ではない。検証を統合した書き戻し境界の内部でのみ実行される。

## メッセージエントリモデル

メッセージエントリは、一つのホスト文書領域を一つの MF2 メッセージに結び付ける単位である。エントリは次の要素で構成される。

- 安定した具体エントリキー
- カタログキーの比較ドメイン
- 将来のカタログレベルのグループ化に使用する論理カタログキー
- ホスト文書内の UTF-8 バイト範囲として表される、ホスト値の生スパン
- MF2 メッセージテキスト
- メッセージから生データへのオフセットマップ
- 安全に再エスケープできない値を示す読み取り専用マーカー

利用側に依存しない不変条件は次のとおりである。

- 抽出は、生スパンの開始位置が厳密に増加する順序でエントリを返す。異なるエントリの生の値スパンは、単に数学的な半開区間として重ならないだけでなく、後述する置換規則に照らして編集が互いに独立（edit-disjoint）していなければならない。
- メッセージテキストは、i18n ランタイムがホスト形式から受け取る文字列値と完全に同一でなければならない。アダプターは、メッセージテキストのトリム、正規化、または末尾への追加を行ってはならない。[Phase 3B のファイルフレーミング契約](./007-ox-mf2-phase-3b-formatter-design.md#file-framing)に従い、メッセージレベルのコア API は、注入された末尾改行を受け取ることも、メッセージ先頭の内容を失うこともない。
- エスケープ解除後の値または構造上の識別子がこれらの不変条件を満たせないホスト文字列候補は、抽出全体を `resource_entry_unsupported` で失敗させる。黙って省略することは決してない。これには、[002-ox-mf2-phase-1-rust-parser-design.md](./002-ox-mf2-phase-1-rust-parser-design.md) に記録されたパーサーレベルのソーステキスト方針によって対応されるまでの間、候補の値または構造パス内で対応のないサロゲートを表す JSON エスケープシーケンスも含まれる。

この全体失敗規則は、大規模なカタログに未対応の候補が一つしかない場合でも意図的に適用する。`readOnly` は、`message_text`、構造上の識別子、またはオフセットマップを構築できないエントリの逃げ道ではない。これは、解析は可能だが安全に再エスケープできない、それ以外の点では完全なエントリを表す。表現可能なエントリだけを返すと、オプトインされたカタログの一部を黙って省略しながら、lint 出力が完全であるように見えてしまう。部分解析を支援するには、拒否されたエントリを表す結果モデルと、不完全な結果に関するセマンティクスを別途設計する必要がある。そのため、初期アーティファクトは、部分的なエントリリストも、部分的な lint またはフォーマット結果も返さない。

単独の `.mf2` ファイルでは、このモデルの振る舞い上の形を、単一エントリだけを持つ退化したホスト文書として再利用するが、具体的なリソースアーティファクトまたは `MessageEntry` 型は再利用しない。エディターが管轄する単独ファイル用アーティファクトは、空の `StructuralPathKey`、`occurrence: 0`、`CatalogKeyDomain::StandaloneMf2`、および空の `CatalogKey` を持つ。文書 URI はキャッシュおよび診断識別子の外部名前空間のままとし、カタログレベルのグループ化には決して参加しない。そのマップは無条件に恒等写像となるわけではない。エディターアダプターは Phase 3B の[ファイルフレーミング](./007-ox-mf2-phase-3b-formatter-design.md#file-framing)の読み取り契約を適用し、削除した BOM と末尾改行を、自身のフレーミングを認識する文書マップに保持する。エディターは、共通の識別子、メッセージテキスト、生スパン、読み取り専用状態、および `map_span` の振る舞いを含む、エディター内部の一つの読み取り専用エントリビューを通じて、単独ファイルとカタログのエントリを公開する。カタログ専用の `EntryHandle` と統合されたリソース書き戻しは、そのビューの外側に留める。このメッセージ単位の統一ビューは、[009-ox-mf2-phase-3d-lsp-editor-design.md](./009-ox-mf2-phase-3d-lsp-editor-design.md) が管轄するエディターワークフロー上の関心事である。CLI のフォーマットと lint は、既存の `.mf2` ファイル向け直接パスを変更しない。リソースワークフローは、埋め込まれたメッセージに単独ファイル用のフレーミングを適用せず、単独のメッセージファイルをカタログ抽出経由に変更することもない。

### Rust の型とマッピング API

ワークスペース内部の Rust 契約は、リソース層が所有する UTF-8 バイトスパン型とエントリ識別子型を使用する。両方のスパン型が同じ半開区間の `u32` 表現を使用していても、リソースクレートがパーサークレートに依存してはならないため、`crates/intlify_resource` は `ox_mf2_parser::Span` を再利用しない。

```rust
#[derive(Debug, Default, Clone, Copy, PartialEq, Eq, Hash)]
pub struct Utf8ByteSpan {
    pub start: u32,
    pub end: u32,
}

#[derive(Debug, Clone, PartialEq, Eq, Hash, PartialOrd, Ord)]
pub struct StructuralPathKey(Arc<str>);

#[derive(Debug, Clone, Copy, PartialEq, Eq, Hash, PartialOrd, Ord)]
pub enum CatalogKeyDomain {
    StandaloneMf2,
    JsonPointer,
    YamlTypedPath,
    Xliff12,
    Xliff2,
}

#[derive(Debug, Clone, PartialEq, Eq, Hash, PartialOrd, Ord)]
pub struct CatalogKey(Arc<str>);

#[derive(Debug, Clone, PartialEq, Eq, Hash, PartialOrd, Ord)]
pub struct EntryKey {
    structural_path: StructuralPathKey,
    occurrence: u32,
}

#[derive(Debug, Clone, Copy, PartialEq, Eq, Hash)]
pub struct EntryHandle {
    // Private process-unique u64 artifact identity and u32 entry index.
}

#[derive(Debug, Clone)]
pub struct MessageEntry {
    handle: EntryHandle,
    key: EntryKey,
    catalog_key_domain: CatalogKeyDomain,
    catalog_key: CatalogKey,
    display_key: Option<Arc<str>>,
    raw_value_span: Utf8ByteSpan,
    message_text: String,
    offset_map: MessageOffsetMap,
    read_only: bool,
}

#[derive(Debug, Clone)]
pub struct MessageOffsetMap {
    // Private, validated representation.
}

#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub enum OffsetMapError {
    Reversed { start: u32, end: u32 },
    OutOfBounds { end: u32, message_len: u32 },
}
```

- `StructuralPathKey` は、アダプターがシリアライズした構造パスからの構築と、読み取り専用の文字列アクセスを提供する。利用側はその表現を変更しない。空文字列は JSON 文書ルート値を表す RFC 6901 の識別子であるため、有効なままとする。
- 一つのアーティファクト構築用インターナーが、`StructuralPathKey`、`CatalogKey`、および値が存在する `displayKey` で使用する正確な UTF-8 ペイロードを所有する。異なるフィールドまたはエントリに現れる場合でも、バイト列が等しい文字列は一つの `Arc<str>` 割り当てを共有する。newtype は引き続き意味上の違いを保持する。照合は Unicode 正規化を行わず、バイト単位で厳密に行う。エントリフィールドが共有割り当てを保持するため、アーティファクト公開後はインターナーの検索テーブルを破棄してよい。`message_text` はインターンせず、別に定める制限に従う。
- `CatalogKeyDomain` は、パーサーのディスパッチまたは外側のホスト形式とは独立して、カタログキーの比較セマンティクスを識別する。シリアライズされた `CatalogKey` 文字列が等しいことに意味があるのは、そのドメインも等しい場合だけである。JSON、JSONC、JSON5 は `JsonPointer`、YAML は個別のパスが JSON 互換であっても `YamlTypedPath`、XLIFF 1.2 は `Xliff12`、構造上互換性のある XLIFF 2.0、2.1、2.2 プロファイルは `Xliff2` を使用する。合成された Vue エントリは、`Vue` ドメインを受け取るのではなく、内側のアダプターのドメインをコピーする。`StandaloneMf2` は、退化した単独ファイル形状のための比較不能なカタログメタデータであり、カタログレベルのグループ化には決して入らない。将来のアダプターまたはプロファイルが既存のドメインを再利用できるのは、そのカタログキー同値性の全体が設計と fixture で確定された後に限る。それ以外の場合は、ドメインのバリアントを追加する。
- `CatalogKey` は、将来ロケール横断でグループ化する際に使用する論理メッセージ識別子である。読み取り専用の文字列アクセスを提供し、シリアライズ値が等しい場合でも、具体的なホスト識別子とは区別される。JSON と YAML は構造パス値をコピーし、合成された Vue エントリは内側のカタログキーをコピーし、XLIFF は末尾の source/target 側セグメントを除去して両方のロケールバリアントが一つの論理キーを共有するようにする。利用側は `CatalogKeyDomain` と組み合わせた場合だけ比較する。外側のホスト形式が等しいことだけでは、比較可能性を成立させることも妨げることもない。
- `EntryKey` は、その構造パスとゼロ起点の出現番号を組み合わせる。読み取り専用アクセサーとして `structural_path()` と `occurrence()` を公開する。等価性、ハッシュ、順序付けには両方のフィールドを使用する。連結した接尾辞表現を内部で構築したり、公開したりしない。
- `EntryHandle` は、抽出時に生成される、不透明でアーティファクトローカルな参照である。非公開表現にはプロセス内で一意な `u64` アーティファクト識別子と `u32` のエントリインデックスの両方を含むため、別のアーティファクトに由来する同じインデックスのハンドルは、誤ったエントリを参照するのではなく拒否される。`Copy` のままとし、シリアライズ、永続化、またはカタログ識別子としての使用は行わない。利用側は、対応するエントリを返した `ExtractedCatalog` と組み合わせてのみ使用する。
- アーティファクト識別子 `0` は無効値として予約する。プロセス全体で共有するスレッドセーフな単調増加アロケーターが、正常に構築された各アーティファクトに `1..=u64::MAX` の値を一つ割り当て、そのアーティファクトとエントリが破棄された後も含め、そのプロセス内では再利用しない。アロケーターはラップアラウンドしない。未使用値が残っていない場合、アーティファクト構築は `internal_error`、`details.reason: "resource_artifact_identity_exhausted"`、および進行中の構築に応じて `details.phase` が `"extract"` または `"validate_write_back"` で失敗する。アーティファクト、候補、置換結果、または部分的な対象結果は返さない。並行構築で識別子が重複しない限り、アトミック順序と公開前の割り当てタイミングは内部実装の詳細とする。
- `message_text` は所有権を持つ `String` とする。抽出アーティファクトは、自己参照するホスト文書オブジェクトから借用することなくキャッシュでき、エントリ単位の並列処理へ移動できなければならない。
- `MessageEntry` のフィールドは非公開とし、ハンドル、具体エントリキー、カタログキードメイン、カタログキー、省略可能な表示キー、生の値スパン、メッセージテキスト、オフセットマップ、および読み取り専用状態について、読み取り専用アクセサーを公開する。利用側は、アダプターが生成した不変条件を変更したり、抽出アーティファクトとは独立してエントリを構築したりできない。
- この構築制限は、具体的なカタログ型に適用する。エディターが所有する単独ファイル用の型は、より単純なフレーミング契約を独立して検証し、エディター内部の共通ビューを実装する。この型は、リソースの `MessageEntry`、`EntryHandle`、または `MessageOffsetMap` を構築も偽装もせず、それらの公開コンストラクターを必要ともしない。
- `raw_value_span` とマップ内のすべての生スパンは、ホスト文書の絶対 UTF-8 バイトオフセットを使用する。メッセージスパンは `message_text` からの相対位置とする。
- マップのクレート非公開ビルダーは、`Identity`、`Unescape`、`RawOnly` の各セグメント種別を持つ不変なセグメント列を使用する。`raw_value_span` 内のすべての生バイトが、順序どおり、隙間なく、重複せずに網羅されていること、メッセージバイト全体が順序どおり網羅されていること、セグメント種別ごとの長さ規則、および UTF-8 境界を検証する。暗黙の生データ上の隙間は認めない。メッセージバイトを消費しない構文は、`RawOnly` で表現しなければならない。アダプターが生成した無効なマップは、内部不変条件の違反である。
- 利用側はセグメントを調べるのではなく、`MessageOffsetMap::map_span(Utf8ByteSpan) -> Result<Utf8ByteSpan, OffsetMapError>` を通じてコアの範囲をマッピングする。検証では、まず `start > end` の場合に `Reversed` を返し、次に `end` がマップの完全なメッセージバイト長を超える場合に `OutOfBounds` を返す。それ以外で順序が正しく範囲内にあるバイトスパンには、UTF-8 スカラー境界であるという条件を課さない。空でない範囲には、後述するエスケープ境界と生データ専用ギャップの規則を適用する。空スパンは、次のメッセージバイトの直前のホスト位置へマッピングする。メッセージ末尾の場合は、末尾の生データ専用構文の直前へマッピングする。したがって、一つの `Unescape` 単位の出力内にある空位置は、その単位の生エスケープ開始位置へマッピングされる。返された `OffsetMapError` は、公開済みマップに対する利用側内部の誤用を示し、`details.reason: "resource_offset_map_failed"` および `details.phase: "map"` を持つ `internal_error` に変換する。既知のエントリに対してマップを呼び出した場合、利用側はそのエントリキーを含める。
- ホストまたはメッセージテキストを `u32` の UTF-8 バイトオフセットで表せない入力は、エントリを構築する前に `resource_limit_exceeded` で失敗する。

初期マイルストーンでは、安定したバインディング向けのリソースエントリ型またはオフセットマップ型を提供しない。リソース用の N-API または WASM パッケージを予定していないためである。将来、Rust 以外の具体的な利用側が必要になった場合は、内部セグメント表現を ABI 契約として公開するのではなく、これらの Rust 型から導出した、別途設計する camelCase の読み取り専用 DTO を受け取らなければならない。

## ホスト形式アダプター契約

ホスト形式アダプターは、一つのホスト形式を分類、解析してメッセージエントリへマッピングし、フォーマット済みメッセージテキストをホストの生テキストへ再エスケープする、形式固有のコンポーネントである。

### Rust レジストリおよび抽出アーティファクト API

利用側に公開する Rust 境界は、公開アダプタープラグイントレイト、またはステートレスな抽出関数と再エスケープ関数の組ではなく、不透明な抽出済み文書アーティファクトである。概念上のシグネチャは次のとおりである。

```rust
pub enum ResourcePhase {
    Extract,
    ValidateWriteBack,
}

pub fn preflight_host_bytes(
    observed_bytes: usize,
    phase: ResourcePhase,
) -> Result<(), ResourceError>

HostFormatRegistry::extract(
    &self,
    format: HostFormat,
    source: Arc<str>,
) -> Result<ExtractedCatalog, ResourceError>

ExtractedCatalog::source(&self) -> &str
ExtractedCatalog::entries(&self) -> &[MessageEntry]
ExtractedCatalog::build_and_validate_write_back(
    &self,
    formatted_entries: &[FormattedEntry<'_>],
) -> Result<WriteBackOutcome, ResourceError>

pub struct FormattedEntry<'a> {
    pub entry: EntryHandle,
    pub formatted_message: &'a str,
}

pub enum WriteBackOutcome {
    Unchanged,
    Changed(ValidatedWriteBack),
}

pub struct ValidatedWriteBack {
    replacements: Vec<RawReplacement>,
    candidate: ExtractedCatalog,
}

pub struct RawReplacement {
    entry: EntryHandle,
    span: Utf8ByteSpan,
    raw_text: String,
}
```

`preflight_host_bytes` は、固定された `host_bytes` 制限のためにリソース層が所有するエントリポイントである。デコードされていないホストバイト列を所有する利用側は、UTF-8 デコード前にこの関数を呼び出さなければならない。`observed_bytes` を型付きエラーカウンターへ正確に変換し、上限値ちょうどは受け入れる。上限を初めて超えた値は、指定された phase、`resource: "host_bytes"`、固定上限、`observed_bytes` と等しい `actual`、および site なしの `resource_limit_exceeded` を返す。このヘルパーは、入力のデコード、分類、解析、または保持を行わない。

`HostFormatRegistry` は、リリースごとに組み込まれるアダプター集合を保持する。カタログの所属関係と明示形式の解決によって、`extract` へ渡す `HostFormat` が生成され、レジストリは対応する組み込み実装へディスパッチする。サードパーティー製アダプタープラグインは対象外であるため、具体的なアダプターと、そのディスパッチ用トレイトまたは enum は、公開拡張契約にせずクレート非公開とする。

`HostFormatRegistry` と、そこから到達可能なすべての組み込みディスパッチ状態は `Send + Sync` でなければならない。複数のスレッドが、同じレジストリに対して `extract` を並行して呼び出してよい。各呼び出しは、アーティファクトビルダー、ホストパーサーの作業状態、カウンター、インターナー、およびリソース予算の計上を所有する。アダプターは、呼び出しローカルな可変状態をレジストリへ保持したり、別の抽出呼び出しと共有したりしてはならない。プロセス全体の状態は、アーティファクト識別子アロケーターなど、並行利用に安全な機能だけに限定する。また、抽出内容、エントリ順、エラー選択、またはリソース計上がスレッドスケジューリングに依存してはならない。不透明でプロセス内一意のアーティファクト識別子は割り当て順を反映してよく、決定的なレポート識別子またはキャッシュ識別子ではない。

利用側のスケジューリング境界を越え得る所有された公開リソース値は、`ResourceError`、`ExtractedCatalog`、および `WriteBackOutcome` を含め、`Send` でなければならない。不変な共有を意図するレジストリ、解決済み設定、およびアーティファクト状態は `Send + Sync` でなければならない。これらの境界は、初期フィールド配置によって偶然得られる auto trait ではなく、クレート契約の一部である。

トップレベルの `extract` 呼び出し一回につき、非公開のアーティファクトビルダーを一つ作成する。ビルダーは、アーティファクト全体の順序付きエントリ、出現回数カウンター、識別子インターナー、および累積リソース予算を所有する。合成された外側のアダプターは、同じビルダーと内側の領域の絶対ソース基準位置を渡して、非公開の内側アダプターエントリポイントを呼び出す。公開 `extract` を呼び出したり、内側のアーティファクト識別子を発行したり、領域間で累積カウンターをリセットしたりはしない。マッピングまたは書き戻しに必要な内側パーサー状態は、外側のビルダーへ非公開で付加してよい。この非公開の合成境界により、トップレベルアダプターと同じ形式固有の解析および再エスケープ実装を再利用しながら、公開されるアーティファクトとアーティファクト全体の不変条件を一つに保つ。

`ExtractedCatalog` は、正確な元のホストソースを `Arc<str>` として所有し、順序付きの `MessageEntry` 値と、不変でクレート非公開のアダプター状態を保持する。その状態には、`MessageEntry` を通じて公開することなくマッピングと書き戻しに必要となる、解析済み構造、生の綴り、スカラースタイル、外側／内側アダプターの合成、その他の形式固有情報を保持する。アーティファクトには、パーサー、フォーマッター、またはリンターのコア結果を含めない。テキストネイティブおよびプログラムからの呼び出しに対する多層防御として、`HostFormatRegistry::extract` はホスト解析前に `preflight_host_bytes(source.len(), ResourcePhase::Extract)` を呼び出す。この内部での再検査は、利用側が生バイト列から開始する場合に必須となるデコード前の呼び出しを代替しない。

アーティファクト構築では、`extract` が返る前にすべてのエントリスパンを検証する。すべての `raw_value_span` は UTF-8 境界上にあり、元のソース内に収まり、開始オフセットが厳密に増加する順序でなければならない。空でない二つのスパンは一つの境界で接してもよいが、重複してはならない。位置 `p` にある長さゼロのスパンは、他のすべての長さゼロのスパンと異なる位置を持ち、`p` は別エントリの空でないスパンの内側にも、どちらかの境界にもあってはならない。これらの編集非交差規則により、空でないスパンの置換と長さゼロのスパンへの挿入は、置換順序またはプロトコル編集の適用順序に依存せず一つの結果を持つ。不正な場合は、アーティファクトまたは部分的なエントリリストが利用側へ届く前に抽出全体を中止し、`details.reason: "resource_adapter_invariant_failed"` および `details.phase: "extract"` を持つ `internal_error` へマッピングする。ホスト構文エラーは引き続き `resource_parse_failed` とする。この不変条件エラーは、無効なユーザー入力ではなく、組み込みアダプターの欠陥を示す。

`FormattedEntry` は、選択された書き込み可能な一エントリについて、テキストが変更されたかどうかにかかわらず、フォーマッターが正常に生成した完全な出力である。フォーマッター結果にパーサー診断が含まれるエントリ、エディターの範囲要求で選択されていないエントリ、およびすべての読み取り専用エントリは、パーサー診断をレポート用に保持したうえで利用側が省略する。選択されたすべてのフォーマッター呼び出しが運用エラーなく完了した後、利用側は残ったすべての出力を一つの `build_and_validate_write_back` 呼び出しへまとめて渡す。空の入力も有効である。

`build_and_validate_write_back` は、利用側が呼び出せる唯一の再エスケープおよび候補構築境界である。最初に、指定されたすべてのハンドルをアーティファクトに対して検証し、別アーティファクト由来、重複、存在しない、または読み取り専用の入力を、`details.reason: "resource_invalid_entry_handle"` および `details.phase: "validate_write_back"` を持つ `internal_error` として拒否する。有効な入力は、呼び出し側スライスの順序で処理するのではなく、アーティファクト内のエントリでインデックス化する。アダプター固有の再エスケープと置換適用は非公開の実装手順のままとするため、CLI とエディターの利用側は、低レベルメソッドを呼び出して候補の制限検査または完全な検証を回避できない。

有効な入力に対し、このメソッドはまず、すべてのアーティファクトエントリを生スパン順に走査する計測パスを実行する。各エントリでは、指定された場合はフォーマット済みメッセージを、そうでなければ元のメッセージを選択し、`message_bytes` を検査し、実行中の `total_message_bytes` を更新して検査した後、指定され、かつバイト列が異なるフォーマット済みメッセージに対して非公開で再エスケープを計画する。計画は、アダプター固有のすべての再エスケープ条件を検証し、置換テキストを割り当てることなく、完全な生置換の正確な UTF-8 バイト長を記録する。これにより、生順で最初に失敗するエントリが優先される。同じエントリでは、メッセージ単位の制限、実行中の合計制限、再エスケープ計画の検証の順に優先する。

計測パス全体の完了後、このメソッドは元のホスト長、置換されるすべての生スパン長、および計測済みのすべての置換長から、最終的な `host_bytes` の予測値を一つ計算する。後の置換によって候補全体が制限内に収まる場合、一時的に拡張する中間プレフィックスは拒否しない。その最終予測値が包含的なホスト上限内に収まった場合に限り、二回目の生順パスで計画済みの各 `raw_text` を実体化し、生スパンの降順で置換を適用し、同じ組み込みアダプター合成を用いて、候補ホスト文書全体を再度解析および抽出する。フォーマッターの運用エラーは、このメソッド全体より優先する。選択されたフォーマッター呼び出しのいずれかが運用上失敗した場合、利用側は後述の並列性規則に従って、生順で最も早いフォーマッターエラーを選択し、書き戻し構築を開始しない。

フォーマット済み入力のどれも元のメッセージと異ならない場合、結果は `WriteBackOutcome::Unchanged` となる。再エスケープ済みテキスト、`RawReplacement`、候補ソース、または候補アーティファクトは構築しない。それ以外の場合、`Changed` は元の生エントリ順に並んだ置換と、完全に検証済みの候補アーティファクトを含む。`ValidatedWriteBack` は `replacements()` と `candidate()` への読み取り専用アクセス、および候補の所有権を消費するアクセスを公開する。`RawReplacement` は非公開で、元のアーティファクトハンドル、元のスパン、および実体化された生テキストだけを保持する。期待されるフォーマット済みメッセージを複製または保持しない。読み取り専用の `span()` はエントリの元の `raw_value_span` と正確に一致し、`raw_text()` はそのスパンに対する完全な再エスケープ済みホスト値である。利用側は、統合メソッドとは独立して置換を構築または変更できない。CLI の利用側は候補ソースを使用する。エディターの利用側は、プロトコル編集を生成するまで置換を保持し、メソッドが成功した後に限って候補アーティファクトを破棄する。

多層防御として、非公開の置換適用処理は、統合メソッドの公開 API を経由すれば違反が到達不能である場合でも、各ハンドル、その変更されていない元スパン、エントリごとの一意性、および編集非交差スパン規則を再検証する。

候補の検証では、再抽出された `EntryKey`、`CatalogKeyDomain`、`CatalogKey`、省略可能な `displayKey`、および `readOnly` の各列が、元の列と正確に等しいことを要求する。`FormattedEntry` の借用が `build_and_validate_write_back` 内で有効な間に、このメソッドは各候補エントリのメッセージテキストを、実効的に期待される値と比較する。期待値は、バイト列が同一の場合を含め、指定されていれば呼び出し側のフォーマット済みメッセージ、そうでなければ元のアーティファクトメッセージである。借用した検索表は返却前に破棄し、`RawReplacement` へコピーすることはない。将来利用側に公開するエントリメタデータ、たとえばアーティファクトの一部となった時点の内在ロケールは、既定で等価性を検査する。フィールドを導入する設計は、候補時の変更を偶発的に継承するのではなく、変更を許す場合は明示的に根拠を示さなければならない。

候補の抽出では、新しく割り当てたアーティファクト識別子のもとで、新しい `EntryHandle` 値が必然的に発行される。検証済みの置換で必要となる場合、生の値スパン、オフセットマップ、生の綴り、候補ソース全体、およびアダプター非公開の解析状態も変化し得る。これらのアーティファクトローカルおよびソース座標上の変更によって、前述した利用側に公開する識別子および能力メタデータの等価性要件が緩和されることはない。候補がリソース上限を超えた場合は、引き続き `details.phase: "validate_write_back"` を持つ `resource_limit_exceeded` とする。これはアダプター不変条件の失敗ではなく、想定された入力サイズ処理である。アーティファクト識別子の枯渇は、前述の専用内部理由を維持する。それ以外の解析、抽出、識別子、メタデータ、または値の不一致は、処理全体を `details.reason: "resource_write_back_failed"` を持つ `internal_error` として失敗させる。`WriteBackOutcome`、候補アーティファクト、または部分的な置換集合は返さない。変更ありの正常な結果には、正確な候補ソース、エントリ、およびオフセットマップを含む、検証済み候補の完全な再抽出済み `ExtractedCatalog` を含める。

エディターが一つのエントリだけを変更する場合でも、CLI とエディターの利用側には候補全体の検証を必須とする。サイズしきい値、エディター設定、または高速パスによって未検証の編集を返してはならない。最適化されたパスが前述のすべての不変条件を確立し、完全な再解析および再抽出と fixture 上で同等である場合に限り、実装は不変の解析状態を再利用したり、増分検証を導入したりしてよい。これは内部最適化であり、弱い API モードではない。インタラクティブなコストは、集約書き戻し時間の中へ隠すのではなく、後述するエディター単一エントリベンチマークで個別に計測する。

アーティファクトと、そこから到達可能なすべてのアダプター状態は `Send + Sync` でなければならない。抽出は、アーティファクト公開前に、変更可能なホスト解析をすべて完了する。並行する読み取りアクセス、オフセットマッピング、および独立した `build_and_validate_write_back` 呼び出しは、アーティファクトを変更しない。ソースの所有権と所有されたメッセージ文字列により、自己参照する借用なしでアーティファクトをキャッシュできる。`EntryHandle` が意味を持つのは、その発行元アーティファクト内だけである。一方、安定したキャッシュおよびレポート用識別子には引き続き `EntryKey` を使用する。

正確な `ResourceError` バリアントと CLI 運用エラーへのマッピングは、後述するリソースエラーモデルに従う。この API は、リソースクレートから CLI の JSON エラーオブジェクトを公開しない。

### 抽出

![ox-mf2 リソース抽出パイプライン](./assets/013-ox-mf2-resource-extraction-pipeline.svg)

`HostFormatRegistry::extract` は、一つの全体失敗型パイプラインである。

1. **事前検査とディスパッチ。** バイト列を所有する利用側が UTF-8 デコード前に `host_bytes` を検査し、`extract` は指定された `Arc<str>` に対して検査を繰り返す。すでに解決済みでリリース済みの `HostFormat` が一つの組み込みアダプターを選択する。レジストリのディスパッチでは、カタログの所属関係または設定解決を繰り返さない。
2. **スパンを保持するホスト解析。** 選択されたアダプターは、生のトークンと値のスパンを保持し、パーサーローカルなネスト上限を適用しながら、ホスト構文全体を解析する。通常の `serde` スタイルによる値だけのデシリアライズでは不十分である。構文または深さの失敗は、エントリの走査前に返す。
3. **生順の走査。** 完全な解析の後、アダプターは、対応済みおよび未対応の文書構造と、候補となる文字列リーフをソース順に走査する。ここが、文書機能、エントリの表現可能性、およびリソース制限の失敗について、決定的に選択する境界となる。
4. **エントリ構築。** 各候補に対し、共有アーティファクトビルダーは構造識別子とカタログ識別子を導出し、生順に出現番号を割り当て、ランタイムが受け取る正確なメッセージテキストへエスケープ解除し、生の値スパンとオフセットマップを構築し、読み取り専用状態を記録し、累積予算とインターナーを更新する。合成された外側のアダプターは、内側のアーティファクトを公開するのではなく、同じビルダーへ絶対ソース基準位置を渡す。
5. **アーティファクト全体の検証。** 公開前に、ビルダーは、順序付きで編集が互いに独立したエントリスパン、完全なオフセットマップ、識別子と出現番号の不変条件、および合成されたすべての領域にわたるアーティファクト全体の全上限を検証する。
6. **確定。** 完全かつ有効なビルダーだけが、プロセス内で一意なアーティファクト識別子を一つ受け取り、正確なソース、順序付きエントリ、およびアダプター非公開の書き戻し状態を含む、不変の `Send + Sync` な `ExtractedCatalog` になる。

どの段階で失敗しても、型付きの `ResourceError` を一つ返す。部分的なエントリリスト、途中まで消費されたリソース予算、または不完全なアーティファクトが利用側から観測可能になることはない。

### エントリ識別子

- `StructuralPathKey` は、その形式契約のもとでトークン境界とキーペイロードが曖昧にならないようシリアライズされたホストパスである。JSON アダプターは RFC 6901 JSON Pointer を使用するため、リテラルのカタログキー `"a.b"` と、`a` → `b` というネストされたパスは、ランタイムが両方を `a.b` に平坦化する場合でも区別される。RFC 6901 のトークンは、包含する JSON ノードの種別を意図的にエンコードしない。同じトークンパスにある、`"0"` という名前のオブジェクトメンバーと配列インデックス `0` は、一つの構造パス文字列を共有する。一方、YAML アダプターは、マッピングキーとシーケンスインデックスを区別し、各スカラーキーについて解決済みのタグと値を保持する、型付き構造パスを使用する。初期段階で未対応の複雑なマッピングキー配下に候補文字列がある場合、抽出は `resource_entry_unsupported` および `details.reason: "structural_path_unsupported"` で失敗する。YAML アダプターが JSON Pointer によるシリアライズを使用できるのは、JSON 互換で文字列キーだけを含むパスに限る。したがって、YAML の文字列キー `"1"`、整数キー `1`、シーケンスインデックス `1` が衝突することはない。
- `EntryKey` は一意な出現識別子であり、表示用文字列ではない。同じ構造パスを持つ抽出済みエントリの集合ごとに、アダプターは生のソース順で `0` から `occurrence` を割り当てる。パスが一度しか現れないエントリも `occurrence: 0` を持つ。このフィールドを省略したり、後から発見した重複から推定したりすることはない。
- ホスト文書に重複キーまたは重複した祖先パスが含まれる場合、生データ上の各リーフ出現箇所を別々のエントリとする。番号付けは完全な構造パスが同じエントリ間で行うため、識別子へバイトオフセットを含めなくても、ネストされた重複を曖昧なく識別できる。カタログキーの重複を問題として報告する機能は、将来のカタログレベル lint であり、抽出失敗ではない。
- `CatalogKey` は、具体的なホスト上の役割とは独立して、ランタイム形式の論理メッセージを表す。JSON と YAML は `StructuralPathKey` と同じシリアライズ値を設定する。ホスト識別子にロケール役割の構文を含む形式は、明示的な形式契約のもとで別の値を導出してよい。将来のロケール横断グループ化では、`CatalogKeyDomain` と `CatalogKey` を比較スコープおよびロケールと組み合わせて使用する。そのため、一方のカタログがオブジェクト経由で、別のカタログが配列経由で末尾の数値トークンへ到達する場合でも、同じ JSON Pointer トークンは一つの論理 JSON キーとして比較される。一方、ホストの重複キールールは、出現番号を除いた `StructuralPathKey` でグループ化する。そのため、異なる XLIFF の source と target の役割は、カタログキーを共有するだけでは重複にならない。出現箇所と具体的な証拠には、常に `EntryKey` を使用する。
- アダプターは UI とレポート向けに、人間が読みやすい表示キーを追加で公開してもよいが、既定では省略する。アダプターが表示キーを生成する前に、形式固有の契約で、決定的な導出方法と衝突時の振る舞いを定義しなければならない。実装だけの規約では不十分である。Tier 1 の JSON アダプターは、その JSON Pointer 構造パスがすでに表示に適しているため、`displayKey` を生成しない。JSON エントリに人間が読めるラベルが必要な場合、利用側はそのパスを使用する。レポート、エディター診断の識別子、およびメッセージキャッシュの識別子には完全な `EntryKey` を使用する。表示用識別子が、どちらの安定したキー型を置き換えることもない。

YAML は、ルートを空文字列とするポインターとして、その型付き構造パスをシリアライズする。各パスステップでは、`/` の後に型付きセグメントを一つ追加する。RFC 6901 のエスケープと同様に、完全なセグメント内の `~` は `~0`、`/` は `~1` としてエンコードする。パーセントエンコーディング、Unicode 正規化、または表示キーへの平坦化は行わない。初期セグメント形式は次のとおりである。

| YAML パスステップ        | エスケープ前の型付きセグメント      | シリアライズされたパスの例 |
| ------------------------ | ----------------------------------- | -------------------------- |
| 文字列マッピングキー     | `k:str:<resolved string>`           | `/k:str:greeting`          |
| null マッピングキー      | `k:null`                            | `/k:null`                  |
| boolean マッピングキー   | `k:bool:true` または `k:bool:false` | `/k:bool:true`             |
| 整数マッピングキー       | `k:int:<canonical integer>`         | `/k:int:1`                 |
| 浮動小数点マッピングキー | `k:float:<canonical float>`         | `/k:float:15e-1`           |
| シーケンスインデックス   | `i:<zero-based decimal index>`      | `/i:1`                     |

文字列ペイロードは、解決済みの正確な Unicode スカラー列とする。たとえば、キー `a/b~c` は `/k:str:a~1b~0c` になる。整数の正規形は任意精度の十進表記とし、先頭のプラス記号と不要な先行ゼロを持たず、ゼロのあらゆる綴りを `0` とする。有限の浮動小数点値は、正確な十進係数と指数として正規化する。係数には先行ゼロも末尾ゼロもなく、指数には先頭のプラス記号も不要なゼロもなく、負のゼロを含むゼロは `0e0` とする。たとえば、`1.0` は `1e0`、`1.50` は `15e-1` になり、`1e3` は `1e3` のままとなる。正の無限大、負の無限大、および NaN は、それぞれ `.inf`、`-.inf`、`.nan` とする。この正規化処理はリソース層が所有し、依存ライブラリの debug または display フォーマットを使用してはならない。

したがって、解決済みの Core タグと値が同じで、ソース上の綴りだけが異なるキーは一つの構造パスを共有し、それぞれ異なる出現番号を受け取る。一方、文字列キー `"1"`、整数キー `1`、シーケンスインデックス `1` は、`/k:str:1`、`/k:int:1`、`/i:1` のままである。明示された対応済み Core タグも、同じ解決済み型のシリアライズを使用する。複雑なマッピングキーには初期セグメント形式が存在しない。不透明または情報を失うパスを割り当てるのではなく、前述の `resource_entry_unsupported` を発生させる。

### メッセージから生データへのオフセットマッピング

各エントリは、メッセージテキスト内のメッセージローカルな UTF-8 バイト範囲と、生の値スパン内にあるホストの生 UTF-8 バイト範囲を対応付ける、単調なオフセットマップを持つ。

- `raw_value_span` 内のすべての生バイトは、正確に一つのマップセグメントに属する。マップに表現されない生データ上の隙間はない。空の生の値スパンは、空のセグメント列で網羅する。
- 正規形では、メッセージスパンと生スパンの両方が連続する隣接 `Identity` セグメントを、可能な限り結合する。同様に、同じ長さゼロのメッセージ位置にあり、生スパンが連続する隣接 `RawOnly` セグメントも、可能な限り結合する。`Unescape` セグメントは、別のセグメントと決して結合しない。隣接する単位を一つの大きな単調範囲で表現できる場合でも、エスケープ解除処理上の各不可分単位は正確に一つのセグメントとして残す。ビルダーは、検証、公開、およびリソース計上より前に、増分的に正規化する。
- メッセージテキストのバイトと生バイトが等しい連続区間は、一つの恒等写像セグメントを共有する。
- 単一のホストエスケープ、JSON の UTF-16 サロゲートペアのように一つのスカラー値へ共同でエスケープ解除される複数のエスケープ、または XML エンティティといった、エスケープ解除処理上の各不可分単位を一つのセグメントとし、メッセージテキストのバイトを生のエスケープ範囲全体へマッピングする。
- 文字列の引用符や CDATA の区切り文字など、生データにしか存在しない構文は、置換に使用する生の値スパン内には残るものの、メッセージテキストのバイトを消費しない。診断範囲が区切り文字を含まないよう、オフセットマップはこれらの境界ギャップを保持する。単独ファイル用エディターアダプターでエントリの形を再利用する場合、削除したファイルフレーミングも同種の生データ専用の境界ギャップとして表現する。
- 一つのエスケープの出力内に位置するメッセージローカルな位置は、範囲の開始位置であれば生のエスケープの先頭に、範囲の終了位置であれば生のエスケープの末尾にマッピングする。これにより、マッピング後の範囲がホストのエスケープシーケンスを分割することはない。
- バイト位置は Unicode スカラー境界である必要はない。`Identity` セグメント内ではバイト相対オフセットを維持し、`Unescape` セグメント内では前述のエスケープ境界規則を使用する。`map_span` は、呼び出し側から渡された無効な範囲をクランプ、入れ替え、またはその他の方法で修復することは決してない。

### 書き戻し時の再エスケープ

![ox-mf2 リソースの検証済み書き戻しパイプライン](./assets/013-ox-mf2-resource-write-back-pipeline.svg)

再エスケープは、`ExtractedCatalog::build_and_validate_write_back` 内部で行う非公開のアダプター処理である。利用側は、未検証の生置換を要求できない。統合パイプラインは次のとおりである。

1. **入力を検証する。** 指定されたすべての `EntryHandle` は、そのアーティファクトに属し、一度だけ存在し、書き込み可能なエントリを識別しなければならない。このメソッドは有効な入力をアーティファクト内のエントリでインデックス化するため、呼び出し側スライスの順序は振る舞いに影響しない。
2. **生順に計測する。** 各アーティファクトエントリについて、指定されたフォーマット済みメッセージまたは元のメッセージを選択し、メッセージ単位および実行中の合計上限を検査し、バイト列が異なる書き込み可能な入力だけに対して、決定的な再エスケープ計画を作成する。計画では表現可能性を検証し、出力サイズに比例するテキストを割り当てずに、正確な生出力長を記録する。指定されたメッセージが一つも異ならない場合、このメソッドは置換または候補アーティファクトを構築せずに `WriteBackOutcome::Unchanged` を返す。
3. **候補全体のサイズを予測する。** 変更された入力について、正確な一つの `u128` 計算で、元のホストバイト数、除去する生スパン、および計測済みのすべての置換長を組み合わせる。出力の割り当て開始前に `host_bytes` を満たさなければならないのは、拡張中の中間プレフィックスではなく、候補全体の予測値である。
4. **計画を実体化する。** 受け入れた各計画が、正確な生置換テキストを生成する。その結果は、計測済みの長さおよび選択済みのシリアライザーブランチと一致しなければならない。
5. **置換を適用する。** このメソッドは、ハンドルと編集非交差スパンを再検証し、生スパンのオフセット降順で置換を適用し、候補ホストソース全体を一つ構築する。
6. **再解析および再抽出する。** 候補は、すべての構文、表現可能性、マッピング、識別子予算、およびリソース上限の検査を含め、同じアダプター合成および抽出契約を通過し、新しいアーティファクト識別子を受け取る。
7. **候補の等価性を検証する。** 候補のエントリ識別子、カタログキードメインとキー、表示用識別子、読み取り専用状態、およびすべての実効メッセージ値は、元のアーティファクトへ指定されたフォーマット済みメッセージを反映したものと等しくなければならない。その後に限り、このメソッドは、生順の置換と検証済みの候補アーティファクトを含む `WriteBackOutcome::Changed` を返す。

どの失敗でも、`WriteBackOutcome`、候補アーティファクト、部分的な置換集合、または利用側から見える編集は返さない。この検証は、CLI の書き込み、check、stdin、およびエディターフォーマットで同一である。これらの利用側は、正常な結果の利用方法だけが異なる。

- 再エスケープは値同一性を満たさなければならない。再エスケープされたテキストをエスケープ解除した結果は、フォーマット済みメッセージテキストと完全に同一でなければならない。
- そのスタイルでフォーマット済みテキストを表現できる場合、再エスケープでは、引用符やスカラー形式など、ホスト値のスタイルを保持すべきである。表現できない場合、アダプターは、メッセージ値の外側にあるホスト文書の意味を同一に保ちながら、表現可能なスタイルへ切り替えてよい。
- あるホスト構造に対して、アダプターが値同一の再エスケープを保証できない場合、抽出時にそのエントリを読み取り専用としてマークする。フォーマッターは、パーサー診断を見える状態に保つため、読み取り専用のメッセージテキストも引き続き評価する。ただし、正常なフォーマット済み出力は破棄し、置換または check 上の差分へ変換することはない。
- 書き込み可能なすべてのアダプターは、非公開で決定的な再エスケープ計画処理を一つ提供する。計画処理はフォーマット済みメッセージを借用し、正確なシリアライザーブランチとホストスタイルを選択し、実体化時に行うすべての値表現可能性検査を実行し、完全な生置換の正確なバイト長を `u64` として計算する。計画には、上限のあるメタデータと借用した入力だけを含める。計測した出力に比例するバッファを割り当ててはならない。
- 実体化は、文書全体の候補サイズ検証後に限って実行し、事前に検証済みの計画から出力する。計測済みの長さを予約し、正確にその数の UTF-8 バイトを生成しなければならず、異なるスタイルまたはシリアライザーブランチを選択できない。実体化の失敗または計測長の不一致は、`details.reason: "resource_adapter_invariant_failed"` および `details.phase: "validate_write_back"` を持つ `internal_error` とする。候補または部分的な置換結果は返さない。

### 抽出失敗の表示

ホスト文書自体を解析できない場合、文書全体の抽出が失敗する。ホストの構文エラーを MF2 診断へ変換することはない。ホスト形式のツール群がそれらを管轄する。CLI は `resource_parse_failed` を対象ローカルな運用エラーとして報告する。一方、エディターアダプターはこのコードを一時的なホスト編集中状態として扱い、直近で成功した MF2 診断の公開状態を保持する。

それ以外の対処可能なリソース入力失敗は、一時的な解析状態ではない。CLI は後述する安定した運用コードを報告する。エディターは古い MF2 診断を消去し、形式、エントリ、文書、または上限に関する入力失敗について、別の `ox-mf2-resource` 診断を一つ公開する。設定および内部の失敗は、代わりにエディターの運用エラーチャネルを使用し、直近で成功した状態をトランザクションとして保持する。正確なエディター公開および回復の振る舞いは [009-ox-mf2-phase-3d-lsp-editor-design.md](./009-ox-mf2-phase-3d-lsp-editor-design.md) が管轄する。どの抽出失敗でも、フォーマット編集または部分的なアーティファクトは返さない。

## ホスト形式レジストリとカタログ検出

利用側は、ホスト形式レジストリを通じて、各入力ファイルまたは文書を最大一つのホスト形式アダプターへ解決する。

- 単独 MF2 の分類は、カタログの所属判定より先に実行する。`.mf2` ファイル、または大文字小文字を区別して正確に `messageformat2` という language id を持つエディター文書は、そのパスが広範なプロジェクトまたはアドホックなカタログパターンにも一致する場合でも、常に単独ファイル用ワークフローを使用する。そのカタログ一致は、形式の競合ではなく、予約された単独ファイル対象には適用不能なものとする。ファイルをリソース抽出経由へ変更することは決してない。
- カタログのホスト形式は、厳密なオプトイン方式とする。任意の JSON、YAML、XML ファイルに MF2 メッセージが含まれると想定してはならない。誤検出は無関係なファイルに誤った診断を報告し、さらに悪いことには、フォーマットによる書き戻しの対象にしてしまう。設定を必須にする方が安全である。
- プロジェクト設定によるカタログのオプトインは、後述する統合プロジェクト設定の `resources` セクションから取得する。エディターアダプターは、その結果に、[009-ox-mf2-phase-3d-lsp-editor-design.md](./009-ox-mf2-phase-3d-lsp-editor-design.md#configuration-sources) で定義された、明示的にエディター専用で正規化済みのアドホックなカタログ設定を、後述の固定された優先順位で追加してよい。ファイル名規約がカタログ所属を暗黙に意味することはない。`*.mf2.json` のような名前を推奨 `include` パターンとして文書化してよいが、どの Tier でも自動オプトインの既定値にはしない。
- CLI リソースワークフローとエディターアダプターは、まずプロジェクト設定による所属関係とホスト形式を同一に解決する。プロジェクトとの一致は権威を持ち、エディター設定によって再分類できない。重複するアドホックな一致が同じ形式へ解決される場合、プロジェクト所有の対象として重複を除去する。異なる形式へ解決される場合はエディター設定エラーとし、プロジェクト結果を上書きまたは隠蔽することはない。
- アドホック層がオプトインできるのは、解決済みプロジェクト設定に一致しないままの文書だけである。これには、プロジェクトカタログの `exclude` で除去されたパスも含む。重複するアドホックなエントリにも同じ規則を使用する。同じ形式への一致は重複を除去し、異なる形式への一致はエディター設定エラーとする。無効なプロジェクト設定は、オーバーレイで迂回するのではなく、オーバーレイ適用前に報告する。
- エディター専用のアドホックな対象は、プロジェクト設定へ永続化されるまで、意図的に CLI と CI の処理対象外となる。エディター統合では、その状態を CI でカバーされているかのように見せず、区別する。
- 重複するカタログ設定によって、具体的な一ファイルが複数のホスト形式にマッピングされる場合、それは設定検証の失敗であり、暗黙の優先順位ではなく運用エラーとする。競合は、glob の交差を静的に解析しようとするのではなく、具体的に選択された文書またはエディターで開かれた文書のパスから評価する。

### 形式 ID と拡張子分類

レジストリ ID は、正規形の小文字 ASCII 文字列とする。拡張子から導出する分類では、正規化済み論理パスをスラッシュで区切った最後の basename に対し、プラットフォームに依存しない一つの字句規則を使用する。この互換性サーフェスを OS のパス API へ委譲しない。basename 内の最後の `.` が拡張子を開始するのは、それが先頭でも末尾でもない Unicode スカラー値である場合だけとする。条件を満たす拡張子は、その `.` から basename の末尾までの正確な部分文字列とする。それ以外の場合、ファイルには拡張子がなく、保持される元の拡張子表記は `""` となる。したがって、`messages.json`、`.config.json`、`messages..json` は `.json` を保持するが、`.json` と `messages.` には拡張子がない。レジストリ検索では、ファイルシステムの大文字小文字区別とは無関係に、すべての OS で、条件を満たす拡張子を ASCII の大文字小文字を畳み込んで比較する。一方、運用エラーの details には元の綴りを保持する。この設計で確定する初期および延期済みマッピングは次のとおりである。

| レジストリ ID | 導出されるファイル拡張子 | Tier |
| ------------- | ------------------------ | ---- |
| `json`        | `.json`                  | 1    |
| `vue`         | `.vue`                   | 2    |
| `yaml`        | `.yaml`, `.yml`          | 3    |
| `jsonc`       | `.jsonc`                 | 3    |
| `json5`       | `.json5`                 | 3    |
| `xliff`       | `.xlf`, `.xliff`         | 3    |

分類では、固定された既知 ID の語彙と、現在のリリースで提供されるアダプターを分離して保持する。

```rust
pub enum KnownHostFormatId {
    Json,
    Vue,
    Yaml,
    Jsonc,
    Json5,
    Xliff,
}

pub enum HostFormat {
    Json,
}

pub enum HostFormatClassification {
    Shipped(HostFormat),
    KnownButUnshipped(KnownHostFormatId),
    UnrecognizedExtension,
}
```

`KnownHostFormatId` は、リリース Tier にかかわらず、この設計で確定したすべての正規 ID を含む。`HostFormat` は、現在のリリースで提供されるアダプターだけを含み、各 Tier の実装に伴って拡張する。拡張子分類は `HostFormatClassification` を返す。その入力コンテキストは、エラー details 用に元の拡張子表記を保持する。`KnownButUnshipped` と `UnrecognizedExtension` は抽出前に `resource_format_unsupported` となる。一方、`HostFormatRegistry::extract` へ到達できるのは `Shipped(HostFormat)` だけである。設定内の明示的な `format` は、リリース済み形式だけを含む `HostFormat` へ直接デシリアライズされる。そのため、延期された実行時分類を生成するのではなく、スキーマ検証で失敗する。

重複するカタログ定義については、明示形式、または拡張子から導出した `Shipped`／`KnownButUnshipped` の各結果が、形式割り当てを一つ提供する。`UnrecognizedExtension` への一致はカタログの所属関係を成立させるが、割り当ては提供しない。提供されたすべての割り当てが一つの正規既知 ID を指す場合、その ID を採用する。認識不能な一致が追加で存在する場合でも、同一 ID の定義は重複を除去する。どの定義も割り当てを提供しない場合、オプトインされた対象は `format` を省略した `resource_format_unsupported` となる。割り当てが二つ以上の異なる既知 ID を指す場合、対象は `catalog_format_conflict` となる。最初／後続の競合ポインターを選択する際に参加するのは、実際にそれらの割り当てを提供した定義だけである。したがって、拡張子から導出する広範な定義は、拡張子のない、または任意の名前のファイルに対する特定の `format: "json"` 上書きと重複しても、その明示的分類を損なわない。

たとえば、`.YML` は Linux、macOS、Windows のすべてで正規形の `yaml` を導出するが、エラー details には引き続き `.YML` を保持する。明示的なカタログ `format` は、正確な正規 ID としてスキーマ検証する。`yml` や `xlf` などの別名、`JSON` などの大文字値、前後の空白、およびコンマで連結した値は、正規化するのではなく無効とする。`HostFormat`、生成される設定スキーマ、および `supportedFormats` は、そのリリースで提供されるアダプターだけを公開する。一方、拡張子分類は延期済み ID を認識でき、そのアダプターがリリースされるまでは `resource_format_unsupported` を報告してよい。

これらのファイル名規則は、カタログの stdin 仮想パスの分類にも使用する。埋め込まれた Vue の `lang` 宣言には適用しない。その正確な ID と明示的な `yml` 別名は、後述の専用箇所で定義する。また、内容のスニッフィングは行わない。

オプトインされたカタログ文書内では、既定の抽出範囲を、配列内の文字列要素を含む、ホスト上で文字列型となるすべてのリーフ値から開始する。その後、アダプターは候補のパスとエスケープ解除後の値を、前述のメッセージエントリ表現可能性の不変条件に照らして検証する。いずれかの候補が失敗した場合、ホスト文書全体の抽出を `resource_entry_unsupported` で失敗させる。部分的なアーティファクトまたはエントリリストは返さず、lint、フォーマット、および書き戻しはそのファイルを処理しない。複数の候補が失敗した場合、生ソースオフセットが最も小さいエラーを決定的に選択する。

文字列以外のスカラーはメッセージエントリ候補集合の外にあり、エラーを生成しない。将来の明示的なキーセレクターは、一致した文字列を設定済み抽出範囲から意図的に除外し、同様にエラーを生成しない。これはアダプターの対応範囲不足ではなく、ユーザーが選択した範囲を表す。

エントリ単位のフォーマットと lint では、一つの MF2 メッセージの解析、フォーマット、lint がロケールに依存しないため、ロケール識別子は必須ではない。[将来のカタログチェック向けロケール関連付け](#将来のカタログチェック向けロケール関連付け)で定義する将来の locale binding は、メッセージ抽出またはメッセージレベルのコア API を変更せずに、カタログレベルおよびロケール横断の lint ルール向けロケールメタデータをエントリへ関連付ける。

## ホスト形式の Tier

ホスト形式は、同じアダプター契約の背後に Tier として導入する。Tier は実装工数とエコシステムからの需要に基づく順序を示すものであり、契約を変更するものではない。

| Tier | ホスト形式 | ステータス |
| --- | --- | --- |
| 1 | JSON カタログ（`.json`） | 初期リソースマイルストーン |
| 2 | アダプター合成による Vue SFC の `<i18n>` カスタムブロック | 後続対応として延期 |
| 3 | YAML カタログ（`.yaml`、`.yml`）、JSONC および JSON5 カタログ、XLIFF 1.2 / 2.x | 後続対応として延期、需要に応じて対応 |

初期リソースマイルストーンで実装するのは Tier 1 だけである。Tier 2 と Tier 3 の形式は延期し、[後続対応として延期した事項](#後続対応として延期した事項)で追跡する。以下の各サブセクションでは、共有契約が各 Tier に耐えられるよう、契約レベルの設計メモを記録する。これらは初期実装へのコミットメントではない。

ARB、gettext PO、Java properties、およびその他の交換形式は、Tier 3 の構成要素ではなく、Tier 未割り当てで需要に応じて対応する候補とする。候補が Tier に入るのは、その正規レジストリ ID、拡張子マッピング、抽出範囲、構造識別子とカタログ識別子、オフセットマッピング、未対応構造、および値同一の再エスケープ契約を一体として設計し、fixture とともに `KnownHostFormatId` へ追加した後に限る。

### JSON カタログ

JSON カタログは RFC 8259 の文字列エスケープを解除する。前述の JSON Pointer エントリ識別子規則を適用する。候補文字列値内で対応のないサロゲートを表すエスケープシーケンスがある場合、抽出は `resource_entry_unsupported` および `details.reason: "message_text_unrepresentable"` で失敗する。オブジェクトメンバー名内の対応のないサロゲートエスケープは、そのメンバーが少なくとも一つの文字列値候補へ至るパス上にある場合に限り、`resource_entry_unsupported` および `details.reason: "structural_path_unsupported"` で失敗する。位置には、そのパス内で最初の該当エスケープを使用する。そのメンバーの値サブツリー全体に文字列値候補がない場合、表現不能な名前は抽出範囲の外側に留まり、カタログを失敗させない。そのため、パーサーフロントエンドは、すべてのメンバー名を最初から Rust の `str` にすることなく候補への到達可能性を判断できるよう、生のメンバー名構造を十分に保持しなければならない。抽出範囲外にある対応のないサロゲートだけを原因とする依存ライブラリ側の拒否を、`resource_parse_failed` にしてはならない。フォーマット済み matcher など、複数行の MF2 フォーマット出力では、改行を一行の JSON 文字列値内の `\n` エスケープとして再エスケープする。

すべての JSON エントリは `CatalogKeyDomain::JsonPointer` を使用する。

JSON Pointer の比較は、コンテナー種別ではなく、意図的にトークンを基準とする。たとえば、`items` が `"0"` メンバーを持つオブジェクトであるか、最初の要素がメッセージである配列であるかにかかわらず、`/items/0` は同じ構造キーおよびカタログキーを表す。重複した祖先によって、一つのソース内に両方の具体形を配置できる場合、生順の出現番号によって `EntryKey` 値を区別する。一方、将来のファイル内重複ルールは、出現番号を除いた共有ポインターをグループ化する。ロケール間のオブジェクト対配列の形状差を診断する将来のルールは、別の構造チェックであり、エントリ識別子またはカタログ識別子を変更しない。

JSON アダプターは、すべてのエントリで `displayKey` を省略する。人間が読みやすい出力には、エントリの JSON Pointer `StructuralPathKey` を使用する。ドット連結または配列インデックスの表示構文を独自に作らない。

JSON カタログは、先頭に正確に一つの UTF-8 BOM を持ってよい。アダプターはそのバイトをホストスパン `0..3` に保持し、BOM より後のソースだけを JSON 構文パーサーへ渡し、すべてのパーサースパンおよびエラーオフセットに 3 バイトの基準オフセットを加算する。BOM はすべてのエントリとオフセットマップの外側にあり、書き戻しではバイト単位で保持する。二つ目の先頭 BOM、または JSON 文字列外の BOM コードポイントは通常の無効な JSON であり、`resource_parse_failed` を生成する。文字列内の `U+FEFF` はメッセージデータのままとする。JSON カタログは、単独 `.mf2` のファイルフレーミングを使用しない。末尾の `LF` または `CRLF`、末尾改行の欠落、および置換される値スパン外のその他すべてのバイトは変更しない。

JSON エントリの `raw_value_span` には、両側の二重引用符を含める。引用符はオフセットマップの `RawOnly` セグメントとし、その間の内容は恒等写像およびエスケープ解除セグメントを通じてマッピングする。このアダプターの JSON には、書き込み可能な別の引用符スタイルは存在しない。

再エスケープは、正確に二つの経路を使用する。

- 指定されたフォーマット済みメッセージが、エントリの元の `message_text` とバイト単位で同一の場合、統合書き戻しメソッドはそのエントリの置換を生成しない。そのため、変更されていないメッセージに対して、既存の省略可能なエスケープ表記を正規化することはない。
- メッセージが変更された場合、非公開の JSON 再エスケープ処理は、フォーマット済みメッセージ全体を一つの正規形 JSON 文字列値へシリアライズする。変更されていないメッセージ文字を元のホストエスケープ表記と対応付けようとはしない。

変更値用の正規シリアライザーは、開始および終了の `"` を出力し、次のスカラー規則を順に適用する。

- `U+0022` の quotation mark は `\"`、`U+005C` の reverse solidus は `\\` とする。
- `U+0008`、`U+0009`、`U+000A`、`U+000C`、`U+000D` は、それぞれ `\b`、`\t`、`\n`、`\f`、`\r` を使用する。
- `U+0000` から `U+001F` までのその他すべてのスカラーは、`\u` に正確に四桁の小文字十六進数を続ける。
- `U+002F` の solidus は `\/` ではなく `/` として出力する。
- `U+003C`、`U+2028`、`U+2029` を含む、その他すべての Unicode スカラー値は UTF-8 として直接出力する。変更値を ASCII だけの JSON へ変換しない。これはトップレベル JSON の規則である。後述するとおり、非公開の Vue インライン埋め込みコンテキストでは、解決後の文字列値を変更せず、`U+003C` を `\u003c` へ上書きする。

Rust の `str` 入力は対応のないサロゲートを含められない。そのため、そのような値へエスケープ解除される JSON ホスト値は、このシリアライザーへ到達する前に `resource_entry_unsupported` を生成する。シリアライザーを固定することで、変更値の出力をホストパーサー依存ライブラリから独立させ、CLI とエディターの各利用側で決定的にする。

### Vue SFC の `<i18n>` カスタムブロック

Single-file component の `<i18n>` ブロックは、`.vue` ホスト文書内にカタログ領域を埋め込む。これはアダプター合成である。外側のアダプターがブロック領域と宣言された言語を特定し、内側のカタログアダプターがその領域のテキスト上で動作し、オフセットマップを文書座標へ合成する。この合成に対してエントリモデルは閉じているため、利用側に公開する新しい振る舞いは必要ない。

初期 Vue アダプターは、インライン `<i18n>` ブロックだけを抽出する。`src` 属性を持つブロックは外部リソース参照であり、`.vue` 抽出アーティファクトにエントリを提供しない。リソース層は、そのパスを解決せず、推移的に読み込まず、存在を検証せず、SFC を経由して書き込まない。参照先ファイルは独立して `resources.catalogs` に一致しなければならず、その後、CLI およびエディターの利用側によって個別のカタログ対象として処理される。この明示的な外部リソース境界により、診断の重複を防ぎ、読み込み、エラー、キャッシュ識別子、および編集を、メッセージバイト列を実際に所有するファイルへ帰属させる。そのため、`src` ブロックは意図的にインライン候補集合の外側にあり、`resource_entry_unsupported` にはならない。

`src` ブロックは、空の内容スパンを持つか、HTML の空白文字（`U+0009`、`U+000A`、`U+000C`、`U+000D`、`U+0020`）だけを含んでよい。その内容は、外部参照周辺のレイアウトとして無視する。ブロックタグの間にその他の文字が現れる場合、`.vue` 対象全体の抽出を、`resource_document_unsupported`、`details.feature: "vue_src_with_inline_content"`、および最初の非空白文字を覆うスパンで失敗させる。`src` が存在するというだけでインラインカタログデータを黙って破棄することはなく、外部リソース境界よりインライン内容を優先することもない。

合成で使用するのは、この Tier の実装時点でリリース済みの内側アダプターだけである。`<i18n>` ブロックの既定値である JSON 言語のブロックは Tier 1 JSON アダプターを合成し、`lang="yaml"` または `lang="json5"` のブロックには、それぞれの Tier 3 アダプターが必要になる。インラインブロックが未リリースのアダプターを必要とする内側形式を宣言する場合、`.vue` 対象全体の抽出を `resource_format_unsupported` で失敗させる。同じ SFC 内の対応済みブロックを、部分的なアーティファクトとして返すことはない。未対応のインラインブロックが複数ある場合、埋め込み宣言位置が最初に始まるエラーを選択する。`src` ブロックは、内側形式のディスパッチ前に外部リソース参照として除外するため、その `lang` が `.vue` 対象内でこのエラーを発生させることはない。

外側のアダプターは、非公開の Vue インライン埋め込みコンテキストを、内側の書き戻し計画および実体化処理へ渡す。インラインブロック内で変更された JSON、JSON5、または YAML の各メッセージ値に対し、そのコンテキストは `U+003C` の less-than sign を、形式上有効なエスケープ `\u003c` として出力するよう要求する。トップレベルアダプターは通常のシリアライザー規則を維持する。JSON と JSON5 は、その正規形の変更済み文字列内で置換を適用する。YAML はエスケープを使用しながら、元の二重引用符スタイルを保持してよい。ただし、`<` を含む変更値は plain または一重引用符の出力を使用できず、正規形の二重引用符形式へフォールバックする。変更されていないメッセージは、既存の高速パスを通じて、元の正確な生の綴りを保持する。

すべての `<` に適用するこの保守的な規則により、リソース層がパーサー固有の終了タグ先読みまたは大文字小文字規則を複製することなく、フォーマット済み値が `</i18n>` を含む新しいリテラル HTML/SFC 終了タグ開始部を実体化することを防ぐ。計測ではコンテキスト固有のエスケープ済みバイト長を使用し、実体化ではその正確な計画を出力しなければならず、外側の SFC 全体の再解析および再抽出は引き続き多層防御として必須とする。したがって、`<` を含む有効な変更済みメッセージは、トップレベルの内側シリアライザーであればリテラルとして出力するという理由だけで `resource_write_back_failed` へ落ちるのではなく、通常の検証済み候補を生成しなければならない。

`lang` 属性のないインラインブロックは `json` を選択する。属性がある場合、トリムまたは大文字小文字の畳み込みを行わず、正確に照合する。認識する宣言は `json`、`json5`、`yaml`、`yml` であり、`yml` だけを `yaml` レジストリ ID へ正規化する。値なしまたは空の属性、前後の空白、`JSON` などの値、および未知の値は JSON へフォールバックせず、内容スニッフィングも発生させない。これらは `resource_format_unsupported` を生成する。認識済み ID を使用できるかどうかは、対応する内側アダプターがリリース済みかどうかに引き続き依存する。これらの規則では、SFC 構文パーサーが返す静的属性値を使用し、Vue 式を言語宣言として解釈することはない。

トップレベル JSONC アダプターのリリース後も、`jsonc` は意図的に Vue 埋め込み言語宣言へ追加しない。そのため、インラインの `lang="jsonc"` ブロックは、埋め込みの `resource_format_unsupported` を生成する。`src` から参照される `.jsonc` ファイルは、その外部パスが `resources.catalogs` にオプトインされていれば、引き続き独立して処理できる。Vue アダプターが `lang="json"` を寛容な JSONC として扱うことは決してない。

インライン `<i18n>` ブロックの静的 `locale` 属性は、将来の `locale: { "from": "host" }` 関連付けについて、Vue アダプターが文書化する提供元となる。カタログレベルのロケール利用側がその関連付けを有効にすると、ブロックから持ち上げられた各エントリは、ホストが提供するロケールとして属性の正確な静的文字列を受け取る。値をトリム、正規化、または BCP 47 として検証しない。`locale` 属性がない、値がない、または空である場合、これらのエントリにはホストロケールがなくなる。黙って除外するのではなく、将来の提供元欠落エラー契約に従う。初期 Tier 2 の抽出、メッセージレベルの lint、およびフォーマットでは、この値を `MessageEntry` へ付加せず、この値に基づいて振る舞いを変えない。`src` ブロックの `locale` 属性が、独立して処理される参照先カタログへ伝播することはない。

`crates/intlify_resource` は、組み込み Vue SFC 外側アダプターと、内側カタログアダプターとの合成を所有する。利用側は完全な `.vue` ホスト文書を共有レジストリへ渡し、`<i18n>` ブロックを独立して特定してはならない。外側のアダプターは、ブロック内容のスパンと宣言された言語を保持し、対応済みのインライン領域を絶対内容開始位置順に走査し、リリース済みの各内側アダプターを一つの外側アーティファクトビルダーに対して呼び出す。内側アダプターは、中間 `ExtractedCatalog` を公開するのではなく、持ち上げたエントリ、最終的な絶対オフセットマップセグメント、および非公開の書き戻し状態を、そのビルダーへ直接出力する。これにより、CLI とエディターの利用側で、分類、抽出、リソース計上、および書き戻しを同一に保つ。

持ち上げたエントリの `StructuralPathKey` は、内側アダプターの構造パスのままとし、Vue ブロックインデックスまたはソースオフセットを含めない。エントリは絶対生スパン順に追加し、共有ビルダーは、同一の各構造パスについて SFC 全体で `occurrence` を割り当てる。したがって、二つのブロックにある同じ `/greeting` パスは、出現番号 `0` と `1` になる。ハンドルを発行するのは、完成した外側アーティファクトだけである。各エントリのブロック識別子と内側アダプター状態を、マッピングおよび書き戻し用に非公開で保持し、内側のハンドルは存在しない。`CatalogKeyDomain`、`CatalogKey`、`displayKey` も同様に、内側アダプターの比較、論理、および表示用識別子のままとするが、外側アーティファクトの共有インターナーを使用する。そのため、インラインブロックが異なる内側形式を使用する場合、Vue アーティファクトは複数のカタログキードメインのエントリを含み得る。これにより、論理キーへブロック位置を混入させない。完全な `EntryKey` は、重複した出現箇所間のソース順を意図的に反映する。

外側アダプターを所有することは、リソースクレート自身が Vue 構文解析を実装しなければならないことを意味しない。アダプターに必要なスパン保持 SFC 構文情報を公開し、利用側に公開する二つ目の抽出経路にならない限り、専用のパーサー依存ライブラリまたは非公開のワークスペースヘルパーを使用してよい。

解析エラーでは、ソースを拒否したアダプター層を識別する。無効な SFC 構文は、`details.format: "vue"` を持ち、`outerFormat` を持たない `resource_parse_failed` を生成する。SFC は有効だがインラインカタログが無効な場合、`format` は正規化済みの内側レジストリ ID、`outerFormat` は `"vue"` となる。どちらの場合も、外側アダプターが内側エラー位置を持ち上げた後の `offset`、`line`、`column` は、完全な `.vue` ソース内の絶対座標となる。別のブロックの解析が失敗した場合、対応済みのブロックを返すことはない。

内側アダプターが `resource_entry_unsupported` または `resource_document_unsupported` を返す場合にも、失敗層の帰属について同じ規則を適用する。`format` は内側アダプター、`outerFormat` は `vue` を示し、外側アーティファクトは主要な未対応位置を絶対 SFC 座標へ持ち上げる。一方、SFC 構造自体の失敗では `format: "vue"` を使用し、`outerFormat` を省略する。

### YAML カタログ

YAML カタログは、YAML 1.2 Core Schema のもとで、タグ解決後に文字列となるスカラーからエントリを解決する。アダプターはこのスキーマを明示的に設定し、パーサー依存ライブラリの既定値または YAML 1.1 互換モードを継承しない。そのため、plain の `true`、`false`、`null`、および認識される数値形式は文字列以外のスカラーとなる。一方、`yes`、`on`、およびタイムスタンプ形状のテキストなどの値は、対応済みの明示タグが型を変更しない限り文字列のままとなる。fixture により、依存ライブラリの更新をまたいでこれらの境界を固定する。

個別の構造パスがたまたま JSON 互換であるエントリも含め、すべての YAML エントリは `CatalogKeyDomain::YamlTypedPath` を使用する。

`%YAML` ディレクティブがない場合、この固定された YAML 1.2 の振る舞いを選択する。構文的に有効な `%YAML 1.2` ディレクティブは、その振る舞いを変更せずに受け入れる。`1.1` または将来のバージョンを含む、その他の構文的に有効な宣言バージョンは、ディレクティブトークン位置で、文書全体の抽出を `resource_document_unsupported` および `details.feature: "yaml_version"` で失敗させる。アダプターは、文書ごとにスキーマを切り替えたり、宣言を黙って無視したりしない。不正な形式のディレクティブは引き続き無効なホスト構文であり、代わりに `resource_parse_failed` を使用する。

YAML カタログは、JSON と同じソース保持規則のもとで、先頭に正確に一つの UTF-8 BOM を持ってよい。アダプターはバイト `0..3` を保持し、その後のソースを解析し、すべてのスパンとエラーオフセットを 3 バイト分持ち上げる。書き戻しは BOM をバイト単位で保持する。二つ目の先頭コードポイントを含む、それ以降の `U+FEFF` はフレーミングではなく、通常のソースまたはスカラーデータとして YAML パーサーへ渡す。パーサーが拒否した場合は `resource_parse_failed` とする。固定 YAML 規則が文字列内で許可する `U+FEFF` はメッセージテキストのままとする。YAML カタログ処理では単独 `.mf2` の末尾改行フレーミングを適用しないため、置換されるスカラースパン外のすべてのバイトは、正確な改行表記を保持する。

明示タグは、標準の `!!` 省略記法を含め、YAML 1.2 Core タグ集合である `tag:yaml.org,2002:str`、`null`、`bool`、`int`、`float`、`seq`、`map` に制限する。したがって、明示的な文字列タグにより、`!!str true` は文字列エントリとなる。一方、対応済みの明示的な非文字列タグは、宣言された型を維持する。この集合外のローカルまたはグローバルタグを持つノードは、タグトークン位置で、抽出全体を `resource_document_unsupported` および `details.feature: "custom_tags"` で失敗させる。アダプターがタグを無視し、基礎となるスカラーまたはコレクションから型を推測することは決してない。宣言されたが一度も使用されない `%TAG` ディレクティブは、抽出を失敗させない。

初期 YAML アダプターは、YAML 文書を一つだけ受け入れる。空のストリームと一つの空文書は、エントリ数ゼロの有効なカタログである。二つ目の文書に到達した場合、その文書の開始位置で抽出全体を `resource_document_unsupported` で失敗させる。最初の文書から得たエントリを返したり処理したりしない。文書終了マーカー自体が別文書を生成することはなく、省略可能な最初の文書開始マーカーは引き続き有効である。

重複したマッピングキーは、拒否または集約せず保持する。アダプターは、スパンを保持するノードまたはイベント表現を使用し、具体的な値を生のソース順に走査し、前述した解決済みタグと値による正規パス表現でマッピングキーを比較する。同一の完全な構造パス配下の各文字列リーフは、重複した祖先マッピング配下のリーフも含め、文書全体で出現番号 `0`、`1` 以降を持つ別々のエントリとなる。マップへデシリアライズする、ノードを調べる前に重複を拒否する、またはランタイム上の勝者だけを保持する依存ライブラリのモードは、このアダプターには適さない。抽出では first-wins または last-wins のランタイムセマンティクスを決定しない。将来のカタログレベル重複キー lint が曖昧性を報告する一方で、具体的な各生値は独立して診断および書き込みできる状態に保つ。

この Tier の実装時点で、plain、一重引用符、二重引用符の各文字列スカラーは読み書き可能とする。literal および folded ブロックスカラーは、診断用の正確なオフセットマップを生成するが、初期 YAML アダプターマイルストーン全体では読み取り専用のままとする。ブロックスカラーの書き込み対応は、Tier 3 YAML アダプターのリリース条件ではなく、別の後続マイルストーンとする。

ブロックスカラーの `raw_value_span` は、`|` または `>` インジケーターから始まり、ヘッダー全体と、そのスカラーが所有する本文行を含み、スカラー外にある後続の最初のトークンまたは行より前で終わる。先行する明示タグまたはアンカーは、そのスパンの外側にある。ヘッダー、省略可能なインデント／chomping インジケーターまたはコメント、および除去された各本文インデントプレフィックスは `RawOnly` とする。解決後も変更されず残る内容バイトは `Identity` を使用する。解決後に折り畳まれた空白または保持された改行となる各物理改行単位は、一つの `Unescape` セグメントを使用するため、診断がその変換を分割することはない。strip または clip chomping によって除去された改行は `RawOnly` とし、clip で一つだけ保持される改行と keep で保持されるすべての改行は、対応するメッセージ改行へマッピングする。空行と追加インデントされた行は、単純化した改行置換ではなく、同じ YAML 1.2 折り畳み規則に従う。

アダプターは、解析後のデコード済み部分文字列を検索するのではなく、スパンを保持するブロックトークンからこれらのセグメントを導出しなければならない。対応済みブロックスカラーに対して完全な検証済みマップを構築できない場合は、`details.reason: "resource_offset_map_invariant_failed"` を持つ `internal_error` とする。診断範囲をスカラー全体へ格下げしてはならない。有効なブロックスカラーは引き続き `read_only: true` として返し、通常のエントリ単位 lint およびフォーマッター評価へ参加する。フォーマッターはパーサー診断を保持するが、正常なフォーマット結果を破棄する。そのため、共有の読み取り専用契約に従って、書き戻しと差分報告だけをスキップする。

複数行の plain、一重引用符、および二重引用符スカラーは読み書き可能なままとし、同様に正確なマップを必要とする。その `raw_value_span` は、周囲の引用符を含み、先行するタグまたはアンカーを除外しながら、スカラー表現全体と所有する継続行を覆う。引用符の区切り、除去された継続行インデント、および値ではない構文は `RawOnly`、変更されない内容の連続区間は `Identity`、二重化された一重引用符、二重引用符内のエスケープ、および解決後に折り畳まれた空白または保持された改行となる物理改行は、不可分な `Unescape` セグメントとする。`LF`、`CRLF`、`CR` の各物理改行は一つの生データ単位とし、マッピング範囲が `CRLF` を分割しないようにする。後続のインデントを含む、二重引用符内でエスケープされた行継続は、メッセージバイトを生成せず `RawOnly` とする。これらのマップは、部分文字列検索ではなく、YAML 1.2 折り畳み規則のもとでパーサートークンから導出する。

このようなスカラーのフォーマットでは、後述する元スタイルでの試行と正確な再解析検査を使用する。変更された複数行値が元の表現を正確に維持できない場合、アダプターは完全な生スカラースパンを、物理一行の正規二重引用符形式へ置換する。タグ、アンカー、周囲のコメント、およびそのスパン外のすべてのバイトは変更しない。

YAML の再エスケープには JSON と同じ未変更値用の高速パスがある。フォーマット済みメッセージテキストが `message_text` とバイト単位で同一の場合、統合メソッドはそのエントリの置換を生成しない。変更された plain、一重引用符、または二重引用符スカラーについて、リソース層が所有するスタイルシリアライザーは、スカラーの既存の構文コンテキストで新しい値を曖昧なく表現できる場合に限り、最初に元のスタイルで候補を生成する。アダプターは候補を再解析し、解決済みタグが文字列のままであり、解決済み値がフォーマット済みメッセージと正確に等しい場合に限って受け入れる。元スタイルの構築または検証に失敗した場合、代わりにリソース層が所有する正規二重引用符形式を生成する。綴りまたはスタイル選択を依存ライブラリの emitter へ委譲することはない。これらのスカラー単位の検査後も、`build_and_validate_write_back` 内部の最終的な文書全体検証は必須である。

plain スタイル候補は、フォーマット済みメッセージをそのまま使用する。物理一行で、C0 または C1 制御文字、および `U+2028` または `U+2029` を含まない場合に限って試行する。コンテキスト依存インジケーター、前後の空白、コメント、および暗黙の Core 型付けは、必須の再解析／タグ／値検査へ委ねる。一重引用符候補は同じ一行文字条件を持ち、値を `'` で囲み、埋め込まれた各 `'` を `''` へ置換する。バックスラッシュと二重引用符はリテラルのままとする。そのため、変更された複数行値は YAML の折り畳みに依存せず、二重引用符形式へフォールバックする。

正規二重引用符シリアライザーは、開始および終了の `"` を持つ物理一行を出力し、次の規則を順に適用する。

- `U+0022` の quotation mark は `\"`、`U+005C` の reverse solidus は `\\` とする。
- `U+0008`、`U+0009`、`U+000A`、`U+000C`、`U+000D` は、それぞれ `\b`、`\t`、`\n`、`\f`、`\r` を使用する。
- `U+0000..U+001F` または `U+007F..U+009F` にあるその他すべてのスカラーは、`\x` に正確に二桁の小文字十六進数を続ける。
- YAML が物理改行として解釈しないよう、`U+2028` と `U+2029` は `\u2028` と `\u2029` を使用する。
- `U+002F` の solidus は `/` として出力する。トップレベル YAML カタログでは、`U+003C` を含むその他すべての Unicode スカラー値を UTF-8 として直接出力する。変更値を ASCII だけの YAML へ変換せず、`\0`、`\a`、`\e`、`\N`、`\_`、`\L`、`\P` など、省略可能な YAML 名前付きエスケープを使用しない。非公開の Vue インラインコンテキストでは、前述の合成規則に従って、代わりに `U+003C` を `\u003c` とする。

その後、シリアライザーは固定 Core Schema のもとでこのスカラーを再解析し、文字列値が正確に一致することを要求する。予期しない不一致は、`details.reason: "resource_write_back_failed"` を持つ `internal_error` とする。依存ライブラリが選択する別の綴りへフォールスルーすることは決してない。

直接定義されたノード上のアンカー注釈は、そのノードの抽出を変更しない。定義位置のスカラーは通常のエントリを生成し、参照されないアンカーは受け入れる。実際に使用される alias ノードは、抽出全体を `resource_document_unsupported` および `details.feature: "aliases"` で失敗させる。alias パスを黙って省略するとランタイムカタログと乖離し、展開すると複数の論理エントリが一つの生ソース値を共有してしまうためである。YAML のマージキーも同様に `details.feature: "merge_keys"` で失敗する。`<<: *defaults` では、ネストされた alias トークンではなく、マージキートークンを主要な未対応位置とする。初期アダプターでは、alias またはマージ展開から部分的または読み取り専用のエントリを生成しない。

ブロックスカラーエントリを書き込み可能へ昇格させるのは、アダプターが literal および folded の両形式、インデントインジケーター `1` から `9`、strip／clip／keep chomping、空および先頭空の内容、末尾改行と末尾空行、および追加インデントされた行について、値同一の再エスケープを支援した後に限る。fixture は生成されたすべての候補を再解析し、解決済み文字列値をフォーマット済みメッセージテキストと比較し、フォーマットの冪等性を証明しなければならない。これらの要件を実装するまで、lint はマッピング済み診断を通常どおり報告する。fmt もパーサー診断を報告するが、書き込みモードは置換を構築せず、`--check` は正常なフォーマット差分として数えない。

後続マイルストーンでブロックスカラーを書き込み可能にした後、再エスケープには固定されたフォールバック順を使用する。最初に、必要に応じてインデントおよび chomping インジケーターを調整しながら、再解析によってフォーマット済みメッセージテキストを正確に生成できる場合は、元の literal または folded スタイルを保持する。folded スタイルで値を同一に表現できない場合は literal スタイルへ切り替える。literal スタイルでも安全に表現できない場合は二重引用符スタイルへ切り替える。各候補を再解析し、値が正確に等しい場合に限って受け入れる。すべての候補が失敗した場合は `resource_write_back_failed` 内部エラーとし、ファイルへ書き込まない。

### JSONC および JSON5 カタログ

`jsonc` アダプターは、RFC 8259 JSON 上に固定された ox-mf2 プロファイルを実装する。JSON の空白が許可されるすべての場所で、JavaScript 形式の `//` 行コメントと、ネストしない `/* ... */` ブロックコメントを追加する。また、空でないオブジェクトの最後のメンバー、または空でない配列の最後の要素の後に、省略可能な末尾コンマを一つ許可する。配列の要素省略、`#` コメント、一重引用符文字列、引用符なしのメンバー名、十六進数または非有限数、その他の JSON5 構文は許可しない。これは、パーサーの既定値を継承するのではなく、[JSONC 仕様](https://jsonc.org/)がプロファイルごとに選択可能としている末尾コンマのオプションを、意図的に有効にするものである。

JSONC の文字列値、メンバー名の解決、重複出現識別子、省略可能な単一の先頭 BOM、オフセットマッピング、変更されていない生の綴りの保持、変更値の正規シリアライズ、および候補全体の検証には、Tier 1 JSON と正確に同じ規則を使用する。コメントと末尾コンマは、文字列値スパンの外側にあるホスト構文であり、MF2 のフォーマット中もバイト単位で同一のままとする。パーサー依存ライブラリは、それらのスパンと重複ノードを保持しなければならない。完全なソースマップなしにコメントを除去して一時 JSON バッファを作るだけでは不十分である。

すべての JSONC エントリは `CatalogKeyDomain::JsonPointer` を使用する。そのため、ホストパーサーの文法が異なっても、カタログキーを同じ JSON または JSON5 キーと比較できる。

JSONC はトップレベルのカタログ形式に限る。Vue `<i18n>` の内側言語 allowlist には追加しない。外部 `.jsonc` リソースは、前述した `src` 境界のもとで別の対象のままとする。

`json5` アダプターは、任意の JavaScript オブジェクトリテラルパーサーまたは依存ライブラリ固有の寛容モードではなく、完全な [Standard JSON5 1.0.0 文法](https://spec.json5.org/)を実装する。JSON5 のコメント、末尾コンマ、識別子形式のメンバー名、拡張された数値構文、一重および二重引用符文字列、追加のエスケープ、および行継続は、その文法が定義する場合に限って受け入れる。文字列以外の値は、メッセージ候補集合の外側に留める。文字列および識別子形式のメンバー名は、JSON Pointer 構造識別子用に正確な文字列値へ解決する。そのため、一つの解決済み名に対する別表記はパスを共有し、出現番号を使用する。構文上許可された引用符付きメンバー名を Rust 文字列にできない場合は、JSON の候補到達可能性規則に従う。文字列値候補へ至るパス上にある場合だけ `structural_path_unsupported` とし、それ以外では抽出範囲外に留める。

すべての JSON5 エントリは `CatalogKeyDomain::JsonPointer` を使用する。

JSON5 は、JSON 系の先頭 BOM 一つという規則ではなく、その文法の `U+FEFF` 規則に従う。文字列外の任意個数の `U+FEFF` コードポイントは、許可されたすべての位置で通常のトークン区切り空白となり、ホスト文書内でバイト単位に同一のままとする。アダプターは解析前に先頭の一つを除去しない。文字列内の `U+FEFF` はメッセージデータである。そのため、すべての絶対スパンとエラーは生の UTF-8 バイトを一つ残らず直接数え、書き戻しでは置換対象値スパン外の空白を変更しない。

JSON5 文字列エントリの `raw_value_span` には、正確な一重または二重引用符の区切りを含める。区切りと、行継続内のバックスラッシュおよび完全な物理行終端列は `RawOnly`、変更されない内容は `Identity`、各エスケープは不可分な一つの `Unescape` 単位とする。後者には、一つのスカラーへ解決される複合 UTF-16 サロゲートエスケープも含む。`CRLF` を分割することは決してない。対応のないサロゲートへ解決される文字列エスケープは、JSON の UTF-8 表現可能性規則と同様に、抽出全体を `resource_entry_unsupported` および `details.reason: "message_text_unrepresentable"` で失敗させる。

フォーマット済みメッセージが変更されていない場合、JSON5 の再エスケープは、元の完全な生文字列をバイト単位で保持する。変更された値の場合、元の引用符区切りを保持し、次の規則で物理一行を出力する。

- 使用中の区切りは、一重引用符文字列では `\'`、二重引用符文字列では `\"` とする。もう一方の引用符はリテラルのままとし、reverse solidus は `\\` とする。
- `U+0008`、`U+0009`、`U+000A`、`U+000C`、`U+000D` は、それぞれ `\b`、`\t`、`\n`、`\f`、`\r` を使用する。
- `U+0000..U+001F` または `U+007F..U+009F` にあるその他すべてのスカラーは、`\x` に正確に二桁の小文字十六進数を続ける。特に、変更された出力ではコンテキスト依存の `\0` または省略可能な `\v` を使用しない。
- `U+2028` と `U+2029` は `\u2028` と `\u2029` を使用する。solidus は `/` として出力する。
- トップレベル JSON5 カタログでは、`U+003C` を含むその他すべての Unicode スカラー値を UTF-8 として直接出力する。出力を ASCII だけにせず、行継続を使用しない。非公開の Vue インラインコンテキストでは、前述の合成規則に従って、代わりに `U+003C` を `\u003c` とする。

リソース層が所有するシリアライザーは、文字列または識別子形式のメンバー名の綴りを変更しない。それらのトークンはメッセージ値スパンの外側にあるためである。候補全体を Standard JSON5 1.0.0 として再解析し、通常の統合書き戻し契約のもとで、キー列と正確なメッセージ値の両方を保持しなければならない。依存ライブラリのジェネレーターは決して使用しない。

### XLIFF

XLIFF は XML ホスト形式である。XLIFF 1.2 は `<trans-unit>` 内の `<source>`／`<target>` 要素に、XLIFF 2.x は `<unit>`／`<segment>` 内の `<source>`／`<target>` 要素にメッセージテキストを格納する。エントリキーは file／group／unit と、存在する場合は segment の識別子パスをシリアライズする。エスケープ解除では XML エンティティ、CDATA セクション、および XML の行末正規化を扱う。`xml:space` はホストメタデータとして保持するが、アダプター定義のテキスト正規化を発生させない。

アダプターは、ルート要素の展開済み namespace 名と正確な `version` 属性から、明示的なプロファイルを一つ選択する。初期対応する組み合わせは次のとおりである。

| プロファイル | ルート namespace                        | `version` |
| ------------ | --------------------------------------- | --------- |
| XLIFF 1.2    | `urn:oasis:names:tc:xliff:document:1.2` | `1.2`     |
| XLIFF 2.0    | `urn:oasis:names:tc:xliff:document:2.0` | `2.0`     |
| XLIFF 2.1    | `urn:oasis:names:tc:xliff:document:2.0` | `2.1`     |
| XLIFF 2.2    | `urn:oasis:names:tc:xliff:document:2.2` | `2.2`     |

namespace prefix は任意であり、照合には参加しない。namespace または version の欠落、組み合わせの不一致、および未知の将来バージョンは、有効な XML ではあるが未対応の XLIFF プロファイルである。抽出全体を `resource_document_unsupported` および `details.feature: "xliff_version"` で失敗させる。不正な形式の XML は引き続き `resource_parse_failed` とする。別バージョンの追加には、明示的でテスト済みのプロファイルが必要である。アダプターは、任意の `2.x` 文書が互換性のある構造を持つと仮定しない。プロファイル選択では抽出に必要な XLIFF 構造語彙を認識するが、このリソース層を完全な XSD または XLIFF 制約バリデーターにはしない。

明示的にオプトインされた XLIFF カタログでは、対応済みの主要なプレーンテキスト `<source>` および `<target>` 要素を、それぞれ独立した読み書き可能な MF2 エントリとする。XLIFF 1.2 の候補は、1.2 namespace の `<trans-unit>` の直接の子であり、XLIFF 2.x の候補は、選択された 2.x namespace の `<segment>` の直接の子である。照合では、namespace prefix または local name だけではなく、展開済みの名前と正確な親子関係を使用する。構造パスは unit または segment 識別子の後に、明示的な `source` または `target` の役割セグメントを含む。そのため、二つの側を区別するためだけに出現番号へ依存することはない。片側がない場合、合成エントリを生成しない。カタログのオプトインによって両側のフォーマットを許可し、同じ値同一 XML 書き戻し契約を使用する。初期設定には、形式固有の source／target 選択を追加しない。

XLIFF 1.2 の `<alt-trans>`、XLIFF 2.x の `<ignorable>`、メタデータ、拡張要素、またはその他の主要ではないコンテキスト内にある `<source>` または `<target>` 要素は、意図的に候補集合の外側とする。その内容がプレーンテキストであっても、エントリも未対応エントリエラーも生成しない。これは、アダプターの対応範囲不足ではなく、意味上のスコープ境界である。初期アダプターは、カタログで有効なメッセージを表す主要翻訳値だけを処理する。

XLIFF は、将来の `locale: { "from": "host" }` 関連付け向けに、内在ロケールの提供元も定義する。XLIFF 1.2 では、主要 source エントリは、有効な `xml:lang` があればそれを使用し、なければ包含する `<file source-language>` を使用する。target は有効な `xml:lang` を使用し、なければ `<file target-language>` を使用する。XLIFF 2.x で対応するフォールバックは、ルートの `<xliff srcLang>` と `<xliff trgLang>` である。有効な `xml:lang` は、通常の XML 祖先継承に従う。解決済み文字列は、BCP 47 検証、大文字小文字の畳み込み、または正規化を行わず、正確に保持する。このメタデータは、エントリレベルの抽出、lint、またはフォーマットに影響せず、将来のカタログリンターがホストロケール関連付けを明示的に有効にした場合だけ使用する。

`translate`、`state`、承認、レビュー属性などの XLIFF ワークフローメタデータは、抽出または書き込み可能性を変更しない。リソース層は TMS 状態機械ではない。バージョン固有の継承を編集ロックとして解釈せず、フォーマット後に status、timestamp、provenance、または approval を変更しない。これらの属性は生メッセージスパンの外側に留まり、バイト単位で保持する。エントリを読み取り専用にするのは、そのホスト表現が値同一の再エスケープを満たせない場合だけである。XLIFF ファイルをフォーマッター処理へ明示的にオプトインすることで、それ以外では書き込み可能な主要 source および target 値について、構文だけのフォーマットを許可する。

XLIFF の構造パスは、形式固有の型付きセグメントとともに、型付き YAML ポインターと同じ、スラッシュ区切りの `~0`／`~1` エスケープ規則を使用する。XLIFF 1.2 では、空でない `<file original>` 値を `file:original:<value>` に使用し、`<group id>` または `<trans-unit id>` を `group:id:<value>` および `unit:id:<value>` に使用する。XLIFF 2.x では、`<file>`、`<group>`、`<unit>`、`<segment>` 上の空でない `id` 値を、それぞれ `file:id:<value>`、`group:id:<value>`、`unit:id:<value>`、`segment:id:<value>` として使用する。ネストされた group は、階層ごとにセグメントを一つ追加する。末尾のセグメントは、常に `side:source` または `side:target` とする。

識別子属性がないか空の場合、`segment:i:0` のように、`<kind>:i:<zero-based same-kind sibling index>` を使用する。アダプターは、XLIFF スキーマが要求する識別子を適用するためだけに文書を失敗させない。属性ペイロードは、ポインターエスケープ前にトリム、大文字小文字の畳み込み、または Unicode 正規化を行わない、XML 解決済み Unicode 値とする。重複した属性識別子は意図的に同じ構造パスを生成し、具体的なエントリは生のソース順で文書全体の出現番号を受け取る。代表的なパスの例は、`/file:original:app.json/group:id:menu/unit:id:welcome/side:source` と `/file:id:f1/group:i:0/unit:id:u1/segment:id:s1/side:target` である。ランタイム向けの `displayKey` はこの階層を短縮してもよいが、レポートおよびキャッシュの識別子には完全なパスを使用する。

XLIFF の `CatalogKey` は、同じシリアライズ済み階層から、末尾の `side:source` または `side:target` セグメントを除いたものとする。XLIFF 1.2 のエントリは `CatalogKeyDomain::Xliff12`、XLIFF 2.0、2.1、2.2 のエントリは `CatalogKeyDomain::Xliff2` を使用する。これにより、一つの unit または segment 配下の両エントリは、異なる具体的な `EntryKey` 値を維持しながら、`/file:original:app.json/group:id:menu/unit:id:welcome` または `/file:id:f1/group:i:0/unit:id:u1/segment:id:s1` を論理メッセージ識別子として共有する。一つの選択済み XLIFF プロファイルファミリー内では source と target を比較できるが、構造の異なる 1.2 と 2.x ファミリーの同じシリアライズ文字列は比較できない。そのため、ホストロケール関連付けは、診断、キャッシュ、または書き戻しから物理的な役割を消去せず、source と target をロケールバリアントとして比較できる。

DTD 処理は、すべてのエンティティ解決または外部アクセスより前に無効化する。整形式の `DOCTYPE` 宣言は、`<!DOCTYPE` の位置で、抽出全体を `resource_document_unsupported` および `details.feature: "xml_dtd"` で失敗させる。アダプターは system identifier を読み込まず、ネットワーク要求を行わず、内部エンティティ宣言を展開しない。不完全な宣言を含む不正 XML は、引き続き `resource_parse_failed` とする。テキストと識別子属性に使用できるのは、あらかじめ定義された五つの XML エンティティと、十進または十六進の数値文字参照だけである。各参照は、一つの `Unescape` オフセットマップセグメントとする。宣言されていない名前付きエンティティは無効な XML であり、`resource_parse_failed` を生成する。コメント、CDATA セクション、または通常のエスケープ済み文字データ内にあるバイト列 `<!DOCTYPE` は、宣言ではなくデータである。

XLIFF 入力には、引き続き共有 UTF-8 ホストソース契約と XML 1.0 処理規則を適用する。Tier 3 XLIFF 対応を実装した時点で、CLI 入力層は、分類済み XLIFF 対象の生プレフィックスを、汎用 UTF-8 デコード前に検査する。UTF-16 little-endian BOM（`FF FE`）または big-endian BOM（`FE FF`）は、`details.reason: "unsupported_encoding"` および `details.detectedEncoding: "utf-16le"` または `"utf-16be"` を持つ `input_read_failed` を生成する。対象をトランスコードしたり、アダプターへ渡したりしない。BOM のない UTF-16 は推測せず、通常の `invalid_utf8` 入力失敗のままとする。この事前検査は、エンコーディング依存のスパンマッピングまたは書き戻しを導入せず、一般的な TMS 出力に対するエラーを改善する。

UTF-8 のデコード成功後、XML 宣言がない場合は XML 1.0 を選択する。宣言には正確な `version="1.0"` が含まれなければならない。その他の構文的に有効なバージョンは、その version 値の位置で、抽出全体を `resource_document_unsupported` および `details.feature: "xml_version"` で失敗させる。アダプターは XML 1.1 の文字または行末規則を適用しない。

宣言では `encoding` を省略してよい。存在する場合、encoding 名は XML の ASCII 大文字小文字を区別しない比較において `UTF-8` と等しくなければならない。その他の構文的に有効な encoding 宣言は、その encoding 値の位置で、抽出全体を `resource_document_unsupported` および `details.feature: "xml_encoding"` で失敗させる。アダプターはトランスコードも無視もしない。XML で必須の version がない場合を含む、不正な形式の宣言は `resource_parse_failed` とする。

先頭の UTF-8 BOM は正確に一つだけ受け入れ、バイト `0..3` に保持し、XML 解析時だけ除去し、すべてのスパンとエラーオフセットへ加算して戻す。書き戻しはバイト単位で保持する。二つ目の先頭 `U+FEFF` は通常の XML ソースであり、XML 文法が拒否する場合は失敗する。一方、`<source>` または `<target>` の文字データ内で認められた `U+FEFF` はメッセージテキストのままとする。XLIFF カタログ処理では、単独 `.mf2` の末尾改行フレーミングを適用しない。

メッセージ値は、あらかじめ定義されたエンティティまたは数値エンティティの解決と、XML 行末正規化を行った後の、正確な XML 文字データ列である。アダプターは、先頭または末尾の空白をトリムせず、内部の連続区間を折り畳まず、フォーマット用インデントを除去せず、有効な継承済み `xml:space` 値に基づいて改行を追加しない。そのため、この層では `default` と `preserve` の両方が同じ文字列を保持する。アプリケーションは後からそのメタデータを解釈してよいが、リソース抽出は、正確な値の契約を満たしながらアプリケーション固有の空白方針を推測できない。

XML が `LF` へ正規化する生の `CRLF` および `CR` 単位は、不可分な `Unescape` マップセグメントとする。変更されず残る生の `LF` は `Identity` とする。通常の文字データ内と CDATA 内で同じ規則を使用する。属性の綴り、namespace 宣言、および `xml:space` 値はメッセージスパンの外側に留まり、書き戻しでバイト単位に保持する。

自己終了しない `<source>` または `<target>` の `raw_value_span` は、開始タグ末尾と終了タグ開始位置の間にある完全なバイト範囲とする。エンティティ参照は不可分な `Unescape` セグメント、CDATA の開始および終了区切りは `RawOnly`、XML 処理後も残る CDATA および通常テキストのバイトは `Identity` または前述の行末セグメントとする。隣接する通常テキスト、参照、および CDATA チャンクは、一つのメッセージと一つのオフセットマップを形成する。

自己終了する `<source/>` または `<target/>` は、省略または読み取り専用の特殊ケースではなく、書き込み可能な空文字列エントリである。その `raw_value_span` は、自己終了要素全体とする。マップは、タグを先頭および末尾の `RawOnly` 構文に分割し、唯一の空メッセージ位置を終了 `/>` の直前に置く。変更されていない空の値は、元の要素をバイト単位で返す。空でない置換が要求された場合、再エスケープは末尾の `/` だけを除去し、結果の開始タグ内で正確な生の qualified name、prefix、属性、引用符、および空白を保持し、正規 XML 文字データを挿入し、同じ qualified name を使用した終了タグを追加する。展開された要素を書き込めるのは、文書全体の再解析によって、同じエントリ識別子と期待値を確認した後だけである。

一方、`<target></target>` のように明示的に対となる空要素は、挿入位置に長さゼロの内容スパンを持ち、通常の内容置換経路で書き込み可能なままとする。置換検証ではこのような長さゼロのスパンを許可するが、同じエントリに対する重複置換は引き続き拒否する。異なる空要素は、異なる絶対挿入オフセットを持つ。

フォーマット済みメッセージが変更されていない場合、XLIFF の再エスケープは、元の内容スライス全体をバイト単位で保持する。変更された値の場合、内容スパン全体を、リソース層が所有する一つの正規 XML 文字データ列へ置換する。新しい値を古いチャンクへ対応付けたり、CDATA 境界および省略可能なエンティティ表記を保持したりはしない。正規テキストには、次の規則を順に適用する。

- `&` は `&amp;`、`<` は `&lt;` とする。
- `>` は、禁止された文字データ列 `]]>` を完成させる場合を除いてリテラルのままとする。その場合、その `>` を `&gt;` とする。
- 再解析によって `LF` へ正規化されないよう、`U+000D` は十六進数値参照 `&#xd;` とする。Tab と `LF` はリテラルのままとする。
- 置換は属性ではなく要素内容であるため、二重引用符とアポストロフィはリテラルのままとする。XML 1.0 で許可されるその他すべての Unicode スカラーを、UTF-8 として直接出力する。

候補内容は、完全な XLIFF 文書内で再解析し、フォーマット済みメッセージへ正確に解決されなければならない。XML 1.0 で表現できないスカラーを含むフォーマット済み値、または予期しない値の不一致は、`details.reason: "resource_write_back_failed"` を持つ `internal_error` とし、候補または部分的な書き込みを生成しない。

初期 XLIFF アダプターが抽出するのは、エンティティ参照または CDATA セクションで表現された文字データを含め、内容がプレーンな XML 文字データである主要候補だけである。character、standalone、paired、spanning、marker、annotation 要素を含む、任意のインライン子要素を持つ候補 `<source>` または `<target>` は、抽出を `resource_entry_unsupported` および `details.reason: "inline_content_unsupported"` で失敗させる。汎用リソースアダプターは、インライン要素を sentinel placeholder へ置換したり、MF2 マークアップへ変換したりしない。そのためには、識別子、対応付け、ネスト、移動、削除、およびラウンドトリップ保持に関する、形式およびプロジェクト固有の規則が必要になるためである。

それ以外ではプレーンな主要候補内にある XML コメントおよび処理命令は、インライン要素ではなく、抽出を失敗させない。メッセージバイトを提供せず `RawOnly` とする。その前後の文字データを文書順に連結して、正確なメッセージ値とする。変更された正規内容置換によって、そのホストメタデータが削除または移動されるため、エントリ全体を読み取り専用としてマークする。それでも、生データ専用ギャップの両側にある範囲を含め、通常どおりマッピングされた lint 診断を生成する。フォーマッター評価も同様にパーサー診断を保持するが、正常なフォーマット済み出力を破棄する。そのため、共有の読み取り専用契約に従い、書き込みモードは置換を構築せず、check モードは正常な差分を報告しない。CDATA 区切りだけでは、エントリを読み取り専用にしない。変更された書き込み可能な内容であれば、同等の正規文字データへ安全に置換できるためである。

具体的なプロジェクトまたは TMS ワークフローが後からインライン要素対応を必要とする場合は、別の明示的な XLIFF プロファイルとして導入する。そのプロファイルは、インライン要素を含むエントリを抽出可能にする前に、XLIFF から MF2 への表現、保護編集セマンティクス、検証規則、および可逆なラウンドトリップの振る舞いを定義しなければならない。XLIFF 内でそもそも MF2 メッセージテキストを運ぶかどうかは、プロジェクトまたは TMS の規約のままである。アダプターは、設定が明示的にオプトインしたカタログだけを処理する。

## カタログ設定

カタログの所属関係は、フォーマット、lint、およびエディターの各サーフェスが同様に使用するため、カタログのオプトインを、既存の `fmt` および `lint` セクションと並ぶ、統合プロジェクト設定の新しい `resources` セクションとして追加する。このセクションは [006-ox-mf2-phase-3a-tooling-foundation-design.md](./006-ox-mf2-phase-3a-tooling-foundation-design.md) の Phase 3A 設定契約に従う。すなわち、ルートだけでの探索、未知フィールドの厳密な検証、Rust 設定モデルを唯一の正とすること、およびコミット対象の `config.schema.json` アーティファクトへのスキーマ生成である。

共有プロジェクト設定の検証はコマンドに依存せず、`root`、`fmt`、`lint`、`resources` の固定されたセクション順で実行する。カタログ対応を追加しても、複数のセクションが無効な場合に既存のどのエラーが優先されるかを変更しないよう、新しいセクションは確立済みのフォーマッターおよびリンター検証の後に追加する。`resources` 内部の決定的な検証順は、次のように固定する。

セクションローカルな検証は、次の順序で最初の失敗を返す。

1. `resources` が存在する場合はオブジェクトでなければならない。その後、未知の `resources` フィールドを ASCII 昇順で検査する。
2. `resources.catalogs` が存在する場合は配列でなければならない。
3. カタログ定義を配列インデックス順に最後まで処理する。定義はまずオブジェクトでなければならない。その後、未知のフィールドを ASCII 昇順で検査する。
4. その定義内では `include` を最初に検査する。存在、配列形状、空でないこと、配列順のエントリ、の順とする。各エントリでは、文字列形状をリソース glob 構文の検証より先に行う。
5. `exclude` が存在する場合は次に検査する。配列形状を検査した後、配列順でエントリの文字列形状とリソース glob 構文を検査する。
6. `format` が存在する場合は最後に検査する。スカラー文字列形状の後、リリース済みの正確な `HostFormat` 値であることを検証する。

したがって、前の定義にあるパターンまたは形式の失敗は、後の定義にあるすべての失敗より優先される。検証では、意味検査より前に全定義の形状だけをまとめて検査するパスを実行しない。省略可能な `resources`、`catalogs`、`exclude`、`format` が省略されている場合、適用対象となる包含側の形状および未知フィールド検査に成功した後に限り、文書化された既定値へ到達する。

構造上のリソース設定失敗は、次の安定した `details.reason` と JSON Pointer の組み合わせを持つ `config_validation_failed` を使用する。

- 存在する `resources` 値がオブジェクトでない場合、`pointer: "/resources"` を持つ `invalid_section_shape`。
- `resources` オブジェクトまたはカタログ定義内の未知プロパティについて、正確な未知プロパティ位置に `details.field` を持つ `unknown_field`。
- 存在する `catalogs` 値が配列でない場合、`pointer: "/resources/catalogs"` を持つ `invalid_catalogs_shape`。
- 配列エントリがオブジェクトでない場合、`pointer: "/resources/catalogs/<index>"` を持つ `invalid_catalog_definition_shape`。
- `include` が存在しない、配列でない、または空である場合、`pointer: "/resources/catalogs/<index>/include"` を持つ `invalid_catalog_include_shape`。
- 存在する `exclude` 値が配列でない場合、`pointer: "/resources/catalogs/<index>/exclude"` を持つ `invalid_catalog_exclude_shape`。空の配列は有効である。
- 正確な `include` または `exclude` 配列エントリが文字列でない、またはその文字列がリソース glob 文法を満たさない場合、その位置の `invalid_catalog_glob`。
- 存在する値が文字列でない、またはリリース済み形式だけを含む `HostFormat` 集合内の正確な正規 ID でない場合、`pointer: "/resources/catalogs/<index>/format"` を持つ `invalid_catalog_format`。

`details.value` を含めるのは、`pointer` で拒否された設定値がスカラー JSON 値である場合だけとする。配列、オブジェクト、欠落値、および存在しない値をエラー details へコピーしない。したがって、文字列でない glob エントリが配列またはオブジェクトである場合は `value` を省略し、無効な文字列パターンと無効な文字列形式は、正確な文字列を保持する。割り当て時の `catalog_format_conflict` は、これらの構造形状の一つを使用せず、後述する二つの定義に関する専用 details を維持する。

両段階は、共有の `config_validation_failed` コードを意図的に再利用するが、一つの検証パスではない。前述のすべての構造上の理由は、ignore 設定または探索より前に選択し、固定された共有およびセクションローカル順序のもとで最初のエラーだけを返す。一方、`catalog_format_conflict` は、具体的に選択されたパスを対象とするため、割り当て時の設定失敗である。保持された探索／入力エラーの後へ追加し、後述するワークフロー契約のもとで対象集合全体を中止する。利用側は、二つの段階を統合するためだけに構造失敗を後へ移動したり、glob の交差を静的に解析しようとしたりしてはならない。

初期セクションの形状は次のとおりである。

```jsonc
{
  "resources": {
    "catalogs": [
      {
        // Required. Files covered by this catalog definition.
        "include": ["locales/**/*.json"],
        // Optional. Excluded files within include.
        "exclude": ["locales/**/generated.*.json"],
        // Optional. Single per-file classification override; when omitted,
        // each matched file's format defaults from its own extension.
        "format": "json"
      }
    ]
  }
}
```

正規化済み Rust 設定型は次のとおりである。

```rust
#[derive(Debug, Clone, Default, PartialEq, Eq)]
pub struct ResourcesConfig {
    pub catalogs: Vec<CatalogConfig>,
}

#[derive(Debug, Clone, PartialEq, Eq)]
pub struct CatalogConfig {
    pub include: Vec<String>,
    pub exclude: Vec<String>,
    pub format: Option<HostFormat>,
}

pub struct ResolvedResources {
    // Private compiled membership patterns and definition metadata.
}

pub struct ProjectRelativeResourcePath(Arc<str>);

pub enum CatalogResolution {
    Unmatched,
    Matched(ResolvedCatalogAssignment),
}

pub struct ResolvedCatalogAssignment {
    // Private classification plus matching and assigning definition indices.
}

pub struct CatalogAssignmentConflict {
    // Private earlier and later assigning definition indices.
}
```

リソースクレートは、利用側が再解釈できる設定データを公開するのではなく、セクションローカルな正規化と解決を所有する。統合設定ローダーが、共有検証順の `resources` ステップへ到達した後、このセクションの正確な形状、フィールド、パターン、および形式検査を、リソース層が所有するバリデーターへ委譲する。このバリデーターは、成功時だけ `ResourcesConfig` を返す。`ResourcesConfig::resolve(self) -> Result<ResolvedResources, ResourceConfigError>` は、すでに検証済みの glob 文字列をコンパイルし、決定的な証拠に必要な定義インデックスを保持する。コンパイルによって、依存ライブラリ固有の二つ目のパターン文法または検証順を導入してはならない。

`ProjectRelativeResourcePath` は、`projectRoot` から字句上相対的で、空のセグメント、`.` セグメント、`..` セグメントを含まない、空でないスラッシュ正規化済み Unicode パスからだけ構築する。そのコンストラクターはこれらの字句条件を検証するが、OS ネイティブパスを正規化、canonicalize、大文字小文字の畳み込み、または再解釈しない。CLI とエディター統合は、構築前に、引き続き `projectRoot` の確定、正確な Unicode 表現可能性、およびルート内関係を確立する責任を持つ。

利用側に依存しない検索境界は、概念上次のとおりである。

```rust
ResolvedResources::resolve_path(
    &self,
    path: &ProjectRelativeResourcePath,
) -> Result<CatalogResolution, CatalogAssignmentConflict>
```

`Unmatched` は、その定義自身の exclude を適用した後、どの定義もパスを include していないことを意味する。`Matched` は、最終的な `HostFormatClassification`、分類に使用した元の拡張子表記、一致したすべての定義インデックス、および解決された一つの形式割り当てを提供した定義の部分集合への読み取り専用アクセスを公開する。割り当てを提供する定義がない一致結果は `UnrecognizedExtension` を持つ。一意な既知割り当ては、必要に応じて `Shipped` または `KnownButUnshipped` を持つ。競合は、固定された解決順序で選択した先行および後続の割り当て定義インデックスを公開する。呼び出し側は、それらのインデックスを、すでに指定されたパスと読み込み済み設定パスと組み合わせて、サーフェス固有のエラー details を形成する。認識不能な一致は、割り当て部分集合にも競合の証拠にも決して現れない。

CLI とエディターの利用側は、プロジェクトカタログの所属関係、同一形式の重複除去、拡張子分類、および競合選択のために、このリゾルバーを呼び出さなければならない。結果の周囲にサーフェス固有の選択、レポート、またはエディターのアドホックオーバーレイの振る舞いを追加してよいが、リソース glob をコンパイルしたり、プロジェクト定義の重複アルゴリズムを独立して繰り返したりしない。

`HostFormat` は、[形式 ID と拡張子分類](#形式-id-と拡張子分類)で定義した、リリース済み形式だけを含む enum である。ルートの `resources` セクションとその `catalogs` フィールドは省略可能であり、存在しない場合は `ResourcesConfig { catalogs: [] }` へ正規化する。各カタログ定義には、空でない `include` 配列が必要である。`exclude` は省略可能であり、既定値は空配列とする。`format` は省略可能であり、既定値は拡張子に基づく分類とする。初期 `HostFormat` スキーマは `"json"` だけを受け入れ、延期済みアダプターのリリースに伴って拡張する。

複数のホスト形式で MF2 メッセージを管理するプロジェクトは、それらをすべて一度にオプトインする。分類はファイル単位であるため、混合集合に `format` フィールドは不要である。

```jsonc
{
  "resources": {
    "catalogs": [
      // Mixed-format catalogs: format omitted, classified per file extension.
      { "include": ["locales/**/*.json", "locales/**/*.yaml"] },
      // SFC catalogs: the file classifies as the SFC host format; embedded
      // block languages come from each <i18n> block's lang attribute.
      { "include": ["src/**/*.vue"] }
    ]
  }
}
```

オプトインされたファイルの形式 Tier がまだリリースされていない場合は、後述する `resource_format_unsupported` 規則に従う。そのため、プロジェクトは延期済み Tier の実装に伴って include glob を広げる。

初期のエントリレベルのフォーマットおよび lint ワークフローでは、ホスト形式の混在を引き続き許可する。後から導入するロケール対応カタログチェックワークフローでは、外側のホスト形式ではなく、`CatalogKeyDomain` を使用して、一つのスコープ内でキーを比較できるか判断する。単一ドメインのスコープ規則は、[将来のカタログチェック向けロケール関連付け](#将来のカタログチェック向けロケール関連付け)で定義する。

- `include` glob はカタログ定義ごとに必須であり、少なくとも一つの有効な文字列パターンを含まなければならない。`exclude` は省略可能である。両方のフィールドは後述する固定リソース所属文法を使用する。すべてのパターンを `projectRoot` から相対的に解決し、`exclude` は同じカタログ定義の `include` 集合からだけ一致を除去する。
- `format` は、ファイル単位の単一値分類上書きである。レジストリが各ファイルを正確に一つのホスト形式アダプターへ解決するためである。有効化する形式の一覧ではない。`"json,yaml"` のような一覧形式の文字列または配列は無効である。
- 一つのカタログ定義が、複数のホスト形式のファイルを覆ってよい。`format` が省略されている場合、一致した各ファイルの形式は自身の拡張子から既定値を得るため、混在した include 集合にも `format` フィールドは不要である。明示的な `format` 上書きは、定義が一致するすべてのファイルに適用し、拡張子が曖昧な場合のために存在する。異なる上書きを必要とする異種ファイル集合は、複数の `catalogs` 定義として表す。分類に内容スニッフィングを使用することは決してない。
- Vue SFC カタログでは、`format` は外側の SFC ホスト形式だけを識別する。埋め込みブロック言語は、設定からではなく、ファイル内部の各 `<i18n>` ブロックの `lang` 属性から取得する。
- 複数の定義に一致したファイルで、実際の形式割り当てが同じ正規 ID へ解決される場合は、重複を除去して一度だけ処理する。拡張子を認識できない一致は所属関係を提供するが、割り当ては提供しない。そのため、別の一致定義が提供する一意な形式を上書きも競合もしない。構造設定の検証およびファイル探索後、対象ファイルを読み込む前に、具体的に選択されたすべてのパスを全カタログ定義に対して解決する。異なる既知形式へ割り当てられたファイルは、後述する割り当て競合契約のもとで `config_validation_failed` エラーとなる。どの割り当て定義も黙って優先されない。重複し得るが具体的に選択されたパスへ一致しない glob パターンは有効であり、静的な交差解析を必要としない。エディターアダプターは、アドホックオーバーレイを検討する前の分類時に、開かれた具体的な各文書パスへ同じ解決を適用する。
- 明示的な `format` 値は、現在のリリースで提供されるホスト形式に対してスキーマ検証する。その Tier の実装前にオプトインされた `.yaml` カタログなど、導出された形式にリリース済みアダプターがないファイルは、黙ってスキップせず、CLI で `kind: "input"` および `code: "resource_format_unsupported"` を持つ対象ローカルな運用エラーとして報告する。エディターアダプターは、そのような文書を処理しない。
- 初期スキーマでは、キーセレクターを予約済み、nullable、またはプレースホルダーフィールドとして受け入れない。将来の名前とセマンティクスは、そのワークフローとともに設計する。将来の locale binding の形は後述するとおり確定するが、それを使用するカタログレベル lint が実装されるまでは、`locale` フィールドも同様に無効のままとする。
- カタログ単位のフォーマッターおよびリンター上書きも、予約するのではなく初期スキーマから除外する。一様なプロジェクトレベルの `fmt` および `lint` オプションを、すべてのカタログエントリへ適用する。
- 明示的な `--config <path>` は Phase 3A 契約に従う。探索された設定とマージすることなくルート設定探索を置き換え、`projectRoot` は変更しない。その OS ネイティブパスは、設定 I/O より前に Unicode として正確に表現できなければならない。相対的な `--config` パスはプロセスの作業ディレクトリから解決する。一方、明示設定ファイルが別の場所にある場合も含め、リソースの `include` および `exclude` glob は引き続き `projectRoot` からの相対位置とする。
- このセクションのフィールド名と初期セマンティクスは本書で確定し、Rust 設定モデルおよび生成される統合設定スキーマに直接表現しなければならない。

空または存在しない `resources` セクションは、カタログファイルを一つも処理しないことを意味する。単独 `.mf2` のワークフローは、`resources` セクションの影響を受けない。

### リソース所属 glob 文法

リソースの `include` および `exclude` パターンは、既存の CLI またはエディターサーフェスが具体的な文書パスを生成した後に、カタログの所属関係を決定する。独立してファイルを探索せず、CLI の隠しファイル、ディレクトリ、ignore、シンボリックリンク、または明示オペランド規則を上書きしない。

CLI のファイルモードにおける「具体的な文書パスを生成した」とは、共有探索契約の最終的な選択済み対象集合ではなく、拡張子に依存しないファイルシステム列挙段階を指す。その後、対応入力の分類は、単独 MF2 の優先順位に続けて、本書の所属および形式規則を適用する。これにより、リソース glob との照合に必要な具体パスを取得する前に、探索がカタログを分類しなければならないという循環依存を避ける。

カタログの所属関係が存在できるのは、字句上 `projectRoot` 内にあり、空でなく、`.` または `..` セグメントを持たないプロジェクト相対パスとして表現できる、正規化済み論理文書パスだけである。そのルート外の絶対入力パス、別の Windows volume 上のパス、およびそのような相対表現を持たないその他のパスは、カタログ定義に一致できない。所属判定では、物理的な canonicalization より前の、選択された論理パスを使用する。`projectRoot` 内の symlink パスは、そのリンク先が物理的にルート外にある場合でも一致してよい。一方、ルート外の論理 alias がルート内のファイルを指すというだけで、対象になることはない。

文法はプラットフォームに依存しない。

- パターンは、`projectRoot` から相対的な、空でないスラッシュ正規化済みパスとする。先頭の `/`、Windows drive prefix、UNC prefix、空のパスセグメント、および `.` または `..` と等しいセグメントは無効である。backslash をパス区切りとして扱うことは決してない。
- `*` は、一つのセグメント内で `/` 以外の Unicode スカラー値 0 個以上に一致する。
- `?` は、一つのセグメント内で `/` 以外の Unicode スカラー値ちょうど 1 個に一致する。
- `**` が特別な意味を持つのは、それだけで完全なセグメントを構成する場合に限る。その場合、完全なパスセグメント 0 個以上に一致する。`foo**bar` のように、エスケープされていない連続 `**` を埋め込むセグメントは、依存ライブラリ固有に解釈せず無効とする。
- `\` は直後の文字をエスケープし、メタ文字としての意味を取り除く。末尾に対になる文字を持たない `\` は無効である。パス区切りは一つのセグメントの一部になれないため、エスケープされた `/` も無効のままとする。
- 文字クラス、brace expansion、extglob 構文、および否定パターンには対応しない。`!` に否定の意味はなく、別途エスケープされていない限りリテラルの `!` に一致する。
- ファイルシステムの大文字小文字に関する振る舞いとは独立して、すべての OS で正確かつ大文字小文字を区別して照合する。パターンとパスのどちらにも Unicode 正規化を適用しない。
- `include` と `exclude` は同じ文法を使用する。配列順は優先順位を生成しない。所属関係は、一つの定義内で、`include` 一致の和集合から `exclude` 一致の和集合を除いたものとする。
- 構文的に有効で、現在は具体的なパスに一つも一致しない定義も有効な設定であり、警告またはエラーを生成しない。

無効なパターンは、`details.reason: "invalid_catalog_glob"` と、`/resources/catalogs/0/include/1` など正確な配列エントリを識別する `details.pointer` を持つ `config_validation_failed` とする。この文法は、リソース設定の互換性サーフェスである。後から構文を追加するには、依存ライブラリの更新を継承するのではなく、明示的な設計と fixture が必要である。

## 実装場所とパッケージ境界

共有層は、フォーマッターおよびリンター製品のクレート規約に従い、ワークスペース内部の `crates/intlify_resource` クレートに配置する。crates.io への配布物ではなく、`publish = false` とし、`crates/intlify_cli` から利用する。

- `crates/intlify_resource` は、ホスト形式レジストリ、スパンを保持するホストパーサー、抽出、エントリ識別子、オフセットマッピング、書き戻し時の再エスケープ、`resources` のセクションローカルバリデーターと解決済みカタログ設定モデル、および CLI とエディター統合が利用する一つのプロジェクトパス所属／形式リゾルバーを所有する。これには、組み込み Vue SFC 外側アダプターと内側カタログアダプターとの合成を含む。非公開のパーサーヘルパーが、抽出ワークフローを所有せずに SFC 構文スパンを提供してよい。
- このクレートは `ox_mf2_parser`、`intlify_format`、または `intlify_lint` に依存しない。利用側が抽出とメッセージレベルコア呼び出しを合成する。これにより、依存方向を非循環に保ち、どの利用側からも再利用可能な層とする。
- このクレートは同期的でスレッドセーフな処理を公開し、worker thread、thread pool、executor、async runtime、ファイルシステムの読み書き、CLI task scheduling、または reporter 集約を所有しない。その責務は、利用側からの並行呼び出しを安全にすることである。呼び出しを並行させるか、どこで実行するか、どの並行数上限を使うかは利用側が決定する。
- 具体的なホストパーサー依存ライブラリは実装上の選択であり、生の値スパン、バイト単位で正確なエスケープ解除、および値同一の再エスケープという契約で制約し、すべてを fixture で固定する。
- この層で予定する配布物は、このクレートだけである。現時点ではリソース N-API または WASM バインディングパッケージを予定しない。パッケージ配布の再検討は、Rust 以外の具体的な利用側を必要とする延期済みの後続作業である。パーサー、フォーマッター、およびリンターのバインディングパッケージは、リソース API を取り込まない。

## CLI リソースワークフロー

### 並列実行の所有権

`crates/intlify_cli` は、`intlify fmt` と `intlify lint` の並列実行を所有する。セットアップ、探索、ignore フィルタリング、対応入力の分類、および物理識別子の検査が完了した後、各ファイルモードコマンドは、上限付きでコマンド単位の worker-thread pool を最大一つ作成する。実行可能な物理グループが残っていない場合、pool は作成しない。それ以外の場合、デフォルト幅は、実行可能な物理グループ数と OS が提供する利用可能な並列数のうち小さい方とし、最小値は一とする。利用可能な並列数の取得、worker runtime の初期化、dispatch、実行、または join に失敗した場合、共有 CLI 境界で定義する command-fatal な `cli_worker_runtime_failed` internal error とする。リソースワークフローは直列実行へ fallback せず、部分的な対象結果も保持しない。そのコマンドの parser、formatter、linter、および resource 呼び出しは、すべてこの一つのスケジューリング境界で実行し、合成対象のクレートはいずれもネストした pool を作成しない。stdin モードは引き続き呼び出し元スレッド上の一つのワークフローとする。`--threads` などの公開並行数制御は延期するが、自動的なファイルレベル並列実行は延期しない。

scheduler の作業単位は、一つの物理ファイルグループである。異なるグループは並行実行してよい。一方、一つのグループ内の論理 alias は、読み取り、抽出、メッセージコア処理、候補検証、結果構築、および省略可能な書き込みからなる完全な手順を、正規化済み論理パス順で直列実行する。scheduler は backpressure を適用し、選択済みファイルすべてのソースと解析アーティファクトを先に保持するのではなく、パスおよび設定の作業記述を渡す。

worker は stdout または stderr を描画せず、共有結果配列へ直接追加しない。各 worker は、安定した論理識別子を持つ構造化結果を返す。調整スレッドが後述の契約に従って結果を並べ、summary と終了状態を計算した後に、text または JSON を描画する。したがって、worker の完了順、OS のスレッドスケジューリング、および実行時に選択された幅によって、公開結果順、選択エラー、出力バイト、または終了動作が変わってはならない。フォーマッターの書き込みを並行実行できるのは、物理識別子が異なる対象間だけである。書き込み失敗による fail-stop の影響は、そのグループ内の後続 alias だけに留める。

### 入力選択

CLI オペランドまたは glob で選択され、かつ解決済み `resources` 設定によってオプトインされた場合に限り、ファイルはリソース処理へ入る。

- 対応済み CLI 入力集合を、直接の `.mf2` ファイルからオプトイン済みカタログファイルへ拡張する。拡張子だけではファイルをカタログにしない。所属関係には設定が必要である。これは、[008-ox-mf2-phase-3c-linter-design.md](./008-ox-mf2-phase-3c-linter-design.md#file-discovery-and-shared-cli-contract) の共有探索契約が約束する対応入力の所属関係を、その契約の形状を変えずに拡張する。
- glob 展開では、分類時に対応済み入力集合、すなわち `.mf2` ファイルとオプトイン済みカタログファイルとの積集合を取る。glob に一致する、オプトインされていない JSON、YAML、Vue、XLIFF、またはその他のホスト形式ファイルは、列挙済みファイルシステム候補として残るが、選択済み対象になる前にスキップする。
- `.mf2` ファイルでもオプトイン済みカタログでもない明示的なファイルオペランドは、共有探索契約の既存の未対応入力処理に従う。
- その `unsupported_input_file` について、`details.supportedExtensions` は引き続き `[".mf2"]` とする。このフィールドには、別の `resources` 定義のもとでカタログになり得る拡張子ではなく、設定なしで直接分類できるものだけを列挙する。オプトイン済みだがリリース済みアダプターへ解決できない対象は、代わりに `resource_format_unsupported` と、その形式向け details を使用する。
- `projectRoot` 外にある明示的な非 `.mf2` ファイルは、カタログへオプトインできないため、同じ未対応入力処理に従う。ルート外のディレクトリおよび glob 探索は、単独 `.mf2` ファイルを引き続き選択してよいが、そこにある非 `.mf2` エントリはカタログ対象にならない。このような単独ファイルまたは入力エラーのパスに対する絶対パスのレポートは、Phase 3A の機械可読パス規則を引き続き使用する。

Phase 3A 契約のもとで、プロジェクトルート探索と正確な Unicode 表現可能性の検査は、設定探索より前に実行する。そのため、Unicode でないプロジェクトルートは共有の `projectRoot: null` エラーエンベロープを生成し、リソース設定または所属判定には到達しない。表現可能なルートを確立した後、明示的な `--config` パスを設定 I/O より前に正確な表現可能性について検査する。続いて、ignore 設定または探索より前に、構造設定の読み込みと検証を実行する。Unicode でない設定パスまたは構造設定エラーは、リソース所属処理より前に中止する。設定成功後、ignore ソースを読み込む、または探索を開始する前に、すべての OS ネイティブ `--ignore-path` 値を解決し、CLI 順に正確な表現可能性を検査する。最初の表現可能性失敗も同様に、リソース所属前に空の結果で中止する。それ以外の場合、ignore ソースを読み込んで検証し、拡張子に依存しないファイルシステム列挙によって具体的な候補パス集合を生成し、すべてのトップレベル探索／入力エラーを既存の決定的な順序で保持する。対応入力の分類では、表現可能な各候補を、予約された単独 MF2 分類、プロジェクトリソース所属および形式解決、非所属処理の順に処理する。非所属の明示ファイルは `unsupported_input_file` を生成し、ディレクトリまたは glob 経由で到達した非所属候補はスキップする。CLI が対象を読み込む、または処理する前に、オプトインされた具体的な全候補についてカタログ割り当てを解決する。いずれかのパスが競合するホスト形式へ解決される場合、コマンドは、すでに発見または分類された入力エラーの後に、トップレベル `config_validation_failed` を一つ追加する。空の `results` 配列と、コマンド固有対象カウンターなしで対象集合全体を中止し、ignore 照合、物理グループ化、読み込み、またはフォーマットを行わない。競合選択は決定的に行う。まず、競合するスラッシュ正規化済み対象パスの昇順で最初のパスを選び、次にそのパスについて競合する、インデックスが最小の後続カタログ定義を選ぶ。先行する競合定義は、同じパスへ異なる形式を割り当てる、インデックスが最小の定義とする。

共有の `input_path_unrepresentable` ガードは、リソース所属、カタログ形式分類、および ignore 照合より前の探索中にも実行する。表現不能なオペランド、仮想 stdin パス、または到達したファイルシステムエントリは、Unicode スカラーのリソース glob 文法へ到達せず、照合またはレポート可能なパスへ損失を伴って変換されることもない。そのトップレベルエラーは、その他の探索／入力エラーとともに保持する。表現可能な候補は、後のカタログ割り当て競合が対象集合全体を中止しない限り処理を続ける。先行する設定パスと ignore-path のバリアントはセットアップ失敗であるため、どちらも探索エラーと共存できない。

対応入力の所属およびカタログ形式の分類後、ファイルモード対象は、呼び出したコマンドの既存の順序付き ignore 契約を通過する。順序は、ルート `.gitignore`、CLI 引数順の `--ignore-path` ファイル、`fmt.ignorePatterns` または `lint.ignorePatterns` とする。ignore 規則は、探索済み対象だけでなく、明示的なファイルオペランドにも適用する。ignore されたカタログは抽出せず、ignore フィルタリング後に対象集合が空の場合は、既存の対象ゼロ成功の振る舞いを維持する。フォーマッターとリンターの ignore リストは意図的に独立している。そのため、同じオプトイン済みカタログを `fmt` ではスキップし、`lint` では処理する、またはその逆が可能である。

ignore フィルタリング後、ファイルモード対象は、共有 symlink 契約から得た異なる論理パス識別子を保持したまま、物理ファイル識別子でグループ化する。物理識別子はスケジューリングおよび安全性のキーに限る。結果、診断、設定照合、またはキャッシュ名前空間において、正規化済み論理パスを置き換えることは決してない。一つのグループ内の対象を正規化済み論理パス順に並べ、読み込み、抽出、コア処理、候補検証、および省略可能な書き込みからなる完全な処理列を直列に実行する。そのため、後続の各 alias は、後述する書き込み失敗時の停止規則を除き、同じグループの先行 alias がすべて完了した後にソースを開いて読み込む。`matchedFiles`、`results[]`、および人間向けパス出力では、各論理パスを引き続き別の対象として扱う。たとえば、最初の alias をフォーマットした後、後続 alias は更新済みソースを変更なしとして観測し得る。

ファイルシステムへの書き込み試行より前に発生した対象ローカルな運用失敗は、その物理グループを停止しない。これには、一つの論理パス経由の読み込み失敗、抽出失敗、フォーマッターまたはリンターの運用失敗、候補検証失敗、診断マッピング失敗、および正常結果の構築失敗を含む。次の alias は自身の論理パス経由で開かれ、成功する可能性がある。そのため、check モードと lint は、そのような失敗後も常に後続 alias へ進む。書き込みが成功した場合も通常どおり続行し、次の alias が更新済みソースを再度読み込む。

フォーマッター書き込みモードの `output_write_failed` 結果だけを、fail-stop とする。Phase 3B の直接書き込みは atomic ではなく、rollback を保証しないため、失敗後の物理ファイルは、変更されていない、部分的に書き込まれた、またはその他の不確定な状態であり得る。そのため、先行カタログ対象は `status: "error"`、`changed: false`、空の `entries` 配列、および `results[].errors` 内の書き込みエラーだけを報告する。完了済みの書き込み前エントリ結果と診断は破棄する。元ホスト座標も候補ホスト座標も、不確定なファイルを正しく表さないためである。`matchedFiles` と `errorCount` には加算するが、formatted、different、unchanged、または diagnostic 件数には加算しない。その後 CLI は、同じ物理グループ内の後続 alias に対して stat、open、read、parse、format、write のいずれも実行せず、残りの各 alias へ対象ローカルな `alias_processing_blocked` 結果を与える。関係のない物理グループは続行する。共有書き込み境界はファイル状態が変更されていないことを保証しないため、特定の OS エラーが最初の出力バイトより前に発生したように見える場合でも、この規則を保守的に適用する。

物理識別子は対象ファイルに従い、symbolic-link と hard-link の両方の alias をグループ化する。POSIX 実装は device ID と inode の組を使用し、Windows 実装は volume ID と file ID の組を使用する。その他の対応ネイティブプラットフォームでは、ファイルモードのリソース処理を有効にする前に、同等の安定したファイル識別子を提供しなければならない。canonical または正規化済みのパステキストを、識別子のフォールバックには決して使用しない。hard link を確実に検出できないためである。識別子は、ignore フィルタリング後かつ対象処理前に一度取得する。この CLI プロセスが alias を並行スケジュールすることは防ぐが、外部変更の検出は追加しない。取得後の rename、symlink の参照先変更、またはファイル置換は、単独 `.mf2` ファイルと共有する既存の Phase 3B 直接書き込みおよび stale 検査なしの契約に従う。

選択済み対象の物理識別子を検査できない場合、`details.reason: "metadata_failed"` を持つ対象ローカルな `input_read_failed` とする。`details.ioKind` は必須であり、正確に `"not_found"`、`"permission_denied"`、`"not_file"`、`"not_directory"`、`"unknown"` のいずれかへ正規化する。依存ライブラリの debug 名または新しく追加された Rust `ErrorKind` 名を、暗黙に JSON 値へしない。基礎となる I/O エラーが生の OS コードを公開する場合に限り、`details.rawOsError` を含め、符号付き JSON 整数として変更せずシリアライズする。そのため、壊れた symbolic link は、推測した識別子を受け取るのではなく、具体的な metadata 失敗を報告する。その論理対象はグループ化も処理もしない。識別子を正常に取得したその他の対象は、通常の混在入力契約のもとで続行する。

正規化では、対応する移植可能な I/O 条件に対して `not_found` と `permission_denied` を選択し、検査対象が処理可能なファイルを表さなくなった場合に `not_file`、対象へ到達するために必要なパス構成要素がディレクトリでない場合に `not_directory` を選択する。明示的に設計されたマッピングを持たない将来のプラットフォームまたは Rust エラー種別を含む、その他すべての条件は、利用可能な場合は `rawOsError` を保持しながら `unknown` とする。人間向けメッセージには、安定していない OS の説明文を追加してよいが、そのような説明を `details` へコピーしない。

初期リソースワークフローとともに、フォーマッターおよびリンターが共有する `input_read_failed` の理由語彙へ `metadata_failed` を追加する。`unsupported_encoding` をこの語彙へ追加するのは、それを生成する Tier 3 XLIFF 事前検査の実装時点とする。どちらの条件も、別の運用エラーコードを導入せず、対象ローカルな結果配置を変更しない。

一つの物理グループ内で処理可能なすべての対象は、同じワークフロー分類を持たなければならない。安定した分類文字列は `"standalone:mf2"` と、`"catalog:json"` などの `"catalog:<registry-id>"` とする。一つの物理ファイルが異なる分類で選択された場合、そのグループ内の対象を一つも読み込みまたは処理しない。CLI は、競合する各物理グループについてトップレベル `input_target_conflict` を一つ生成し、競合しないグループを続行し、終了コード `2` で終了する。これらのエラーは、保持されたすべての探索／入力エラーの後に置き、各グループの最初の正規化済み論理パス順に並べる。各エラーの path および classification 配列は、そのグループ内で識別子を正常に取得し、ignore されていないすべての論理対象を、正規化済み論理パス順で含む。壊れた symlink と、その他の選択済みファイルの検査失敗は、alias 競合と推測せず、対象ローカルな入力読み込みまたは metadata I/O 失敗のままとする。

### Stdin の選択

明示的な stdin モードは、既存の `--stdin-filepath <path>` オプションを通じてカタログに対応する。仮想パスは存在する必要がない。相対値はプロセスの作業ディレクトリから解決し、表現可能な場合は、ファイルモードの設定照合およびレポートと同じパス識別子を使用して、`projectRoot` から相対的にスラッシュ正規化する。

解決済み仮想パスがカタログ対象になれるのは、その正規化済み識別子が `projectRoot` 内にある場合だけである。ルート外の仮想パスも、`.mf2` 分類を通じて単独ファイル用ワークフローを選択してよい。その他のルート外パスは `resources.catalogs` に一致できず、`unsupported_input_file` となる。CLI は、絶対パスをプロジェクト相対の接尾辞として再解釈せず、`..` による照合を許可しない。

設定読み込み後、CLI は仮想パスを次の順序で分類する。

1. `.mf2` 拡張子または MF2 直接ファイル分類は、既存の単独 stdin ワークフローを選択する。
2. それ以外の場合、パスは解決済み `resources.catalogs` の include に一致し、その定義の exclude パターンを通過し、通常の上書き、拡張子、および重複規則のもとで、正確に一つのホスト形式へ解決されなければならない。
3. カタログへオプトインされていない非 `.mf2` パスは `unsupported_input_file` とする。カタログ形式らしい拡張子だけで stdin がオプトインされることは決してない。
4. 対応入力の分類は ignore 照合より前に行い、ignore された未対応仮想パスも入力エラーのままという既存規則を維持する。分類後、通常のフォーマッターまたはリンターの ignore ソースが対象をスキップしてよい。

ignore されていないカタログ stdin は、UTF-8 デコード前に上限付きのホストバイト事前検査を行い、仮想ファイルシステムパスを検査または書き込むことは決してない。その後、受け入れた入力を完全な UTF-8 ホスト文書として読み込む。単独 `.mf2` の BOM または末尾改行フレーミングは適用しない。抽出は受け入れた正確なホストバイト列を観測し、フォーマッター出力は検証済みの完全なホスト文書となる。通常のフォーマッターモードではそのホストソースに stdout を使用する。一方、`--reporter json` は既存の stdin 契約と同様に、JSON エンベロープ用に stdout を予約する。lint JSON は、仮想パスを `results[0].path` とする、ネストされたカタログ結果形状を使用する。

ignore された stdin も、既存の CLI 契約と同様に UTF-8 検証を受ける。有効な UTF-8 の場合、通常のフォーマッターモードは元のホストバイト列を正確に stdout へ通過させる。フォーマッターの check と lint は人間向け出力を生成せず、JSON は既存の対象ゼロ stdin サマリーを使用し、抽出を実行しない。カタログ stdin も、`--stdin-filepath` とファイル、ディレクトリ、または glob オペランドを併用できないという既存の禁止事項を維持する。

ignore されていない通常フォーマッター stdin では、一つ以上のメッセージエントリがパーサー診断を持つ場合でも、ホスト抽出に成功すれば、完全なホスト文書を必ず一つ stdout へ生成する。診断のない書き込み可能なエントリは、検証済み置換を提供する。パーサー診断があるエントリと読み取り専用エントリは、元の生値を保持する。どのエントリも変更されない場合、元のホストソースをバイト単位で出力する。エントリ診断は stderr へ描画し、終了コード `1` を維持する。安全に処理されたホストソースを出力しても、その結果は成功へ変わらない。

これは、単独 stdin の振る舞いを意図的にカタログ向けへ特殊化したものである。無効なホスト構文または未対応候補エントリを含むホスト抽出失敗、フォーマッターの運用エラー、再エスケープ失敗、または最終候補検証失敗では、ホストソースを出力せず、終了コード `2` で終了する。`--reporter json` では、stdout に JSON エンベロープだけを含め、ホストソースは決して含めない。stdin の check モードでは、ホストソースを出力しない。ほかのエントリに診断がある場合でも、有効なエントリに差分があれば、人間向けモードで仮想パスを出力する。診断は stderr へ描画し、統合結果は終了コード `1` となる。差分がなく診断だけがある場合、パスを出力しない。

### ホスト解析失敗

カタログファイルをホスト形式として解析できない場合、CLI は共有運用エラー形状を使用し、`results[].errors` に対象ローカルな運用エラーを報告する。`kind: "input"`、`code: "resource_parse_failed"`、および後述する必須のホスト位置 details を使用する。このファイルは診断を提供せず、fmt が書き込むこともない。その他の選択済みファイルは引き続き処理する。Phase 3A 出力契約に従い、運用エラーによって `summary.status: "error"` および終了コード `2` となる。

### リソースエラーモデル

`crates/intlify_resource` は型付きの `ResourceError` を返す。そのバリアントは、CLI サーフェスより詳細なままでよい。CLI の利用側は、共有 Phase 3A 運用エラー型への一元的な変換を一度行う。リソース API は CLI JSON 値を構築せず、リソース失敗がパーサー、意味解析、または lint 診断になることは決してない。

トップレベルのリソースコード、共有コードの再利用、およびエディター診断への投影は、[appendix-ox-mf2-error-code.md](./appendix-ox-mf2-error-code.md) に索引化されている。型付きバリアント、details スキーマ、優先順位、および回復の振る舞いについては、引き続きこのセクションを正とする。

初期実装は、固定された包含的なリソース上限を適用する。これらは CLI とエディターの利用側で共有する、コンパイル時の製品定数であり、設定フィールドまたは CLI オプションではない。

| リソース                                                 |                     初期上限 |
| -------------------------------------------------------- | ---------------------------: |
| 生のホストソース                                         | 64 MiB（`67,108,864` bytes） |
| ホストのネスト深さ                                       |                        `256` |
| 抽出済みエントリ                                         |                    `100,000` |
| 一エントリの UTF-8 メッセージテキスト                    |   1 MiB（`1,048,576` bytes） |
| カタログ全体の UTF-8 メッセージテキスト合計              | 64 MiB（`67,108,864` bytes） |
| intern された異なる識別子文字列ペイロード                | 64 MiB（`67,108,864` bytes） |
| メッセージから生データへのオフセットマップセグメント合計 |                  `1,000,000` |

ignore されていない CLI カタログ対象では、UTF-8 デコードまたはホスト解析より前に、`preflight_host_bytes` を通じてホストのバイト長を検査する。ファイルおよび stdin reader は、最初の `67,108,865` バイトまでに制限する。包含的な上限を超える最初のバイトへ到達した時点で読み込みを停止し、完全な長さを調べるためだけに残りを読み込むことなく、`actual: 67108865` を報告する。そのバイトを観測する前の I/O 失敗は、共有の `input_read_failed` の振る舞いを維持する。バイト数を受け入れた後の無効な UTF-8 は、`details.reason: "invalid_utf8"` と、共有の必須 `details.validUpTo`（有効な最長プレフィックスのバイト数）を持つ `input_read_failed` のままとする。そのため、サイズ超過かつ無効な UTF-8 の入力は、最初の超過バイトを観測した時点で `resource_limit_exceeded` を報告し、UTF-8 デコードを試行しない。ignore フィルタリングは対象読み込みより前に実行されるため、ignore された対象は既存の ignore 入力契約に従い、この事前検査のためだけにリソース処理を受けることはない。

エディター文書は、リソースアダプターが受け取る時点ですでに Unicode テキストである。エディターはホスト解析前に、文書の UTF-8 バイト長を指定して同じヘルパーを呼び出し、`extract` は必須の多層防御用再検査を行う。プロトコルより前のバイトデコードはエディター transport の責任であり、このリソース契約の外側にある。候補予測では、割り当て前に、同じ `host_bytes` 上限と `ResourcePhase::ValidateWriteBack` を正確な `u128` 計算へ直接適用する。受け入れた予測値は最大 64 MiB であるため、すべての対応対象で `usize` へ損失なく変換できる。その後、候補の再抽出は解析前に `preflight_host_bytes` を再度呼び出して多層防御を行う。

完全な一つの `ExtractedCatalog` は、累積抽出予算を一つ持つ。そのため、合成された Vue アーティファクトでは、`entries`、`total_message_bytes`、`identity_bytes`、`offset_map_segments` を、対応済みのすべてのインラインブロックにわたり絶対生ソース順で数え、ブロックまたは内側形式ごとにリセットすることは決してない。`message_bytes` はエントリ単位の検査のままとする。識別子インターナーは同じ外側ビルダーに属する。そのため、等しい文字列はブロック間で backing を共有してよいが、異なるペイロードの計上はアーティファクト全体で行う。オフセットマップの計上では、内側スパンを絶対外側座標へ持ち上げた後の、最終的な正規セグメントを課金する。増分ビルダーは、結合可能か判断する際に最後の正規セグメントだけを参照し、結合できない場合に限って新しいセグメントを課金する。そのため、アダプターは、上限のない非正規な中間マップを最初に構築するのではなく、最終的な新規セグメントを割り当てまたは公開する前に上限を検査しなければならない。累積上限を超えた最初のエントリまたはセグメントが、通常の絶対元ホスト site とグローバルな `actual` 値を提供する。

`host_bytes` が計測するのは、公開 `extract` へ渡された完全なトップレベルソースだけである。インラインソース領域は、すでに受け入れたホストのスライスであり、別のホストバイト予算または事前検査を開始しない。ネスト深さだけを、意図的にパーサーローカルな上限とする。これは、一つのパーサーインスタンス内で同時に開いている構造コンテナーの最大数であり、JSON および JSON 系の配列またはオブジェクト、YAML の sequence または mapping、XML または SFC 要素を含む。ルートコンテナーまたは要素の深さは `1`、ルートスカラーの深さは `0` とする。Vue の外側パーサーと、呼び出される各インライン内側パーサーは、自身の深さカウンターをゼロから開始する。そのため、外側要素の深さを内側 JSON または YAML の深さへ加算しない。将来さらに深いアダプター合成を行う場合も、一つの外側累積アーティファクト予算を維持しながら、各パーサー層で同じ規則を独立して適用する。

組み込みアダプターの各ホストパーサーフロントエンドは、構造コンテナーの開始を確認した時点で、そのコンテナーを割り当てる、または再帰的に下降する前に深さ上限を適用しなければならない。正確に設定可能な上限を持つパーサー依存ライブラリは `256` を使用する。その機能を持たない場合は、同じ構造をホスト構文を認識しながら追跡する、リソース層が所有する反復的 token または event フロントエンドでラップしなければならない。公開リソース上限より低い、依存ライブラリ内部の再帰上限は許可しない。依存ライブラリ固有の再帰エラーをホスト構文エラーとして公開することは決してない。

構文と深さの失敗は、生のソース順に従う。フロントエンドが深さ `257` の開始コンテナーを確認するより前にホスト構文エラーへ到達した場合、結果は `resource_parse_failed` となる。先にその開始コンテナーを確認した場合、結果は `resource: "nesting_depth"`、`actual: 257`、および元ソースか候補かに応じた phase を持つ `resource_limit_exceeded` となり、その後の構文は調べない。アダプターコードは、上限内の入力に対して panic または stack overflow を起こしてはならない。この要件に違反する依存ライブラリの panic、stack overflow、または再帰失敗は、`resource_parse_failed` ではなく、アダプター実装の欠陥である。catch できる失敗は、`details.reason: "resource_adapter_invariant_failed"` と進行中の抽出 phase を持つ `internal_error` へマッピングする。候補の再抽出は、同一のフロントエンド、上限、および優先順位を使用する。

完全なホスト構文解析を、抽出段階の失敗選択より前の境界とする。前述したソース順の構文対深さ規則を除き、どのホスト構文失敗も抽出を防ぎ、`resource_parse_failed` を返す。アダプターは、部分的な構文木から文書機能、エントリ表現可能性、または解析後リソース失敗を選択しない。

ホスト解析成功後、抽出は、対応済みおよび未対応の文書構造とメッセージ候補に対し、生ソース順の event 走査を一度実行する。最初に失敗した event で走査を停止し、対象エラーを決定する。そのため、異なるオフセットの失敗は、公開コードにかかわらず、主要な生ソースオフセットが小さい方を選択する。同じ主要オフセットで複数の検査が失敗した場合、固定された優先順位は次のとおりである。

1. 文書レベルの未対応構造。`resource_document_unsupported` を生成する。
2. 候補パスまたはメッセージの表現可能性。`resource_entry_unsupported` を生成する。
3. リソース上限。エントリ数、一メッセージのバイト長、実行中のメッセージ合計バイト長、実行中の異なる識別子合計バイト長、実行中のオフセットマップセグメント合計数、の順に検査する。

この順序により、より具体的な表現不能失敗をエントリ上限で隠すためだけに、表現不能な候補を正常に構築可能なエントリとして数えない。また、失敗の収集、またはリソース超過後の続行も避ける。候補の再抽出は、その結果を候補検証エラー契約へマッピングする前に、同じ event および同位置優先順を使用する。

`identity_bytes` カウンターは、完全な一抽出アーティファクトにローカルであり、ゼロから開始する。それ以外では表現可能な各候補エントリについて、生ソース順で、ビルダーはシリアライズ済み `StructuralPathKey`、`CatalogKey`、存在する `displayKey` の順に考慮する。そのアーティファクト内ですでに intern された文字列はゼロを加算する。新しい正確なバイト文字列は、受け入れる前に完全な UTF-8 長を加算する。空文字列も intern できるが、加算値はゼロである。実行中の合計が `67,108,864` を超える最初の加算は、`resource: "identity_bytes"`、検査した合計と等しい `actual`、および現在のエントリを元ホスト site として持つ `resource_limit_exceeded` を返す。`CatalogKeyDomain`、出現番号、ハンドル、スパン、および固定されたエントリ単位ストレージは、このバイトカウンターへ含めない。それらの個数は、エントリ上限によって独立して制限される。

最初に観測したリソース超過は `resource_limit_exceeded` を返す。部分的なアーティファクト、フォーマッター結果、リンター結果、または書き込みは生成しない。置換を生成できるのは書き込み可能な抽出済みエントリだけであり、固定エントリ上限を超えられないため、置換数には別の公開上限を必要としない。これらの公開上限を変更するには、文書化されていない依存ライブラリまたはプラットフォーム固有の既定値ではなく、明示的な互換性判断と fixture 更新が必要である。

同じ上限を、書き込みモードと check モードのすべての書き戻し候補へ適用する。その適用は `build_and_validate_write_back` が所有する。生エントリ順の計測パス中に、実効候補メッセージを `message_bytes` に照らして検査し、指定されたフォーマット済み値と、省略された、診断を持つ、読み取り専用、または非選択エントリの元の値から、実行中の `total_message_bytes` を計算する。一つの変更済みエントリの再エスケープを計画するのは、そのエントリの二つのメッセージ検査が成功した後だけであり、出力を実体化せずに正確な生出力長を返す。

パス終了後、置換テキストまたは候補の割り当てより前に、候補ホストのバイト長を `original_host_bytes - sum(replaced_raw_span_bytes) + sum(measured_raw_text_bytes)` として一度計算する。すべての項と中間合計は `u128` を使用する。元スパンは互いに素で受け入れ済みホスト内にあり、参加できる計測長は `u64` 型で最大 `100,000` 個であるため、この算術は `u128` を overflow できない。完全な予測値を、`host_bytes` 超過の正確な `actual` とする。実装は、飽和値、`limit + 1` sentinel、プラットフォーム `usize` overflow 分岐、または生順の部分予測を使用しない。一元化されたエラー変換は、浮動小数点表現を経由せず、`u128` 値を負でない十進 JSON 整数として直接シリアライズする。

ホスト上限内にある完全な候補だけが、生テキストの実体化へ進む。`O` を元のホストバイト数、`R` を置換される互いに素な元スパンの合計、`A` を実体化された生置換の合計、`C = O - R + A` を受け入れた候補サイズとすると、`A = C - O + R <= C + O` となる。受け入れられる両ホストは最大 64 MiB であるため、保持される置換ペイロードは最大 128 MiB となる。この導出上限、計画メタデータに対するエントリ数上限、および期待メッセージのコピー除去により、ユーザーから見える一時メモリ上限を別に設けず、書き戻し作業ストレージを制限する。実体化は各計画済みバイト長を検証し、候補構築は受け入れ済みの `C` バイトだけを割り当て、その後、非公開の適用と完全な検証を通常どおり進める。

候補の再抽出は、同じ一ビルダー、一予算の合成を再構築し、新しいアーティファクトローカルインターナーを作成し、`host_bytes`、パーサーローカルな `nesting_depth`、エントリ単位の `message_bytes`、およびアーティファクト全体の `entries`、`total_message_bytes`、`identity_bytes`、`offset_map_segments` を再適用する。候補識別子の等価性検査によって、これらの上限が不要になるとは仮定しない。候補の超過は `details.phase: "validate_write_back"` を持つ `resource_limit_exceeded` を返し、`resource_write_back_failed` へ変換されることは決してない。初期抽出は `details.phase: "extract"` を使用する。check モードが、それ以外では有効な候補を破棄する場合にも同様に適用する。

ユーザーが対処可能なリソース失敗だけが、専用の公開コードを受け取る。

| コード | kind | 終了 | 発生条件 |
| --- | --- | --- | --- |
| `resource_format_unsupported` | `input` | `2` | オプトイン済み対象が、アダプター未リリースの既知または拡張子由来ホスト形式へ解決されるか、対応済みアダプターへ解決できない。 |
| `resource_parse_failed` | `input` | `2` | 選択されたホストアダプターが、完全なホスト文書を無効なホスト構文として拒否する。 |
| `resource_entry_unsupported` | `input` | `2` | 候補ホスト文字列またはメッセージコンテナーを、選択されたアダプターが一つの MF2 メッセージエントリとして表現できない。 |
| `resource_document_unsupported` | `input` | `2` | 有効なホスト文書が、選択されたアダプターで意図的に対応しない文書レベル構造を使用している。 |
| `resource_limit_exceeded` | `input` | `2` | ホストバイト、抽出済みメッセージバイト、エントリ数、またはその他の文書化されたリソース表現が、固定されたアドレス可能上限を超える。 |

リソースワークフローは、別名を追加する代わりに、既存の共有コードを再利用する。

| コード | kind | 終了 | リソースワークフローでの用途 |
| --- | --- | --- | --- |
| `config_validation_failed` | `config` | `2` | 無効な `resources` 形状、未知フィールド、無効な glob または format 値、および競合するカタログ割り当て。 |
| `input_path_unrepresentable` | `input` | `2` | プロジェクトルート、明示設定パス、反復 ignore パス、選択済みオペランド、仮想 stdin パス、またはファイルシステムエントリが、リソース所属前のセットアップまたは探索境界で必要となる正確な Unicode パス識別子を形成できない。 |
| `input_read_failed` | `io` | `2` | XLIFF UTF-16 BOM 事前検査を含む、アダプター抽出前の入力ファイル読み込み失敗または UTF-8 でないホストバイト列。 |
| `output_write_failed` | `io` | `2` | 検証済みのフォーマット済みホストソースを書き込めない。 |
| `internal_error` | `internal` | `2` | 有効であるはずのアダプターアーティファクト、コアスパン、置換、または書き戻し結果が実装上の不変条件に違反する。 |

CLI スケジューラーは、`ResourceError` ではない、リソースワークフロー固有の対象ローカルコードを一つ追加する。

| コード | kind | 終了 | 発生条件 |
| --- | --- | --- | --- |
| `alias_processing_blocked` | `io` | `2` | 同じ物理グループの先行論理 alias が `output_write_failed` を返したため、共有ファイル状態が不確定であり、この alias を処理しない。 |

ワークフロー分類が競合する物理 alias には、`ResourceError` ではなく CLI 入力選択エラーを使用する。

```json
{
  "kind": "input",
  "code": "input_target_conflict",
  "message": "The same physical input was selected with conflicting classifications.",
  "details": {
    "paths": ["locales/en.json", "messages/en.mf2"],
    "classifications": ["catalog:json", "standalone:mf2"]
  }
}
```

`paths` と位置で対応する `classifications` 配列は必須であり、競合するグループ内のすべての論理 alias を含み、少なくとも二つのエントリを持ち、正規化済み論理パスの昇順を使用する。各パスは共有の機械可読パス規則に従う。表現可能な場合はプロジェクト相対、それ以外では絶対かつスラッシュ正規化済みとする。物理競合を所有する論理 alias は一つに定まらないため、エラーはトップレベルに置く。競合するグループは `results[]` 項目または matched-file 件数を提供しない。その他の物理グループは通常の混在入力契約のもとで続行する。物理識別子自体、device number、inode 値、volume ID、および file ID は内部スケジューリングデータであり、決してシリアライズしない。

`alias_processing_blocked` は次の details を使用する。

```json
{
  "reason": "prior_alias_write_failed",
  "predecessorPath": "locales/en.json"
}
```

運用エラーのトップレベル `path` は、処理を阻止された alias とする。`reason` は必須であり、初期値は `"prior_alias_write_failed"` だけである。`predecessorPath` は必須であり、グループを停止させる `output_write_failed` を返した正規化済み論理パスとする。共有のプロジェクト相対または絶対機械可読パス規則に従い、物理識別子ではない。先行対象自身の結果が基礎となる書き込みエラー details を保持し、処理を阻止された結果へコピーしない。処理を阻止された各フォーマッター結果は、`status: "error"`、`changed: false`、該当する場合は空のカタログ `entries` 配列、および `results[].errors` 内のこのエラー一つを持つ。選択済み対象のままであるため、`matchedFiles` と `errorCount` には加算するが、formatted、different、unchanged、または diagnostic 件数には加算しない。人間向け出力は、device または inode データを公開せず、両方の論理パスを識別する。

具体的なカタログ割り当て競合では、読み込まれた設定ファイルを `path` とする、トップレベル `config_validation_failed` を使用する。安定した details は次のとおりである。

```json
{
  "reason": "catalog_format_conflict",
  "pointer": "/resources/catalogs/2",
  "conflictingPointer": "/resources/catalogs/0",
  "targetPath": "locales/en.json"
}
```

`pointer` は後続の競合カタログ定義を、`conflictingPointer` は異なる既知形式を割り当てた最初の定義を識別する。どちらの割り当ても `/format` フィールドが存在せず、拡張子から導出される可能性があるため、両ポインターは定義全体を指す。拡張子分類を認識できない一致定義は形式割り当てを提供しないため、どちらのポインターにも選ばれない。`targetPath` は、具体的なプロジェクト相対スラッシュ正規化済みパスである。エラーは、前述の決定的なパスおよび定義インデックス順で選択し、トップレベル配列内ですでに保持された探索／入力エラーの後に置く。完全で曖昧さのない対象集合が確立されていないため、その JSON エンベロープは `summary.status: "error"`、`results: []`、およびコマンド固有の対象カウンターなしとなる。

したがって、トップレベルのリソース選択エラー順は、共有 Phase 3B パイプライン順を次のように拡張する。探索エラーおよび対応入力の分類エラーは、フォーマッター設計にあるセクションローカル順を維持し、その後に、存在する場合は選択されたカタログ割り当て競合を一つ置く。競合がない場合は、すべての物理グループ競合をグループパス順に置く。構造設定エラーは、この集約より前に中止する。対象処理開始後、metadata、読み込み、抽出、フォーマッター／リンター、検証、および書き込みの失敗は、その論理対象の `results[].errors` だけに置き、トップレベル配列へ重複して追加しない。

この層が所有する初期の安定した `internal_error.details.reason` 値は次のとおりである。

- `resource_invalid_entry_handle`: 別アーティファクト由来、重複、存在しない、読み取り専用、またはその他の無効なアーティファクトローカルハンドルが、内部の利用側境界へ到達した。
- `resource_artifact_identity_exhausted`: 元の抽出または候補検証中に、プロセス全体のアーティファクト識別子アロケーターに、未使用の非ゼロ `u64` 値が残っていない。
- `resource_offset_map_invariant_failed`: 抽出中、アダプターが無効なマップを公開しようとした。
- `resource_offset_map_failed`: 有効であるはずのコアスパンを、公開済みマップでマッピングできない。
- `resource_write_back_failed`: エントリを書き込み可能と分類した後に、再エスケープ、置換検証、候補ホスト解析、再抽出、エントリ識別子の保持、または値の等価性が失敗した。
- `resource_adapter_invariant_failed`: その他の組み込みアダプター状態またはレジストリ不変条件が失敗した。

想定されるホスト構文失敗は `internal_error` を使用しない。一方、アダプターが生成した無効な状態を、検証でホストテキストを再解析するというだけで `resource_parse_failed` にしてはならない。CLI は、選択された対象パスと、判明している場合は構造化された `{ path, occurrence }` エントリキーを付加する。空でないリソース運用エラー配列は、共有の `2 > 1 > 0` 優先順位に従って、`summary.status: "error"` と終了コード `2` を生成する。

理由ごとの安定した `details` は、リソース CLI JSON 契約の一部である。

生バイト列が UTF-16 BOM で始まる分類済み XLIFF 対象について、`input_read_failed.details` は `reason: "unsupported_encoding"` を使用し、`detectedEncoding` として正確に `"utf-16le"` または `"utf-16be"` を要求する。その他の UTF-8 でない入力は、共有の `reason: "invalid_utf8"` 形状を維持し、エンコーディングを推測しない。

`resource_format_unsupported` は次の details を使用する。

```json
{
  "classificationSource": "extension",
  "format": "yaml",
  "extension": ".yaml",
  "supportedFormats": ["json"]
}
```

- `classificationSource` は必須であり、`"extension"` または `"embedded"` とする。明示的な設定 `format` がこのエラーへ到達することはない。リリース済み ID は `HostFormat` を構築し、未知、既知だが未リリース、alias、またはその他の無効な値は、`config_validation_failed` によるスキーマ検証で失敗する。
- `format` は正規化済みの小文字レジストリ ID とする。拡張子、設定済み入力、または埋め込み宣言を既知形式 ID と関連付けられない場合に限って省略する。
- `extension` は必須で、先頭の `.` を含み、対象パスの綴りを保持する。拡張子が存在しない場合は `""` とする。
- 拡張子で分類された対象では `outerFormat` を省略する。`"embedded"` では必須であり、すでに選択された外側アダプターの正規化済みレジストリ ID を含める。
- `supportedFormats` は必須であり、レジストリ ID を ASCII 昇順で含める。拡張子分類では、リリース済みトップレベルアダプターを列挙する。埋め込み分類では、その外側アダプターが受け入れるリリース済み内側アダプターを列挙する。
- 選択された対象パスは、`details` に重複させず、運用エラーのトップレベル `path` とする。

たとえば、YAML アダプターのリリース前に、インライン `lang="yaml"` ブロックを持つ Vue SFC は次の details を使用する。

```json
{
  "classificationSource": "embedded",
  "format": "yaml",
  "extension": ".vue",
  "outerFormat": "vue",
  "supportedFormats": ["json"]
}
```

完全な `.vue` ファイルが、選択された対象およびトップレベルエラーパスのままとなる。このコードの安定した CLI details にはエラー位置を追加しない。ただし、型付きエラーは、埋め込み宣言について元ホストの site を保持する。値を持つ `lang` が存在する場合、引用符があるときはそれを含む生の属性値トークン全体を使用する。そのため、空の引用符付き値は引用符で区切られた空トークンを使用し、値のない `lang` は属性名トークン全体を使用する。拡張子で分類された失敗にはソーストークンがないため、site もない。未対応のインラインブロックが複数ある場合、外側アダプターは、人間向けメッセージとエディター投影のために、site の開始位置が最小のものを決定的に選択する。

`resource_parse_failed` は次の details を使用する。

```json
{
  "format": "json",
  "offset": 18,
  "line": 2,
  "column": 4
}
```

- 四つのフィールドはすべて必須である。ホストパーサー依存ライブラリは、アダプターが絶対 UTF-8 バイトオフセットへ変換できる主要構文エラー位置を提供しなければならない。その機能を持たないパーサーは、スパン保持アダプター契約を満たさない。
- `format` は入力を拒否したアダプター層を識別する。直接のホスト解析または外側層の失敗では、`outerFormat` を省略する。合成された内側アダプターの失敗では必須であり、正規化済みの外側レジストリ ID を識別する。
- `offset` は、正確なホスト文書の先頭からのゼロ起点バイトオフセットであり、予期しない入力終端ではソースバイト長と等しくてよい。
- `line` は一起点、`column` はゼロ起点 UTF-8 バイト列とし、共有診断の `SourceLocation` 規約に合わせる。両方とも、依存ライブラリ固有の line／column セマンティクスをコピーせず、正確なホストソースと `offset` から導出する。
- エラーより前のすべての生バイトを、`offset` と最初の行のバイト列へ含める。解析用に先頭 BOM を一つ除去するアダプターは、その 3 バイトを加算して戻す。一方、JSON5 はすべての `U+FEFF` 空白をパーサーへ通し、直接数える。パーサー固有のエラーコード、期待トークン集合、および説明文は、安定した `details` ではない。運用エラーの人間向け `message` は、互換性サーフェスにならない範囲でそれらを要約してよい。

たとえば、有効な Vue SFC 内に無効な JSON がある場合は、次の details を使用する。

```json
{
  "format": "json",
  "outerFormat": "vue",
  "offset": 118,
  "line": 8,
  "column": 6
}
```

位置は、内側パーサーのオフセットを完全な外側ソースへ持ち上げた後に導出する。無効な Vue SFC では、`format: "vue"` を使用し、`outerFormat` を省略する。

`resource_entry_unsupported` は次の details を使用する。

```json
{
  "format": "json",
  "reason": "message_text_unrepresentable",
  "offset": 18,
  "line": 2,
  "column": 4
}
```

- 五つのフィールドはすべて必須である。初期の安定した `reason` 値は `"message_text_unrepresentable"`、`"structural_path_unsupported"`、`"inline_content_unsupported"` とする。アダプターが別の値を追加できるのは、それを必要とする延期済み形式契約と同時に限る。
- `format` は候補を拒否したアダプター層を識別する。直接対象または外側層の失敗では `outerFormat` を省略し、合成された内側アダプターの失敗では必須とする。`resource_parse_failed` と同じ正規化済みレジストリ ID 規則を使用する。
- 型付きエラーの site は、最小オフセットの候補を最初に未対応にする、最小の完全な生構文トークンとする。すなわち、対応のないサロゲートエスケープ全体、未対応の複雑なマッピングキーノード、または最初のインライン子要素の完全な開始タグトークンである。未対応の性質に対して、より狭いトークンがない場合は、候補の完全な `raw_value_span` を site とする。公開 `offset` は正確にその site の開始位置とする。
- `offset`、`line`、`column` は、保持された BOM バイトとアダプター合成による持ち上げを含め、`resource_parse_failed` と同じ正確なホスト UTF-8 座標規則を使用する。運用エラーのトップレベル `path` は、完全な外側ホストファイルを識別する。
- 有効な `MessageEntry` 識別子を構築する前に失敗するため、`entryKey` は付加しない。その対象について、抽出済みエントリ、診断、フォーマット済み候補、または部分的なファイル書き込みは返さない。

`resource_document_unsupported` は次の details を使用する。

```json
{
  "format": "yaml",
  "feature": "multiple_documents",
  "offset": 42,
  "line": 5,
  "column": 0
}
```

- 五つのフィールドはすべて必須である。初期の安定した `feature` 値は、`"vue_src_with_inline_content"`、`"multiple_documents"`、`"yaml_version"`、`"custom_tags"`、`"aliases"`、`"merge_keys"`、`"xliff_version"`、`"xml_dtd"`、`"xml_version"`、`"xml_encoding"` とする。将来のアダプターが値を追加できるのは、そのアダプターで安全に表現できない文書全体の構造に対してだけである。
- `format` は、対応済み文書モデルがその構造を拒否したアダプター層を識別する。直接対象または外側層の失敗では `outerFormat` を省略し、合成された内側アダプターの失敗では必須とする。
- 型付きエラーの site は、認識によって未対応文書機能が成立する、最小の完全な生構文トークンとする。複数文書 YAML ストリームでは、存在する場合は二つ目の文書の完全な `---` マーカー、そうでなければその最初の構文トークンとする。未対応 YAML バージョンでは完全な `%YAML` ディレクティブトークン、未対応 YAML タグでは最初に使用された Core 外タグトークンとする。alias では `*` から始まる完全な alias トークン、マージキーではマージ値自体が alias であっても、完全な `<<` キートークンとする。未対応 XLIFF プロファイルでは、存在する場合は完全な `version` 属性値トークン、そうでなければルート `<xliff>` 開始タグトークンとする。DTD では `<!DOCTYPE` 開始トークン、未対応 XML version または encoding では、対応する宣言値トークンとする。`vue_src_with_inline_content` は、最初の非 HTML 空白文字の完全な UTF-8 スカラースパンを使用する。未対応構造が複数ある場合、site の開始位置が最小のエラーを選択する。公開 `offset` は正確にその開始位置とし、`line` と `column` は、アダプター合成による持ち上げを含め、`resource_parse_failed` と同じ正確なホスト UTF-8 座標規則を使用する。
- 運用エラーのトップレベル `path` は、完全な外側対象を識別する。`entryKey`、部分的なアーティファクト、診断、フォーマット済み候補、または部分書き込みは返さない。

`resource_limit_exceeded` は次の details を使用する。

```json
{
  "phase": "extract",
  "resource": "host_bytes",
  "limit": 67108864,
  "actual": 67108865
}
```

- `phase` は必須であり、元のホストアーティファクトでは `"extract"`、候補アーティファクトでは `"validate_write_back"` とする。
- `resource`、`limit`、`actual` は必須である。二つの数値フィールドは、型付きエラーの `u128` カウンターから正確にシリアライズされた、負でない JSON 整数とし、浮動小数点値を決して経由しない。
- 初期の `resource` 値は、`"host_bytes"`、`"nesting_depth"`、`"entries"`、`"message_bytes"`、`"total_message_bytes"`、`"identity_bytes"`、`"offset_map_segments"` とする。
- `limit` は、その表現で受け入れる固定された包含的上限である。`actual` は、前述の決定的な検査順で、それを超えた最初の観測値である。候補の `host_bytes` 予測は、計測後に文書全体で一度観測する。そのため、その `actual` は完全な候補の正確な予測バイト長となる。内部のバイトスパンおよびエントリハンドルドメインは `u32` のままだが、より小さい公開製品上限を、それらの表現上限へ到達する前に適用する。

型付き `ResourceError` は、省略可能な主要 site も保持する。site は、ユーザーが保持しない書き戻し候補だけではなく、呼び出し側が保持する元のホスト文書を基準に表す。site は元ホストの `Utf8ByteSpan` と、失敗が構築可能なエントリに属する場合は、その `EntryKey` を含む。`resource_format_unsupported`、`resource_entry_unsupported`、`resource_document_unsupported` は、前述のトークンスパン規則を使用する。最初のコードは、拡張子分類では site を持たず、埋め込み分類では埋め込み宣言 site を持つ。後者二つは、完全な原因トークンまたはフォールバック候補 site を常に持ち、有効な `MessageEntry` より前に失敗するため、エントリキーを決して持たない。`resource_limit_exceeded` の初期 site 規則の残りは、次のとおりである。

- `host_bytes` は、元の上限を UTF-8 デコード前に検査し、候補予測を文書全体で行うため、site を持たない。
- 元の抽出における `nesting_depth` は、深さ `257` を成立させる開始構造トークンを指す。エントリキーは持たない。
- `entries`、`message_bytes`、`total_message_bytes` は、違反したエントリの完全な元の `raw_value_span` を指し、そのキーを持つ。同じオフセットの優先順位に従って、エントリの表現可能性検査はすでに成功しているため、返されるアーティファクトへエントリを追加できなくても、このキーは利用可能である。
- `identity_bytes` は、同じ完全な元の `raw_value_span` を指し、構造、カタログ、または表示用識別子によって最初の超過した異なる文字列を導入する、構築可能なエントリキーを持つ。超過した文字列は、アーティファクトインターナーへ受け入れられない場合でも、エラー値が保持してよい。
- `offset_map_segments` は、実行中の上限を超えたセグメントの元の生スパンを指し、それを含むエントリキーを持つ。
- `validate_write_back` 中、エントリに関係する超過は、対応する元のアーティファクトエントリへマッピングし、そのエントリの元の生スパンとキーを使用する。正確に対応する元の構造を識別できない、文書全体または構造上の候補超過は site を持たない。候補だけのバイト座標は公開しない。

`resource_limit_exceeded` では、site が存在する場合、CLI JSON は `resource_parse_failed` と同じ元ホスト座標規則を使用して、`offset`、`line`、`column` を追加する。三つのフィールドは必ず一緒に現れ、site がエントリキーを持つ場合は `entryKey` も現れる。このコードが site を持たない場合、四つのフィールドをすべて省略する。`resource_entry_unsupported` と `resource_document_unsupported` は、必須位置を `site.span.start` からシリアライズする。`resource_format_unsupported` は、型付き site を決してシリアライズせず、前述の安定した details 形状を維持する。エディターの利用側は、対処可能なリソースコードを投影する際に、型付き元ホストスパンを直接使用する。site がないコードインスタンスに限り、文書先頭の空範囲を使用する。

たとえば、元エントリのメッセージ超過は次の details を使用できる。

```json
{
  "phase": "extract",
  "resource": "message_bytes",
  "limit": 1048576,
  "actual": 1048577,
  "offset": 18,
  "line": 2,
  "column": 4,
  "entryKey": {
    "path": "/greeting",
    "occurrence": 0
  }
}
```

リソース層が所有する `internal_error` 値では、`reason` と `phase` の両方を必須とする。

```json
{
  "reason": "resource_write_back_failed",
  "phase": "validate_write_back",
  "entryKey": {
    "path": "/greeting",
    "occurrence": 0
  }
}
```

初期の安定した `phase` 値は、`"registry"`、`"extract"`、`"map"`、`"validate_write_back"` とする。`entryKey` は通常の構造化識別子を使用し、一つのエントリが判明している場合に含める。文書全体の失敗またはエントリ識別子より前の失敗では省略する。ソーステキスト、生置換テキスト、依存ライブラリ固有のエラー、および内部アダプター型名は決して含めない。構造化フィールドを追加するには、任意の Rust debug データを漏らすのではなく、別途文書化された安定した必要性が求められる。

### 決定性と並列性

ファイル結果は安定した正規化済みパス順で報告し、ファイル内のエントリは生スパン順で報告する。この順序には、グループ書き込み失敗後に処理されない後続 alias すべてに対して合成する `alias_processing_blocked` 結果も含む。初期 CLI 統合は、上限付きでコマンド単位の一つの worker pool を用いて物理ファイルグループ間を並列化し、一つの物理ファイルの論理 alias 間では決して並列化しない。失敗した書き込みは、そのグループで処理待ちの alias だけを停止し、実行中または処理待ちの無関係なグループをキャンセルしない。worker は識別子付きの構造化結果を返し、グループ作業の完了後に調整側が安定した集約と reporter 描画を行う。リソースクレートは引き続き scheduler に依存せず、thread pool または runtime を作成しない。

初期実装ではファイル内並列性を導入しない。ファイル内実行を再検討するのは、ファイルレベル並列性のベンチマークによって、単一の非常に大きなカタログがコマンドの pool を実質的に十分活用できないことが示された後だけとする。将来の単位は、一エントリごとの task またはホスト形式の構文サブツリーではなく、生スパン順に連続するエントリの chunk とする。chunk 化により、小さいメッセージのスケジューリングオーバーヘッドを償却し、ホスト形式への依存を避け、クレートが所有するネストした並列性または oversubscription を発生させずに、単一カタログが CLI 所有の同じ pool を共有できる。

chunk の結果は元のエントリインデックスを保持し、レポートまたは書き戻しより前に、生スパン順へ再構築する。フォーマッターの置換合成は、すべての chunk 成功後に、一つの決定的なファイルレベル手順として行う。並行するエントリ処理が複数の運用エラーを生成した場合、task の完了順にかかわらず、生順で最も小さいエントリのエラーを選択する。既存の部分結果なしおよび部分書き込みなしの規則を引き続き適用する。正確な chunk サイズ、最小エントリ数または計測済み作業量のしきい値、および worker スケジューリング戦略は、設定フィールドまたは CLI 互換性サーフェスではなく、ベンチマークで調整する内部詳細とする。

ベンチマークでは、抽出、再エスケープ、コア呼び出し、スケジューリングオーバーヘッド、およびファイル I/O を別々のフェーズとして報告する。ファイル内並列性を導入できるのは、代表的な大規模カタログのベンチマークで、スケジューリングと順序付き集約のコストを含めた後も、再現可能な改善を示した場合だけである。

### カタログ JSON 結果レイアウト

JSON reporter は、選択されたホストファイルごとに一つの `results[]` 項目を維持し、完全なエントリ結果をカタログ対象の内側へネストする。単独メッセージ結果では `diagnostics` を必須とし、`entries` を省略して、Phase 3B および Phase 3C の形状を維持する。カタログ結果では `entries` を必須とし、ファイルレベルの `diagnostics` フィールドを省略する。型付き Rust 結果 DTO は、これらを互いに排他的な enum バリアントとして表現する。診断オブジェクトをファイルレベルとエントリレベルに重複させず、リソース固有の省略可能フィールドを追加しない。

`schemaVersion` が `"0"` の間、内部テスト専用スキーマを含め、コマンド出力 JSON Schema を生成、コミット、または公開しない。設定 JSON Schema は別の互換性サーフェスであり、引き続きコミットおよび公開する。型付き Rust enum／DTO と正確な JSON fixture によって、出力バリアント、必須および省略フィールド、フィールド順、およびサマリー集約を定義し検証する。出力 JSON Schema の公開は、`schemaVersion` が安定した後に限って再検討する。

抽出と完全なエントリ単位処理に成功した場合、`entries` は、clean、unchanged、診断あり、および読み取り専用エントリを含む、抽出済みメッセージエントリすべてを生スパン順で含む。アダプターが表示用識別子を持たない場合、`displayKey` を省略する。リンターバリアントは次のとおりである。

```json
{
  "path": "locales/en.json",
  "status": "problems",
  "entries": [
    {
      "key": {
        "path": "/greeting",
        "occurrence": 0
      },
      "status": "problems",
      "diagnostics": []
    }
  ],
  "errors": []
}
```

lint エントリの `status` は `"clean"` または `"problems"` とし、ファイルレベルの `--quiet` 規則に従い、reporter フィルタリングより前の完全なエントリ診断集合から計算する。フォーマッターバリアントは、ファイル結果と各エントリ結果の両方へ `changed` を追加し、エントリスコープでフォーマッターの status 語彙を使用する。

```json
{
  "path": "locales/en.json",
  "status": "formatted",
  "changed": true,
  "entries": [
    {
      "key": {
        "path": "/greeting",
        "occurrence": 0
      },
      "status": "formatted",
      "changed": true,
      "readOnly": false,
      "diagnostics": []
    }
  ],
  "errors": []
}
```

すべてのフォーマッターエントリ結果は `readOnly` boolean を持つ。完了した書き込み可能なエントリの `status` は `"formatted"`、`"would_format"`、`"unchanged"`、または `"diagnostic"` とし、対応するファイル status と同じ、モードに依存する意味を持つ。パーサー診断のない読み取り専用エントリは、`status: "skipped"`、`changed: false`、`readOnly: true` を使用する。`"unchanged"` は、正常なフォーマット済み出力が差分評価へ参加した書き込み可能なエントリのために予約する。読み取り専用エントリにパーサー診断がある場合、`readOnly` を `true` のままにして `"diagnostic"` を優先し、その診断を隠さない。

完全な JSON エントリ配列を、読み取り専用 skip に対する唯一の参考情報レポートとする。人間向けの write、check、および list-different 出力では、stdout または stderr に表示しない。サマリーへ読み取り専用カウンターを追加せず、skip はファイル status またはプロセス終了コードへ影響しない。特に、有効な読み取り専用エントリだけを含むカタログは `"unchanged"` となり、パスだけを出力する `fmt --check` 結果には現れない。

エントリ結果は `errors` 配列を含まない。ホストレベルの運用エラーは、ファイル結果の `errors` に留める。エントリ単位の運用エラーも、後述の失敗規則に従ってカタログ結果を不完全にする。ファイルは `status: "error"`、空の `entries` 配列を持ち、判明している場合はファイルレベルエラーが同じ `{ path, occurrence }` オブジェクトを `details.entryKey` として持つ。これにより、部分的なエントリ結果を完全なカタログ結果であるかのように提示することを避ける。

書き込み前の完全な処理後に発生した `output_write_failed` も、エントリ評価自体は完了している場合でも、同じ空エントリのエラー形状を使用する。直接書き込みの失敗によって保持されるホストソースが不確定になるため、reporter は、元または候補アーティファクトのどちらにマッピングされた診断も保持せず、サマリーにも数えない。

カタログエントリ結果内のすべての診断 `span`、`location`、`labels[].span` は、マッピング済みのホスト文書座標を使用する。`span` および label span は、適用対象となる完全なホストソース内の絶対 UTF-8 バイト範囲であり、`location` はそのソースから導出する。カタログフォーマットでは、後述のモード固有規則に従って元ソースまたは検証済み候補ソースを選択する。lint は常に入力ソースを使用する。JSON 結果では、並行する `messageSpan` またはメッセージローカルな位置を公開しない。メッセージローカルなスパンは、`MessageOffsetMap` マッピング前のコア診断結果内部で引き続き利用できる。公開座標空間を一つに保つことで、利用側がメッセージローカル範囲をホストファイルへ誤適用することを防ぐ。将来、両方の座標空間を必要とする具体的な利用側には、`span` を多重定義するのではなく、明示的なスキーマ追加が必要である。

利用側は、フォーマッターファイル書き込みまたは通常 text reporter の stdin ホストソース出力より前に、すべての診断マッピングを完了し、正常なエントリ結果データ全体を構築する。マッピングは、エントリを生スパン順、診断を保持されたコア結果順で走査し、各診断について primary span の後に、保持された配列順で label を走査する。その順序で最初の `OffsetMapError` は、`details.reason: "resource_offset_map_failed"` を持つ `internal_error` となる。カタログ結果は、空の `entries` 配列を持つ `status: "error"` を使用し、部分的な診断または正常なエントリ結果を公開しない。フォーマッターファイルモードは書き込まず、フォーマッター stdin はホストソースを出力しない。これは書き込み前の失敗であるため、後続の物理 alias は処理可能なままとする。後から発生する `output_write_failed` だけが不確定ファイル境界を越え、その物理グループ内の後続 alias を阻止する。

## カタログの lint

カタログの lint は、エントリごとにメッセージレベルのリンターを実行し、ファイル単位で集約する。

- 各エントリのメッセージテキストを、`.mf2` ファイルと同じ解決済み lint 設定を使用して `lintMessage(source, options)` に渡す。厳格な `parser -> semantic -> rules` パイプラインを、エントリ単位で適用する。
- エントリは独立している。一つのエントリにパーサー診断があっても、他のエントリの意味検証、lint、またはレポートを抑止しない。
- JSON 内のエントリ診断は、ホストファイル座標だけで報告する。具体的には、エントリのオフセットマップによって生成された、マッピング済みのホスト UTF-8 バイトスパンと、そこから導出した行／列である。各エントリレベル結果はエントリキーを持ち、人間が読みやすい出力のために表示キーを付加してよい。メッセージローカルなスパンは内部に留め、マッピング済みスパンと並べてシリアライズしない。
- エントリ単位のリンター呼び出しのいずれかが、完全な診断結果ではなく運用エラーを返した場合、カタログ対象は既存の対象レベルリンターエラー契約に従う。`status: "error"`、空の `entries` 配列、および `results[].errors` 内のエラーを報告し、エントリが判明している場合は `details.entryKey` を含める。対象結果が不完全であるため、そのカタログ内の他のエントリから収集済みの診断を破棄する。他の選択済みファイルは引き続き処理する。
- エントリレベル結果は、前述のネストされたカタログ JSON 結果 DTO を、リンター JSON 結果契約の拡張として使用する。[008-ox-mf2-phase-3c-linter-design.md](./008-ox-mf2-phase-3c-linter-design.md) の既存のエンベロープ、サマリー、および件数規約の内側に置く。診断件数と `--max-warnings` には `entries[].diagnostics` を含める。
- lint オプションは一様に適用する。初期段階では、カタログ単位のルール上書きはない。後述する証拠を条件とする再検討方針を、lint とフォーマッターの両オプションへ適用する。

正常に抽出され、メッセージエントリがゼロのカタログは、対象ゼロの探索結果でも運用エラーでもなく、完了したリンター対象一つである。そのファイル結果は `status: "clean"`、`entries: []`、`errors: []` とする。`matchedFiles` と `cleanFiles` をそれぞれ一つ増やし、`problemFiles`、診断件数、および `errorCount` は増やさない。ほかに失敗がなければ、text reporter は何も出力せず、コマンドは終了コード `0` で終了する。ignore されていないカタログ stdin は、その仮想パスに対して同じ対象およびサマリー処理に従い、JSON reporter で clean なカタログ結果を一つ出力する。

## カタログのフォーマット

カタログのフォーマットは、エントリごとにメッセージレベルのフォーマッターを実行し、ファイル単位で書き戻し編集を合成する。

- 書き込み可能性にかかわらず、抽出済みの各エントリのメッセージテキストを、`.mf2` ファイルと同じ解決済み fmt 設定を使用して `formatMessage(source, options?)` に渡す。利用側は、メッセージローカル座標のパーサー診断を保持する。読み取り専用エントリの正常なフォーマット済み出力を破棄し、診断を持つエントリを省略する。選択されたすべてのフォーマッター呼び出しが運用エラーなく完了した後、バイト列が同一の出力を含む、残ったすべての書き込み可能な正常出力を、一つの `build_and_validate_write_back` 呼び出しへ渡す。
- 候補の構築と検証は、write、`--check`、`--list-different` の各ファイルモード、および通常と check の stdin モードで同一である。`WriteBackOutcome::Unchanged` は、すでに抽出済みの元アーティファクトを使用し、置換集合または候補を持たない。`Changed` は、生スパンのオフセット降順で内部的に編集を適用した後の、完全な生順置換集合と、再解析、再抽出、値検証済みの候補アーティファクトを提供する。check モードは、そのソースを元のホストソースとバイト単位で比較した後、検証済み候補を破棄する。write モードと同じ、リソース層所有の再エスケープおよび文書全体検証を通過できない差分を報告することは決してない。
- write モードは、Phase 3B CLI のファイル I/O および運用エラー規約を使用しつつ、単独 `.mf2` のファイルフレーミングを使用せず、検証済みホストソースを一度だけ書き込む。置換される値スパン外のすべてのバイトは、ホストファイルの BOM と末尾改行も含め、バイト単位で同一のままとする。キーの並べ替え、メッセージ値外のインデントまたは引用符の変更、およびホストレイアウトの正規化は行わない。check モードはファイルシステムへ書き込まず、通常のカタログ stdin は、既存の stdout 契約のもとでだけ、検証済みホストソースを出力する。
- その一度のファイル書き込みが `output_write_failed` を返した場合、現在の論理対象は `status: "error"`、`changed: false`、`entries: []`、および書き込みエラーを使用し、`matchedFiles` と `errorCount` だけに影響する。その後 CLI スケジューラーは、物理グループの fail-stop 規則を適用する。後続の各 alias は、別のファイルシステムアクセスなしに `alias_processing_blocked` を受け取る。書き込み境界より前の失敗と、書き込みを行わないすべてのモードは、通常どおり次の alias へ進む。
- 抽出済みテキストにパーサー診断があるエントリは、エントリ単位でスキップし、[007-ox-mf2-phase-3b-formatter-design.md](./007-ox-mf2-phase-3b-formatter-design.md) が無効な単独ファイルについて確定するものと同じ、厳格な無効構文セマンティクスをエントリへ限定して報告する。他のエントリは引き続きフォーマットする。一つの壊れたメッセージによって、大規模なカタログ全体のフォーマットが妨げられてはならない。これは [009-ox-mf2-phase-3d-lsp-editor-design.md](./009-ox-mf2-phase-3d-lsp-editor-design.md) のエディターの振る舞いと同様である。
- カタログ対象において、パーサー診断の影響を受けるフォーマット単位は、ホストファイル全体ではなくエントリである。診断があるエントリは決して置換しないが、同じカタログファイル内の他のエントリに有効な変更があれば、ファイルへ一度だけ書き込んでよい。これは、パーサー診断が影響を受けるフォーマット単位を変更しないという Phase 3B の規則を、リソース層に特化したものである。
- 読み取り専用エントリは、フォーマッター評価からではなく、書き戻しおよび差分報告から除外する。パーサー診断を持つ読み取り専用エントリは `status: "diagnostic"` を使用する。それ以外では、write および check モードは正常なフォーマット済み出力を破棄し、`status: "skipped"`、`changed: false`、`readOnly: true` を記録する。`FormattedEntry` として渡さず、そのための `RawReplacement` を構築しない。正常な読み取り専用 skip によって、人間向け出力、サマリー件数、ファイル status、またはプロセス終了コードを報告または変更しない。
- `--check` と `--list-different` は、書き込み可能で構文的に有効なエントリが一つでもフォーマット後に異なる場合、そのカタログファイルを差分ありとして報告する。
- エントリ単位のフォーマッター呼び出し、または `build_and_validate_write_back` が運用エラーを返した場合、そのカタログには書き込まず、部分的なフォーマット結果も生成しない。フォーマッターの運用エラーが一つでもある場合、統合メソッドを呼び出さない。複数のフォーマッター呼び出しが失敗した場合、生順で最も小さいエントリを選択する。ファイル結果は、`status: "error"`、`changed: false`、空の `entries` 配列、および `results[].errors` 内の選択済み運用エラーを使用し、判明している場合は `details.entryKey` を含める。固定リソース上限の超過は引き続き `resource_limit_exceeded` とする。エントリを書き込み可能と分類した後の、その他の再エスケープまたは候補検証失敗は、`details.reason: "resource_write_back_failed"` を持つ `internal_error` とする。その他の選択済みファイルは引き続き処理する。
- 冪等性: 検証済み書き戻しのラウンドトリップ則とメッセージレベルフォーマッターの冪等性から、フォーマット済みのカタログを再度フォーマットしても書き込みは発生しない。fixture でこの性質を固定する。

フォーマッター診断は、利用側がユーザーの保持するホストソースを判断するまで、メッセージローカルのままとする。書き込み前処理中、ファイル write モードは、書き込みによって保持しようとするソースである検証済み候補アーティファクトを通じて診断をマッピングする。マッピングと正常結果全体の構築が完了した後に限って、利用側は書き込みを試行する。通常の text-reporter stdin も同様に、stdout へ候補ソースを出力する前に、そのアーティファクトを通じてマッピングする。`--check`、`--list-different`、stdin check、および JSON reporter を使用する通常 stdin は、変更済みホストソースを公開しないため、元アーティファクトを通じてマッピングする。候補検証により、診断を持つすべてのエントリは置換されず、両アーティファクト内でバイト単位に同一のメッセージテキストを持つことを保証する。そのため、フォーマッターを再実行せず、同じメッセージローカル診断をどちらのアーティファクトでもマッピングできる。

正常に抽出され、メッセージエントリがゼロのカタログは、対象ゼロの探索結果でも運用エラーでもなく、完了したフォーマッター対象一つである。その空の `FormattedEntry` 入力は `WriteBackOutcome::Unchanged` へ解決される。ファイル結果は `status: "unchanged"`、`changed: false`、`entries: []`、`errors: []` とする。`matchedFiles` と `unchangedFiles` をそれぞれ一つ増やし、`formattedFiles`、`differentFiles`、および診断件数は増やさない。ファイル write モードは書き込まず、check および list-different モードは人間向け出力を生成せず、ほかに失敗がなければ終了コード `0` で終了する。通常の text-reporter stdin は、元のホストソースをバイト単位で出力する。JSON reporter を使用する stdin は、通常の単一対象 stdin サマリーのもとで、unchanged なカタログ結果を一つ出力する。

一つのカタログに、有効で変更のあるエントリと、パーサー診断のあるエントリの両方が含まれる場合がある。少なくとも一つの `entries[].diagnostics` 配列が空でない場合、カタログはエントリ診断を持つ。この集約条件を、ファイルレベルの `diagnostics` フィールドを追加または重複させず、ファイル status、サマリー件数、および終了 status に使用する。ファイルレベルの JSON 結果は Phase 3B status enum を維持し、運用上の失敗なく処理に成功した後、次の優先順位を適用する。

1. write モードでは、合成されたバイト変更が一つでもあれば、一つ以上のエントリに診断がある場合でも、`status: "formatted"`、`changed: true` とする。ファイルへ一度だけ書き込み、診断が残るためコマンドの終了コードは引き続き `1` となる。
2. check モードでは、合成されたバイト差分が一つでもあれば、一つ以上のエントリに診断がある場合でも、`status: "would_format"`、`changed: true` とする。ファイルへの書き込みは行わない。
3. 書き込み可能なエントリに差分がなく、一つ以上のエントリに診断がある場合、結果は `status: "diagnostic"`、`changed: false` とする。
4. 書き込み可能なエントリに差分がなく、すべてのエントリ診断配列が空で、運用エラーもない場合、結果は `status: "unchanged"`、`changed: false` とする。

したがって、`formattedFiles` または `differentFiles` と `diagnosticFiles` は、同じカタログファイルを数える場合がある。これらのサマリー件数は、意図的に `matchedFiles` の分割にはなっていない。`diagnosticFiles` は、いずれかのエントリ診断配列が空でない場合にカタログを一度数える。`diagnosticCount` は、それらの配列内のすべてのマッピング済み診断の合計である。運用エラーによって終了コード `2` に引き上げられない限り、エントリ診断または check 時の差分がある場合、`summary.status: "failure"` と終了コード `1` を維持する。

## エディターでの利用

プロジェクト設定で指定されたカタログ文書について、エディターアダプターは、この共有層を通じて同じ `ResolvedResources` パスリゾルバー、レジストリ、抽出、オフセットマップ、および書き戻し時の再エスケープを利用する。これにより、所属関係、ホスト形式の分類、競合の証拠、およびマッピングは、CI とエディターで同一になる。プロジェクト解決を最初に実行し、権威を持つものとする。エディター専用のアドホックなオプトインは、前述の重複規則のもとで、一致しない文書にだけ追加する。アドホックな対象も同じホストアダプター契約を使用するが、プロジェクト設定へ永続化されるまで、その所属関係は意図的にエディター内だけに留まる。単独の `.mf2` 文書は、カタログ抽出ではなく、引き続き [009-ox-mf2-phase-3d-lsp-editor-design.md](./009-ox-mf2-phase-3d-lsp-editor-design.md) で定義されたエディター管轄のアダプターを通る。サーフェスの振る舞いは意図的に異なる。エディターはマッピング済みスパンをエディターの位置エンコーディングへ変換し、文書単位で公開し、一時的なホスト解析失敗の間は以前の診断を保持し、古い文書バージョンに対しては何もしない。一方、CLI はホスト解析失敗を対象ローカルな運用エラーとして報告し、ファイルへ直接書き込む。

## カタログレベルのチェック

カタログレベルおよびロケール横断 lint は、設計の完成と実装の両方について、後続対応として延期する。本セクションの記述は、現在のリソースマイルストーンでルールを予定せず、初期スキーマのフィールドを予約せず、reporter または result バリアントを追加しない。後述する、すでに確定済みのエントリ識別子、キードメイン、locale binding、および比較スコープの制約は、将来の専用リンター追補文書への入力である。最初のルールを実装する前に、その追補文書で製品契約と出力契約を完成させなければならない。

カタログキーの重複、ロケール間でのキーの一致、翻訳の欠落または未使用など、カタログレベルおよびロケール横断のチェックは、この層より上位で複数のエントリとファイルを横断して動作する。ファイル内重複キールールは、出現番号を除いた完全な具体構造パスでグループ化するため、ロケールメタデータなしで動作する。ロケール横断ルールは、`CatalogKey` 値をグループ化し、後述の locale binding と比較スコープを必要とする。未使用翻訳ルールには、製品固有の追加参照入力が必要になる場合がある。これらのルール識別子、入力、preset、および severity は、リンター製品の設計トラックが管轄する。これらは将来の作業であり、二つ目の抽出経路を導入せず、同じエントリモデルを経由しなければならない。

将来のリソース対応 config-lint ルールは、解決済み外部対象が `resources.catalogs` に一致しない Vue `<i18n src>` 参照を検出し、カタログ対応範囲の見えない欠落を防いでよい。これは初期抽出の振る舞いではない。Vue アダプターは引き続き `src` ブロックのエントリを返さず、推移的なパス解決、存在検査、読み込み、または診断公開を行わない。そのようなルールをリリースする前に、リンター設計は、ルール ID と既定 severity、SFC からの静的 `src` パス解釈、非ファイル URI、`projectRoot`、symlink、および欠落ファイルの処理、共有 `ResolvedResources` リゾルバーによる検索、重複報告の抑止、および具体的な finding の所有者とソース範囲を確定しなければならない。

カタログ finding の論理 subject は、一つの比較スコープのもとにある型付きバリアントとする。具体パス subject は、ホスト重複ルール向けに `StructuralPathKey` を持つ。カタログメッセージ subject は、ロケール横断ルール向けに `CatalogKeyDomain` と `CatalogKey` の組を持つ。どちらの subject バリアントにも出現番号を含めない。具体的な evidence は、正規化済みホストパスと、完全な `{ path, occurrence }` の `EntryKey` を参照する。そのため、重複キー finding は、XLIFF source と target を混同せず、一つの具体的な役割について重複した全出現箇所を参照できる。一方、ロケール欠落 finding は、欠落ロケールと、side を含まないカタログキーが別の場所に存在することを示すエントリを識別できる。

カタログ finding は、所有するホストファイルを持たない。リンターは、ロケール欠落 finding を任意の既存エントリへ付加したり、存在しないエントリまたは長さゼロのホストスパンを合成したり、一つの論理 finding を関係する全エントリへ複製したりしてはならない。現在の `results[]` ファイルバリアントと `entries[].diagnostics` は、具体的なメッセージエントリに属する診断だけの契約として維持する。

最初のカタログレベルルールをリリースする時点で、そのリンター設計は、ファイル結果と並ぶ専用のカタログレベル結果および finding collection を追加する。その設計は、シリアライズ済み比較スコープ識別子、型付き subject バリアントとキー、影響を受けるロケール、具体的な関連エントリ参照、省略可能な実在ソーススパン、決定的な順序、人間向けおよびエディター表示、`--quiet` および `--max-warnings` の振る舞い、およびサマリー件数を定義しなければならない。一つの論理 finding は、関連エントリ数にかかわらず一度だけ数える。それらのルールと完全な出力契約を一緒に実装する前に、プレースホルダー JSON フィールド、Rust reporter バリアント、または出力スキーマを追加しない。

## 将来のカタログチェック向けロケール関連付け

カタログレベル lint が初めてロケール識別子を使用するとき、`CatalogConfig` に、省略可能で判別可能な `locale` オブジェクトを追加する。文字列の省略記法は受け入れず、locale capture を `include` glob 構文へ埋め込まない。初期の二つの関連付けバリアントは、次のパス capture とする。

```jsonc
{
  "include": ["locales/*.json"],
  "locale": {
    "from": "path",
    "pattern": "locales/{locale}.json"
  }
}
```

もう一つは固定ロケールである。

```jsonc
{
  "include": ["vendor/english/*.json"],
  "locale": {
    "from": "fixed",
    "value": "en"
  }
}
```

正規化済み Rust 形状は、有効な組み合わせを推定しなければならない省略可能フィールド群ではなく、tagged enum とする。

```rust
#[serde(tag = "from", rename_all = "snake_case")]
pub enum LocaleBindingConfig {
    Path { pattern: String },
    Fixed { value: String },
}
```

初期リソーススキーマおよび Rust の `CatalogConfig` には、`locale` を含めない。フィールドと enum は、最初のカタログレベル利用側とともに実装する。実装後、厳密な未知フィールド検証は、未知の関連付けバリアント、文字列の `locale` 値、およびバリアントに適さないフィールドを拒否する。

内在ロケール提供元を利用する最初のカタログレベル利用側を実装する際、その同じマイルストーンで、スキーマおよび enum にフィールドを持たない `Host` バリアントを追加する。たとえば、XLIFF 定義では次のように使用する。

```jsonc
{
  "include": ["locales/**/*.xlf"],
  "format": "xliff",
  "locale": { "from": "host" }
}
```

```rust
pub enum LocaleBindingConfig {
    // Existing Path and Fixed variants.
    Host,
}
```

`from: "host"` は、ファイルごとに一度ではなく、選択されたアダプターが文書化する提供元を使用して、エントリごとにロケールを解決する。Vue エントリでは静的インラインブロック `locale`、XLIFF では前述の source／target 規則を使用する。少なくとも一つの内在提供元と、その最初のカタログレベル利用側をリリースする前に、プレースホルダーとして受け入れない。この設定を持つカタログ定義が一致できるのは、文書化された内在ロケール提供元を持つアダプターだけである。別の形式に一致した場合は、パス推論へフォールバックせず、その定義位置の `config_validation_failed` とする。提供元の規則のもとで、選択されたエントリのいずれかが空でないロケールを持たない場合、将来のカタログワークフローは明示的な対象ローカル運用エラーを報告し、そのエントリを黙って除外しない。利用側の実装時に、リンター設計でそのエラー契約を追加しなければならない。

`from: "path"` では、スラッシュ正規化済みで `projectRoot` から相対的なファイルパスに対し、[リソース所属 glob 文法](#リソース所属-glob-文法)を使用する。ただし、エスケープされていない `{locale}` capture token を正確に一つだけ追加で許可する。この token は、`/` を越えない Unicode スカラー値一つ以上を capture する。そのセグメントには、`?-{locale}.json` のように、リテラルの prefix または suffix テキスト、エスケープ済みメタ文字、および `?` を含めてよい。エスケープされていない `*` は含められない。`**` は、別の完全なセグメントとしてだけ引き続き有効とする。それ以外のエスケープされていない `{` または `}` は、通常の brace 不許可規則のもとで無効である。`\{locale\}` はリテラルテキスト `{locale}` に一致し、capture token として数えない。

token 数が一つでない、capture が空、capture セグメントに禁止構造がある、またはその他のパターン構文失敗の場合、`details.reason: "invalid_locale_capture_pattern"` および `details.pointer: "/resources/catalogs/<index>/locale/pattern"` を持つ `config_validation_failed` とする。ロケール対応ワークフローが具体的なカタログ割り当てを解決するようになった後、選択されたカタログパスがその定義の有効な capture pattern 全体に一致しない場合、同じ正確な pointer と、プロジェクト相対スラッシュ正規化済みパスを `details.targetPath` として持つ、`details.reason: "locale_capture_mismatch"` の `config_validation_failed` とする。両方の reason 値とフィールドは、最初のロケール対応利用側とともに実装する。`locale` 全体をまだ拒否する初期スキーマでは受け入れない。

path capture、`fixed.value`、およびホストが提供するエントリロケールは、空でない文字列へ解決されなければならない。リソース層は、その文字列をバイト単位で保持する。空白をトリムせず、大文字小文字を変更せず、`_` を `-` へ置換せず、BCP 47 または Unicode locale 構文として検証せず、非推奨 subtag を canonicalize しない。ロケール識別子の比較は、正確かつ大文字小文字を区別して行う。標準準拠と正規表記に関する助言は、将来の設定可能な lint ルールが管轄する。そのため、プロジェクトまたはランタイム固有のロケール識別子を、リソース抽出を変更せずに表現できる。

リソース層は、ロケール識別子を利用側に依存しないエントリメタデータとして解決する。将来のカタログリンターは、そのメタデータをエントリ識別子および比較スコープと組み合わせてグループ化し、ルール定義、severity、およびレポートを所有する。locale binding は、ランタイムロケールを選択せず、ロケールの fallback を定義せず、メッセージテキストを変更せず、それ自体でルールを有効にしない。

ロケールが関連付けられた各 `CatalogConfig` は、既定で一つの比較スコープとなる。その定義に一致するすべてのファイルを、同じ論理メッセージコレクションの一部として扱う。ロケール横断のグループ化では、その暗黙スコープ、正確に解決されたロケール文字列、`CatalogKeyDomain`、`CatalogKey` の組を使用する。完全な `EntryKey` は、そのグループ内で報告される各出現箇所を引き続き識別する。独立したメッセージ名前空間を持つプロジェクトは、それらを別々のカタログ定義へ分割し、同じカタログキーが誤ったファイル横断関係を作らないようにする。

ロケール対応の一比較スコープに含められるエントリは、正確に一つの `CatalogKeyDomain` だけとする。これは、一定義による暗黙スコープと、同じ明示的 `group` で結合されたすべての定義の両方に適用する。初期のエントリレベルフォーマットおよび lint は、混在形式のカタログ定義、または複数の内側ドメインを含む Vue アーティファクトも引き続き処理できる。これらは、エントリまたはファイル間で `CatalogKey` 値を比較しないためである。ロケール対応カタログチェックを実装した後、ワークフローは、カタログルールを実行する前に、正常に抽出されたエントリからドメインを解決する。参加エントリがゼロの場合はドメインを提供せず、それ以外では有効なスコープを失敗させない。二つのドメインのエントリを持つスコープは、`details.reason: "catalog_scope_key_domain_conflict"` を持つ `config_validation_failed` とする。ワークフローが黙って分割することはない。

外側ホスト形式が等しいことは、この検査に必要でも十分でもない。JSON、JSONC、JSON5、およびそれらの内側アダプターから持ち上げられた Vue エントリは、外側ホスト形式が異なっても、一つの `JsonPointer` スコープを共有してよい。逆に、参加ブロックが JSON と YAML を使用する一つの Vue 対象、または XLIFF 1.2 と XLIFF 2.x の両エントリを含む一つのスコープは、ドメインが異なるため競合する。ロケール対応リンター設計は、この将来 reason が公開サーフェスとして生成される前に、そのカタログレベルエラーおよび抽出失敗の優先順位契約の残りとともに、競合するパス、エントリ、ドメイン ID、および pointer details を決定的に確定しなければならない。具体的に割り当てられたファイルがない有効なスコープは、エラーにならない。

別々の固定ロケール定義など、複数のカタログ定義が一つの比較スコープを形成する必要がある場合は、同じ空でない `group` 文字列へオプトインする。

```jsonc
{
  "group": "vendor-messages",
  "include": ["vendor/en/*.json"],
  "locale": { "from": "fixed", "value": "en" }
}
```

```jsonc
{
  "group": "vendor-messages",
  "include": ["vendor/ja/*.json"],
  "locale": { "from": "fixed", "value": "ja" }
}
```

`group` は、正確かつ大文字小文字を区別する識別子であり、トリムまたは正規化しない。同じ group を持つ定義は一つの比較スコープを共有する。group のない定義は、その暗黙スコープを別の定義と決して共有しない。このフィールドは、ロケール対応カタログ lint とともに実装し、初期リソーススキーマでは無効のままとする。ロケールでインデックス化された比較スコープを構築できないため、カタログ定義は `locale` なしに `group` を指定できない。

一ファイルが複数のロケール対応定義に一致する場合、同じホスト形式、比較スコープ、および各抽出済みエントリに対する正確なロケール割り当てへ解決される場合だけ、重複除去を有効とする。path および fixed binding では、割り当ては一つのファイルロケールの繰り返しである。host binding では、エントリ単位の順序付きロケール列である。抽出済みカタログキードメイン列は、一方の定義から独立して割り当てられるものではなく、一つの重複除去済みホストアーティファクトに内在する。異なる割り当てまたはスコープは、一つの物理エントリを複数の比較グループへ置くのではなく、設定競合とする。これは、ロケール対応設定の実装後に、既存の重複定義競合規則を拡張するものである。

## カタログ単位オプション上書きの再検討

カタログ単位のフォーマッターまたはリンターオプション上書きに、予定済み Tier はない。一様なプロジェクトオプション、カタログの `include` および `exclude`、または明示設定ファイルを使用する別々の CLI 呼び出しによって、具体的なワークフローを合理的に表現できない場合に限って再検討する。明らかに異なる方針を必要とする生成済みまたは外部管理カタログは証拠の例だが、上書き追加を自動的に正当化するものではない。

その証拠が存在する前に、`CatalogConfig` 内へ `fmt`、`lint`、`options`、またはその他のプレースホルダーフィールドを予約しない。後から追加する設計は、スキーマを変更する前に、次のすべてを定義しなければならない。

- プロジェクトレベルのフォーマッターおよびリンターオプションからの継承およびマージ規則
- 一ファイルが、解決済み上書きの異なる複数のカタログ定義に一致する場合の競合処理
- CLI とエディターの利用側で同一のオプション解決
- フォーマッター、リンター、抽出、およびエディターキャッシュキーにおける設定識別子と invalidation
- 上書きがエントリレベルルールだけに適用されるか、将来のカタログレベルおよびロケール横断チェックにも適用されるか
- 対象失敗時に実効方針を JSON および人間向け出力で決定的に報告する方法

これらの要件を一体として設計するまで、カタログ処理は単一の解決済みプロジェクトレベルフォーマッターおよびリンター設定を常に使用する。

## ベンチマーク

リソースのベンチマークは Phase 3 のベンチマーク規約に従い、フェーズごとに分離する。

- `resource_extract`: 形式ごとのホスト解析とエントリ抽出
- `resource_write_back`: 正確な再エスケープ計測、延期された生テキストの実体化、編集合成、および候補全体の再解析／再抽出検証。計測と実体化の時間を個別に報告する。
- `editor_catalog_single_entry_format`: 代表的なカタログサイズについて、一エントリの変更後に必須の候補全体検証を行う。再エスケープと検証を個別に報告する。
- `fmt_catalog_check_e2e` および `fmt_catalog_write_e2e`
- `lint_catalog_e2e`
- worker 幅一と自動選択されるデフォルト幅での複数物理ファイルグループ実行。pool scheduling、backpressure、および安定した順序付き集約を含む。
- 大規模な単一カタログの逐次実行および候補 chunk 実行。ファイル内並列性を昇格させる前に、スケジューリングおよび順序付き集約のオーバーヘッドを含める。
- [ox-mf2-parse-artifact-cache.md](./ox-mf2-parse-artifact-cache.md) にあるカタログ形状のキャッシュシナリオ。たとえば、ホストファイルの編集をまたいでメッセージテキストが変わらないエントリなど。

抽出と再エスケープのコストは、パーサー、意味解析、ルール、フォーマッター、およびファイル I/O のコストと分けて報告しなければならない。

次の項目は意図的に初期リソースマイルストーンの対象外としているが、後の作業に備えて明示しておく。

- Tier 2 の Vue SFC `<i18n>` カスタムブロック対応: 前述の Tier の記述に従い、`crates/intlify_resource` が外側のブロック抽出と、ブロック領域上でのアダプター合成を所有する。宣言された言語に対応するリリース済みの内側アダプターがないインラインブロックは、埋め込みの `resource_format_unsupported` 契約により SFC 全体の抽出を失敗させる。外部 `src` ブロックは、別のカタログ対象のままとする。
- Tier 3 の YAML カタログ対応（`.yaml`、`.yml`）: 前述の Tier の記述に従い、固定 Core Schema の解決、型付き構造パス識別子、スカラー形式の処理、未使用アンカー注釈、alias と merge-key の拒否、およびブロックスカラーを読み取り専用とした正確な診断マッピングを行う。書き込み可能なブロックスカラー再エスケープは、前述したラウンドトリップおよび冪等性要件を条件とする、別の後続マイルストーンである。
- Tier 3 の JSONC および JSON5 カタログ対応: 固定されたコメントと末尾コンマの JSONC プロファイルは JSON 文字列の振る舞いを再利用する。Standard JSON5 1.0.0 は、前述した正確な空白、識別子、文字列、マッピング、および引用符を保持する書き戻し規則を追加する。
- Tier 3 の XLIFF 1.2 / 2.x 対応: 正確なバージョン別プロファイル、安全な XML 解析、エンティティと CDATA のマッピング、正確な文字データ空白、主要 source／target の抽出、およびコメントまたは処理命令の読み取り専用処理を行う。インライン要素対応には、保護編集および可逆なラウンドトリップセマンティクスを持つ、別途設計された明示的 XLIFF プロファイルが必要である。
- ARB、gettext PO、Java properties、およびその他の交換形式は、それらの完全なアダプター契約とレジストリ識別子を設計するまで、Tier 未割り当てで需要に応じて対応する候補のままとする。
- 現時点ではリソース N-API および WASM バインディングパッケージを予定しない。配布はワークスペース内部の `crates/intlify_resource` クレートに留める。Rust 以外の具体的な利用側がこの層へ直接アクセスする必要が生じた場合に限り、パッケージ配布を再検討する。
- キーセレクター、locale binding、およびカタログレベルのチェックは、前述のセクションで追跡する、延期済みの設定および lint 設計作業とする。具体的な最初のルールを予定する時点で、専用リンター追補文書により、ルール ID、既定 severity と preset 所属、完全な入力、専用のカタログ result／finding スキーマ、シリアライズ済み scope と型付き subject の識別子、関連エントリの evidence、決定的な順序、text／editor 表示、`--quiet`、`--max-warnings`、summary 件数、および新しいすべての運用エラー details と優先順位規則を決定しなければならない。その時点で locale binding は、前述の固定された判別可能オブジェクトおよび比較スコープ制約を使用する。初期スキーマでは `locale`、`group`、カタログ finding、および placeholder reporter バリアントを予約も受け入れもしない。
- `intlify init` のリソース scaffold は、将来の init 追補文書が管轄する。初期リソースマイルストーンは、その予約済みコマンドを変更しない。追補文書では、生成設定で `resources` を省略するか、`{ "catalogs": [] }` を生成するか、対話的な例を提示するかを選択し、非対話動作、既存設定の更新、冪等性、スキーマバージョンの認識を定義し、ユーザーが選択していないカタログオプトインを暗示しないようにしなければならない。
- カタログ単位のフォーマッターおよびリンターオプション上書きは、予定せず、証拠を条件とする。前述の再検討要件を満たすまで、プレースホルダーフィールドを受け入れない。
- ファイル内並列性は、実装済みのファイルグループ scheduler に対する代表的なベンチマークによって、前述の決定性規則のもとで CLI 所有の同じ pool を用いる連続エントリ chunk 化が正当化されるまで無効のままとする。

## 未解決事項

この設計レベルでは、リソースカタログアダプターに未解決事項は残っていない。延期済みの実装作業と、証拠を条件とする再検討事項は、前述の各セクションで追跡する。
